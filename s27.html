Search Indexes

Comprehensive Study Notes: Lucene, ElasticSearch, and the Inverted Index

1. The Problem with Databases

In most modern websites, there is a need for open-ended search queries (e.g., Google searching, Amazon product filtering, Twitter keyword matching).

Traditional SQL databases are not built for this. If you need to scan millions of documents for a specific keyword, a standard database query will be incredibly slow and resource-intensive.

Instructor's Insight: To optimize these intense queries, we must step away from standard databases and use specialized data structures and technologies designed exclusively for searching.

2. Apache Lucene & Architecture

Created in 1999, Lucene is the most popular open-source search index. It operates "under the hood" as the engine powering major search services like ElasticSearch and Solr.

How it Stores Data (LSM Tree + SSTable)

  • Memory Buffer: Writes are first sent to an in-memory buffer. Crucial Note: Data in this buffer cannot be searched yet.
  • Flushing: Eventually (usually about every 1 second), the buffer is written to an immutable SSTable (Sorted String Table) file on the disk. Only then is it searchable.
  • Compaction: As many small SSTable files accumulate on the disk, Lucene merges and compacts them in the background to save space and keep searches fast.
  • Reads: Because data is spread across multiple SSTables, a read query must scan multiple files and merge the results.

3. Tokenization (NLP Pre-processing)

You can't just throw raw documents into an index. The document must be split into searchable terms through a process called Tokenizing.

  • Casing: Convert everything to lowercase so searches are case-insensitive.
  • Punctuation & Contractions: Strip commas, periods, and handle words like "it's" vs "its".
  • Stop Words: Remove common words ("the", "and", "a") because they take up space and nobody searches for them alone.

4. The Inverted Index (The Bread & Butter)

After tokenizing, words are added to the Inverted Index. Instead of mapping a Document ID to its text, it maps a Term to a list of Document IDs that contain it.

Because these are stored in SSTables, they are sorted alphabetically by term.

5. Prefix & Advanced Searching

Prefix Search ($O(\log N)$ Time)

If a user searches for words starting with "pe", the engine doesn't scan the whole table. Because the SSTable is sorted, it performs a Binary Search to find the upper bound ("pee") and lower bound ("pen") instantly.

The Suffix Hack

You cannot binary search for a suffix (e.g., words ending in "en"). To solve this, Lucene can create a second index where the characters of the terms are reversed.

6. ElasticSearch (Scaling Lucene)

ElasticSearch takes the capability of a single Lucene index and runs it over a distributed cluster of machines to handle massive scale and ensure availability (via replication).

Document Partitioning

ElasticSearch uses Document Partitioning, not term partitioning. Node A doesn't hold the index for words starting with "A". Instead, ElasticSearch creates a bunch of local inverted indexes for specific documents on a given node.

Note: When searching for "apple", the query is broadcasted to all nodes, and the results are merged. If grouping data properly (e.g., routing all logs for today to Node 3), you can query just one node to save network traffic!

7. Read Performance & Caching

ElasticSearch achieves blazingly fast read speeds through multiple layers of caching:

  • OS Level Caching: The Operating System realizes SSTables are being read frequently from the disk, so it caches those actual disk pages in RAM.
  • Query Result Caching: ElasticSearch caches the actual results of a query at the shard level. If you search the exact same thing again (and the data hasn't changed), it returns the result from memory immediately.
  • Component Caching: ES caches parts of queries. If many different queries use the same filter (e.g., "in_stock: true"), ES caches that specific filter's result to reuse across future, different queries.

8. Conclusion & Takeaways

Search indexes are critical for applications that need to find strings of text faster than a traditional database scan. By combining Inverted Indexes with immutable SSTables, Lucene achieves incredible performance.

However, when scaling with ElasticSearch, you must be careful about how you partition your data to avoid hot-spots and limit cross-partition network queries.
