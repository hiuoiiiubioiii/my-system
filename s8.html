The Pitfalls of Relational Databases

A deep dive into SQL architecture, horizontal scaling challenges, and the rise of NoSQL.

Scroll down to begin your feed ↓

1. Relational Databases Refresher

Relational databases (like MySQL and PostgreSQL) are the traditional standard for data storage. They are built on a highly structured philosophy.

  • Pre-defined Schema: Tables hold many rows of structured data. Columns must be defined upfront.
  • Relationships: Rows in one table relate to rows in another using common keys (Foreign Keys).
  • Declarative SQL: You write queries stating what you want (e.g., SELECT * FROM Users).
🎙️ Instructor's Deep Dive: The "Black Box"

Relational databases feature a built-in Query Optimizer. When you write SQL, you don't tell the database how to get the data. The optimizer acts as a "black box," figuring out the most efficient execution path using indexes, parallelism, and B-Trees under the hood.

Implementation Mechanics

  • B-Trees: The primary data structure. This means all reads and writes go directly to a physical disk (which is inherently slower than in-memory RAM).
  • 2-Phase Locking (2PL): Used to support transactions and enforce isolation. This prevents concurrency bugs but is notoriously slow because reads and writes can block each other.

2. The Problem of Scaling

As applications grow, databases must handle more data and traffic. There are two ways to scale:

🖥️ ➔ 🗄️

Vertical Scaling

Increase the power of a single machine (Mainframes). This is the traditional approach.

💻 ➔ 💻 💻 💻

Horizontal Scaling

Adding more generic computers (nodes) to distribute the workload. Essential for modern Cloud computing.

🎙️ Instructor's Deep Dive: Why Horizontal is Hard

Because modern cloud computing (like AWS) relies on renting generic, standardized computers rather than building massive supercomputers, we must scale horizontally. To do this, we have to shard (partition) our dataset across multiple nodes. This is exactly where relational databases start to fail.

3. The Problem with Sharding: WRITES

Imagine I am Venmo (or a bank), and I need to transfer $1000 from Trump to Putin.

Because we scaled horizontally, Trump and Putin's data live on two completely different server partitions.

🎙️ Instructor's Deep Dive: Distributed Transactions

What if the network request fails halfway? Trump loses $1000, but Putin never gets it. The money vanishes! To prevent this, Relational DBs use a Distributed Transaction to ensure the write succeeds on both partitions or fails on both.

Because multiple servers must agree over a network before finalizing the write, cross-partition writes become incredibly slow.

4. The Problem with Sharding: READS

It's not just writes that suffer; reading related data across shards is a nightmare. Consider loading Facebook Messenger to see all your chats.

User Jordan (ID = 1) is in chats: 1, 3, 5, and 7. To get the chat names, we must do a JOIN operation.

Partition A (Chats 1-4)

  • ✅ Chat 1: Cool Kids
  • ❌ Chat 2: Bad coders
  • ✅ Chat 3: Good coders
  • ❌ Chat 4: Poorly endowed

Partition B (Chats 5-8)

  • ✅ Chat 5: Well endowed
  • ❌ Chat 6: 1 plate club
  • ✅ Chat 7: 2 plate club
  • ❌ Chat 8: Not natty
🎙️ Instructor's Deep Dive: Network Overhead

To fetch the screen for User 1, the database must query Partition A, query Partition B over the network, wait for both to respond, and aggregate the results. If you are in 50 chats spread across 10 partitions, that is too many network calls. One failed node or slow network connection ruins the entire query.

5. The Relational Philosophy vs. Scale

The core philosophies that make SQL great on a single machine are the exact reasons it fails in distributed cloud environments.

SQL Philosophy Why it breaks at scale
One copy of data (Normalization) Forces you to split related data across tables. Reassembling it requires slow cross-shard network calls.
Fetch related data via Joins Joins over a network are fundamentally slow.
Hide concurrency bugs via Transactions Distributed transactions require 2-Phase Locking across nodes, creating massive bottlenecks.
Disk-based (B-Trees) Writing to disk is vastly slower than writing to an in-memory buffer (like LSM Trees).

6. The NoSQL Solution

"NoSQL" is a broad umbrella term. In reality, NoSQL databases are simply stripped-down relational databases.

🎙️ Instructor's Deep Dive: Stripping away features

By abandoning the "black box" features of SQL (like rigid schemas, ACID transactions, and automatic cross-shard joins), NoSQL gives developers the freedom to optimize purely for speed and huge horizontal scale.

Design Pattern 1: Document Databases

Objects are self-contained documents (usually JSON). Everything related to the object is stored inside it. No Joins required.

Example: A LinkedIn Profile stored as one single document.

{ "user_id": 1, "name": "Jordan", "experiences": [ { "company": "Amazon", "role": "Engineer" }, { "company": "Google", "role": "Engineer" } ] }
  • Pros: Incredible disk locality. The DB grabs one document sequentially from disk and is done. Extremely easy to shard (just move the whole document). Schemaless.
  • Cons (Data Duplication): If an author writes 10 books, their name is duplicated in 10 book documents. If they change their name, application code must find and update all 10 documents manually.

Design Pattern 2: Graph Databases

  • Perfect for highly interconnected, many-to-many relationships (like social networks).
  • Unlike Document DBs (which struggle with many-to-many relations), Graph DBs link nodes with metadata, making complex relationship queries efficient without standard SQL joins.

7. Final Conclusion & "NewSQL"

Despite these pitfalls, this doesn't mean SQL is dead or infeasible!

  • SQL is Good For Read-heavy applications with highly structured, related data where consistency is crucial (e.g., Banking).
  • NoSQL is Good For Massive write throughput, evolving schemas, and applications that require massive horizontal scale (e.g., Social media feeds, messaging apps).
🎙️ Instructor's Deep Dive: The Rise of NewSQL

Because the SQL interface is so intuitive and universally loved, companies are trying to build engines that keep the SQL language but fix the underlying scaling issues. Technologies like VoltDB and Google Spanner are attempting to make the relational model scale horizontally. This movement is dubbed NewSQL.

Golden Rule: There is no "one size fits all" database. You must understand how they work under the hood to choose the right tool for your specific application.
