03 Data Modeling: KV vs. Wide-Column

Cassandra is a Wide-Column Store, utilizing partition keys and clustering keys to sort structured rows. Riak is a pure Key-Value (KV) Store. In Riak, the "Value" is completely opaque to the database—it can literally be anything (a simple string, a deeply nested JSON document, or even a binary blob like an image).

Cassandra (Wide-Column)

Partition_Key: "User_123" Clustering_Key: "Date_01-01-2024" ↳ Col 1: "Purchase_A" ↳ Col 2: "$15.00"

Great for range queries (e.g., "Get all purchases for User 123 in January").

Riak (Key-Value)

Key: "User_123" Value: { "name": "Alice", "purchases": [...], "profile_pic": "[Binary Blob]" }

Massive flexibility, but terrible for native querying since the DB can't "see" inside the value.

The Querying Trade-off: Because Riak just sees the value as a blob, to do queries on data for something other than the direct Key requires workarounds:

  • Adding extra pieces of metadata to each KV pair for a secondary index (generally inefficient).
  • Integrating native Riak Search indexes to parse your documents.

04 Conflict Resolution (The Biggest Difference)

Because both databases use a leaderless replication system, any node can accept a write. This inevitably leads to race conditions—two clients writing to the same key simultaneously without knowing about each other.

  • Cassandra: Uses Last Write Wins (LWW) based on timestamps. The older write is completely lost/deleted. Server clocks drift, making this unreliable.
  • Riak: Refuses to lose your data. It uses Dotted Version Vectors to detect concurrent writes and stores them both as Siblings.

The Developer's Burden: When an application server reads a key with siblings, Riak hands back all versions. It is up to the developer to write application-level code to merge these in a reasonable way for the next write.

05 CRDTs: The "Easy Button" for Conflicts

If you don’t want to take on the heavy burden of manually writing application code to merge database siblings, Riak provides a native solution.

Riak has excellent built-in support for CRDTs (Conflict-free Replicated Data Types). These are mathematical data structures that the database natively understands how to auto-merge.

  • Counters: Concurrent increments auto-sum perfectly.
  • Sets: Concurrent additions/removals are unionized without data loss.
  • Maps: Key-value dictionaries inside the value that merge smoothly.

06 Conclusion & Summary Cheat Sheet

Riak is a high-throughput database that achieves scale via partitioning, multi-master replication, and LSM trees. Unlike Cassandra, it is a KV store—use it primarily when single key accesses are important rather than range queries over a partition.

The main benefits of Riak are greater flexibility in values and zero data loss on conflicting writes (either by writing app-level merge code or using CRDTs).
