Database Partitioning (Sharding)

A comprehensive, vertical-feed masterclass on scaling massive databases, managing hot spots, and distributed architecture.

🌍 1. Introduction & Context

What is Partitioning? Also known as "Data Sharding," partitioning is the process of splitting a massive database table into smaller, manageable chunks and distributing them across multiple database nodes (servers).

Why do we need it? In large-scale systems (like Instagram or Google), a single table will eventually become too large to fit on a single machine's disk, or the sheer volume of queries will overwhelm a single machine's CPU/RAM.

🔥 2. Objectives & The "Hot Spot" Danger

When splitting data across nodes, the ultimate goal is perfect balance. On each node, we want:

  • Similar amount of data (Evenly distributed storage).
  • Similar amount of reads and writes (Evenly distributed compute load).

The Danger: Hot Spots

If we fail to balance the data or the query load, certain nodes will become overloaded relative to others. This creates a severe bottleneck known as a Hot Spot.

🧠 3. Methodologies for Partitioning

How do we decide which data goes to which server?

🚨 Crucial Warning: Do NOT use Modulo

Do not take a hash of a key and use the modulo operator based on the number of nodes (e.g., hash(key) % 4_nodes). If you add a 5th node, the math changes for every single key, meaning nearly all your data will have to migrate to new servers over the network, completely crashing your system.

Instead of Modulo, we use Ranges.

Approach A: Key Range Partitioning

Data is split based on the continuous alphabetical or numerical range of the partition key.

Visualizing Key Ranges

  • [A - C] ➔ Node 1
  • [D - F] ➔ Node 2
  • [G - M] ➔ Node 3
  • [N - Z] ➔ Node 4

*Ranges don't have to be even. Adjust range size to keep data size even.

✅ Pros: Simple. Highly effective for Range Queries (e.g., "Give me all users with last names starting with B").
❌ Cons: Manual range determination. Highly susceptible to Hot Spots (e.g., partitioning by timestamp means all new writes hit one single node).

Approach B: Hash Range Partitioning

Instead of sorting by the key itself, you pass the key through a Hash Function (like SHA-256). You then partition based on the range of the resulting randomized hash values.

Key: "jordan"
➔ Hash(x) ➔
Hash: "po23av"
➔
Node 2
(Handles hashes m000 - z999)
✅ Pros: Keys are evenly and randomly distributed between nodes, naturally preventing broad data imbalances.
❌ Cons: No more Range Queries. "apple" and "apply" end up on different nodes. You must query every node.

Instructor Pro-Tip: Fixing Single-Key Hash Hot Spots

Hashing doesn't fix a Justin Bieber hot spot because "Bieber" always hashes to the same node. The Fix: Append a random number (1-20) to the key before hashing (jordan_7). This splits the celebrity's data across 20 nodes!

Tradeoff: To read the data, you must now query all 20 nodes and aggregate the results.

🗂️ 4. Indexes in a Partitioned Database

Recall: An index is metadata showing memory addresses of rows corresponding to specific field values, allowing fast lookups.

Handling secondary indexes in a partitioned database is tricky. There are two primary architectural strategies:

Option 1

Local Indexes

Each partition holds a secondary index that only tracks rows located on that specific partition machine.

Node 1 (Local)
Data: [LeBron, Kobe]
Index: SF➔[LeBron], SG➔[Kobe]
  • ✅ Fast Writes: Updating a row only requires updating the local machine's index.
  • ❌ Slow Reads (Scatter-Gather): To find all "Small Forwards", you must query every single partition, wait for them to check local indexes, and combine results.
Option 2

Global Indexes

The secondary index is partitioned independently. One node holds the global index pointing to rows on any node.

Node 1 (Global Index)
Index: Small Forwards ➔
[Node 1: LeBron, Node 3: Middleton]
  • ✅ Fast Reads: Finding "Small Forwards" requires asking only the one node holding that specific global index.
  • ❌ Slow Writes & Complexity: Writing to Node 1 might require updating an index on Node 3. This requires complex Distributed Transactions. If one fails, data is inconsistent.

⚖️ 5. Rebalancing Partitions

When your app scales, you must add servers (nodes). The goal is to move only a small fraction of data to new nodes to save network bandwidth.

A. Fixed Number of Partitions (Pre-splitting)

Create a massive, fixed number of partitions from day 1. If you have 4 nodes, create 20 partitions (5 per node). When a 5th node is added, it simply takes exactly 1 partition from each existing node.

Consideration: Pick a number too low, partitions grow too huge to move. Too high, and overhead wastes disk space.

B. Dynamic Partitioning

The database automatically splits a partition perfectly in half once it hits a max size limit (e.g., 10 GB). One half stays, the other moves to a new node.

⚠️ The Danger: If there is a brief network lag, the database might mistakenly think a node has crashed. It will panic, dynamically re-balance, and move massive data across an already slow network, completely crashing the system.

C. Fixed Number of Partitions Per Node

A hybrid approach (similar to Consistent Hashing). Each node has a set number of partitions that grow in size. When a new node joins, it randomly selects existing partitions, splits them in half, takes half, and leaves half.
