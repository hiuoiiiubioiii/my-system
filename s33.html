System Design: Load Balancers

Comprehensive study notes extracted from video lecture & slides.

1. Background & The Scaling Problem

As systems scale, they inevitably hit limits. The instructor notes that a single server, no matter how powerful, will crash under the weight of millions of users.

  • Vertical Scaling: Upgrading to more powerful hardware (better CPU/RAM). Eventually hits a physical and financial limit.
  • Horizontal Scaling: Adding more nodes/servers to a cluster. Commonly seen in databases (partitioning/replication) and application servers (especially in Microservice architectures).
Instructor's Core Rule: "Anytime there is a variety of nodes in a cluster that can serve the same requests, using a Load Balancer ensures none of the servers become overwhelmed."

2. Architecture: Where Do They Live?

Load balancers are a type of reverse proxy. They don't just sit between the user and your web servers; they can sit between any two layers of your application to distribute traffic.

3. Load Balancing Overview & Heartbeats

To distribute traffic safely, the load balancer needs to know which servers are healthy. It does this by locally keeping state via heartbeats.

🖥️ App Server
  • What it tracks: Number of open connections, current CPU/Memory Load, and the time of the last heartbeat.
  • Local vs. External: This state is kept locally on the load balancer. If it had to query an external coordination service (like Zookeeper) for every single user request, the network delay would create a massive performance bottleneck.

4. Routing Policies (Algorithms)

Once the load balancer knows who is healthy, it uses an algorithm to decide where to send the user.

Least Connections

Traffic is routed to the server with the fewest active, open TCP connections. Simple and effective for long-lived connections.

Least Response Time

The LB tracks historical data on how fast each server responds. Traffic goes to the fastest responding server (implying it currently has the lowest load).

(Weighted) Round Robin

Standard Round Robin goes down the line (A ➔ B ➔ C ➔ A). Weighted Round Robin accounts for hardware differences. If Server A is twice as powerful as Server B, the LB sends two requests to A for every one request sent to B.

Hashing

Routes based on the actual contents of the request (e.g., User ID). (See Section 6 for in-depth details).

5. Layer 4 vs. Layer 7 Routing

Load balancers can operate at different layers of the OSI model, affecting speed and flexibility.

Layer 4 (Transport)

  • Routes based on low-level network details (Source IP, Destination IP, Ports).
  • Pros: Computationally much faster.
  • Cons: Blind to the actual data inside the packet.

Layer 7 (Application)

  • Routes based on header contents (HTTP paths, cookies, application data).
  • Pros: Massive flexibility. Allows for smart hashing.
  • Cons: Requires more CPU to "unpack" and read the data.

6. Consistent Hashing & Cache Locality

When using the "Hashing" routing policy, you should never use a simple modulus (e.g., Hash(ID) % Servers). If a server dies, the denominator changes, mathematically reshuffling every routing path and crashing the system.

Instructor's Deep Dive: Cache Locality
Consistent Hashing ensures that a specific user (or data type) is routed to the same node every time. Why? Because that node can save (cache) the user's data (like DB queries or profile pictures) locally. This eliminates redundant database queries and massively speeds up the system.
  • Consistent Hashing Algorithm: Distributes items into buckets on a hash ring so that adding/removing a server only shifts a tiny fraction of the items, avoiding network panic.

7. Availability (Fixing the SPOF)

A single load balancer is a Single Point of Failure (SPOF). If it dies, your whole application is unreachable, no matter how many backend servers you have. You must implement redundancy.

Active-Active Setup

Two (or more) load balancers route traffic simultaneously. They share configuration states via network-attached storage.

Warning: If one goes down, the remaining LB suddenly absorbs 100% of the traffic, which could overload it.

Active-Passive Setup

One primary LB handles all traffic. A secondary (passive) LB sits idle, constantly exchanging heartbeats with the primary. If the primary fails to respond, the passive LB promotes itself to primary and takes over.

8. Conclusion: Pros, Cons & Hidden Features

In the real world, load balancers act as the gateway to your system. They often handle SSL Termination (Encryption/Decryption) so your internal servers don't have to waste CPU cycles doing it.

The Pros

  • Ensures servers don't become overwhelmed.
  • Maximizes local cache performance (via consistent hashing).
  • Acts as a secure reverse proxy/firewall.

The Cons

  • Performance Bottleneck: If the LB is slow, the whole app is slow.
  • Complexity: Adds another point of failure to monitor, configure, and maintain.
