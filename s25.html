1. The "Why" & Background of Caching

In all technology, caching is a way of providing faster reads to the end client. It achieves this by using a faster form of data storage. In distributed systems, this almost always means storing data in RAM (Memory) instead of a Hard Disk.

🎙️ Instructor's Insight: RAM is significantly faster than disk storage (often well over an order of magnitude, 10x+ faster). Caching also improves data locality, meaning you move the data physically closer to the user to reduce network travel time.

3 Main Objectives of Caching:

  • Storing precomputed values: Instead of making the database calculate heavy analytics repeatedly, do the math once, store the answer in the cache, and serve it to everyone else.
  • Fewer network calls: Stops requests from having to travel all the way to the main database.
  • Fewer loads on the DB: Databases are often the bottleneck in large systems. Caches act as a shield.

2. The Tradeoffs of Caching

While caching provides massive speed improvements, introducing it adds significant system complexity.

If data isn't in the cache, the total trip takes longer than if there was no cache at all.

Pros:

  • Massively faster reads (10x+ faster).
  • Potentially faster writes (depending on the write policy used).

Cons:

  • Thrashing: When your cache size is too small for your user base. Example: You load your profile, it goes in the cache. Your friend loads theirs, it deletes yours to make room. You load yours again -> Cache Miss. The cache constantly overwrites itself, slowing everything down.
  • Poor Eviction Policies: Deleting the wrong data to make room for new data.
  • Write Complexity: You must design schemas to ensure cache data doesn't become "stale" compared to the database.

3. Cache Eviction Policies

RAM is expensive and limited. When the cache fills up, you must evict (delete) old data to make room for new data. The industry standard for interviews and basic implementation is LRU.

  • FIFO / LIFO: First in/Last in, first out.
  • LFU: Least Frequently Used.
  • Random Replacement
  • LRU (Least Recently Used): Discards the data that was accessed the longest ago.
💡 Interview Gold: How to implement LRU
If asked in a System Design or LeetCode interview, LRU is implemented using a Hashmap + Doubly Linked List. This allows for O(1) access and O(1) eviction.

4. Where to put the Cache?

Option A: Application Server Cache

Storing the cache directly on the RAM of the application server.

  • Pro: Zero network calls required.
  • Con: Data is trapped. Server 2 cannot see Server 1's cache. If Server 1 crashes, all its cached data is permanently lost.

Option B: Global Cache (Industry Standard)

Using dedicated, standalone nodes (e.g., Redis, Memcached) exclusively for caching.

  • Pro: Can scale independently of app servers.
  • Pro: Does not crash if an application server crashes.
  • Pro: Single source of truth—can be accessed by any app server.

5. Keeping Cache Updated on Writes

If data changes, your cache cannot hold onto the old data, or users will see outdated information. You must choose a Write Policy.

How it works: Writes to both cache and DB simultaneously.
Tradeoff: Maximum consistency, but very slow. Requires a "2-phase commit protocol" (atomic transaction) to ensure both succeed or both fail.

2. Write Around Cache

How it works: Write to DB, and instantly tell the cache to delete the old key. On the next read, the cache fetches the new data.
Tradeoff: Simple to build, but guarantees a slow "cache miss" on the very first read after a write.

3. Write Back Cache

Cache
2. Write Eventually (Bulk)

How it works: Client writes ONLY to the RAM cache. Later, the cache bulk-uploads updates to the DB.
Tradeoff: Incredibly fast writes. However, if power fails before the bulk upload, data is permanently lost. Unacceptable for critical data (like passwords).

6. Advanced: Cache Consistency & Distributed Locks

What if we want the extreme speed of a Write Back cache, but we also need other servers to instantly see those changes before they are saved to the hard disk database?

The Solution: Distributed Lock Service (e.g., ZooKeeper)

  • When Server A modifies an object in the cache, it grabs a "Lock" from the coordinator (ZooKeeper).
  • If Server B tries to read that object, the coordinator stops it.
  • The coordinator forces Server A to write its cache modifications back to the main database and release the lock.
  • Only then is Server B allowed to read the newly updated data.
⚠️ The Tradeoff: Only one server can hold the lock at a time. This creates a bottleneck, ruining the ability to have high write-throughput from many simultaneous servers.

Caching in Practice

In the real world, enterprise caches (like Redis) are treated exactly like databases. They require:

  • Replication: For fault tolerance (backups if a node dies).
  • Partitioning: Splitting data up for massive datasets.
  • Coordination Services: For cluster management.

🎯 Conclusion & Summary

  • Core Purpose: Caching improves read latency and protects databases from excessive load.
  • Architecture: It is generally better to decouple cache from application servers (Global Cache) to allow independent scaling and fault tolerance.
  • Write Strategy Cheat Sheet:
    • Write Back: Minimal write latency, but eventual consistency (risk of data loss).
    • Write Through: Maximum consistency, but high write latency.
    • Write Around: Simple implementation, but causes an initial cache miss.
