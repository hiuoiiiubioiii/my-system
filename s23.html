CRDTs (Conflict-Free Replicated Data Types)

Course Context: Moving from wide-column NoSQL (Cassandra/Dynamo) to Key-Value stores (Redis/Riak). Understanding how multi-leader databases handle concurrent write conflicts.

1. The Problem: Multi-Leader Conflicts

Modern databases use multi-leader or leaderless replication to increase write throughput. However, because multiple nodes can accept write requests at the exact same time, data conflicts are mathematically inevitable.

The Goal = Convergence:
CRDTs are data structures designed so that even if replicas get temporarily out of sync due to concurrent writes, once all data is shared (via gossip protocols), all replicas will eventually show the exact same value without losing any updates.

2. Real-World Use Cases

  • Collaborative Editing: Google Docs, Figma (Note: Operational Transform is an alternative algorithm for this).
  • Online Chat Systems: Ensuring the chronological ordering of chat messages is exact across all users' screens.
  • Offline Editing: e.g., Calendar apps. Your phone acts as a local node offline. When reconnected, CRDTs merge your offline changes with the cloud without breaking data.
  • Distributed Key-Value Stores: The core conflict-resolution tech inside Riak and Redis.

3. The Two Types of CRDTs

These two approaches are mathematically equivalent but represent a trade-off in network performance vs. logic complexity.

Operation-Based

Nodes only transmit the action (e.g., "Add 1").

  • Pros: Highly efficient if the state is massive but operations are tiny.
  • Cons: Not Idempotent. If a network glitch sends the operation twice, it processes twice.
  • Requirement: Network protocols (like TCP) must guarantee exactly-once delivery.

State-Based

Nodes transmit their entire local state over the network to merge with others.

  • Pros: Works perfectly with background Gossip Protocols.
  • Cons: Network-heavy if the data state is large.
  • Requirements for Merge Function: Must be Commutative (order doesn't matter) and Idempotent (repeating the merge changes nothing).

4. Visualization: Grow-Only Counter

A State-Based CRDT. Each node holds an array of 0s. The length equals the number of nodes. Node 1 updates Index 0, Node 2 updates Index 1.

To merge, nodes use an Element-wise Maximum. The total value is the sum of the array.

5. Incrementing & Decrementing Counters

The Flaw with Grow-Only: You cannot use a "decrement" operation on the counter above, because the "Element-wise Max" merge function would just overwrite and ignore the decrements (e.g., max of 5 and 4 is still 5).

The Solution (Two Arrays):
Each node tracks two arrays: one for Increments, one for Decrements.

Total Value = Sum(Increments Array) - Sum(Decrements Array)

To Merge: Do an element-wise max on the Increment arrays, and a separate element-wise max on the Decrement arrays.

6. Visualization: Sets & The "Re-add" Problem

Sets work like Inc/Dec counters. You have an Added Set and a Removed Set. The current contents are Added Set MINUS Removed Set.

The Flaw: If you add "Apple", then remove "Apple", "Apple" goes into the Removed Set. Because merging uses Unions, you can never re-add "Apple" again. The math will always subtract it.

The Solution (Unique Tags):
Every time an element is added, attach a unique, random tag to it. To remove it, you put that specific tag into the removed set. If you re-add the item later, it gets a brand new tag!

7. Sequence CRDTs (Briefly)

Used specifically for collaborative text editing. We need to determine the exact chronological order of characters in a document so concurrent typing doesn't turn into gibberish.
