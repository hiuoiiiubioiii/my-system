Hadoop File System (HDFS) Design

In-Depth Study Notes & Visualizations | Continuous Vertical Feed

🌍 1. Introduction & Background

Distributed file systems are crucial components of large-scale distributed systems in modern tech companies. HDFS is the most popular open-source distributed file system, originally modeled after the Google File System (GFS) paper published over a decade ago.

Why is it so popular?

  • Commodity Hardware: It doesn't require specialized, expensive supercomputers. It can run on normal desktop computers or standard cloud instances (like Amazon EC2).
  • Batch Processing: It is the enabler for large-scale data processing engines like MapReduce, Spark, and Tez.
  • Database Foundation: Modern architectures use HDFS as a foundational building block to store vast amounts of data before running heavy computations, creating an extra layer for databases to interact with.
Core Philosophy: HDFS is designed with computation in mind, opting to move the computation close to the data rather than moving massive amounts of data across the network to the computation.

🧱 2. Overview of HDFS Architecture

HDFS handles files differently than a standard laptop file system. It is tailored for massive files (Gigabytes to Terabytes).

  • Write-Once, Read-Many: HDFS is optimized for writing a file once and then reading it many times. While appending or truncating is possible, modifying data in the middle of a file is not supported.
  • Chunking (Blocks): Files are split into chunks (usually 64MB or 128MB per chunk) across multiple DataNodes.
  • Parallelism: Chunking allows multiple machines to read or write different parts of a massive file simultaneously, turning a sequential bottleneck into a parallelized speedup.
  • Replication: Chunks are replicated across different nodes to ensure data is never lost if a single machine breaks down.

🧠 3. The NameNode: The Brain of HDFS

The NameNode does not store actual file data. Instead, it stores all the Metadata (file names, directories, chunk locations, version numbers).

In-Memory Operations

To ensure speed, the NameNode keeps all metadata in RAM. However, RAM is volatile. If the server crashes, RAM is wiped. So, how does it save state?

  • EditLog (Write-Ahead Log): All changes to the file system (e.g., creating a file, moving a directory) are immediately written sequentially to the EditLog on disk. Sequential writes to disk are extremely fast.
  • FSImage: Periodically, the system takes a "snapshot" of the metadata and saves it to disk as an FSImage file.
  • Bootup Process: On startup, the NameNode loads the last FSImage into memory, and replays any operations found in the EditLog that occurred after that snapshot was taken.
Crucial Detail: The NameNode never saves the physical locations of chunks to the persistent disk. It only stores them in memory.

Safe Mode & Block Reports

When the NameNode boots, it enters Safe Mode. It doesn't know where any data is! It waits for Block Reports. Every DataNode checks in and says, "Here are the chunks I currently hold." The NameNode uses these reports to construct its in-memory map of where data lives.

If the NameNode realizes a chunk is under-replicated (e.g., only 2 copies exist instead of the required 3) or if a DataNode dies (stops sending heartbeats), it automatically commands other nodes to replicate that chunk to reach the safety threshold.

📍 4. "Rack Aware" Replication Strategy

HDFS doesn't just put replicas on random machines. It uses a "Rack Aware" strategy to balance fault tolerance with network bandwidth efficiency.

Standard rule for a Replication Factor of 3:

  • Replica 1: On the exact same rack as the writer (makes the initial write very fast).
  • Replica 2 & 3: On a single, different remote rack.

Why the same remote rack? If you put Replica 2 on Rack B and Replica 3 on Rack C, you consume double the cross-network bandwidth. Putting them both on Rack B protects against Rack A losing power, while saving precious network capacity.

The Major Consistency Issue

In theory, waiting for all acknowledgements guarantees strong consistency. However, a major flaw exists:

  • Imagine Replica 1 successfully writes the data.
  • Before Replica 1 can send it to Replica 2, the network fails.
  • Because the pipeline broke, the Client receives a "Failure" error.

The Paradox: The client thinks the write failed, but Replica 1 physically holds the new data on its disk. The system is now inconsistent. HDFS expects the client to handle this by retrying the write until a complete success message is returned.

📖 6. How Reads and Writes Flow

Hadoop Reads

  1. Client queries the NameNode for a list of DataNodes carrying a specific chunk.
  2. Client determines which DataNode is physically closest (using Rack Awareness) to minimize latency.
  3. Client caches this location for future reads.
  4. Client reads directly from the chosen DataNode.

Hadoop Writes (Appends)

  1. Client contacts NameNode to see where chunks are located and to pick a Primary Replica.
  2. If a current primary exists and its lease is valid, it uses it.
  3. If the primary has expired, the NameNode picks a new one—specifically choosing one with the most up-to-date version number. (If none exist, data is lost).
  4. All other nodes become secondary. The client pushes the write to the Primary, which pushes it down the replication pipeline.

🚨 7. The Single Point of Failure & High Availability

In the original GFS paper and early HDFS architecture, there was only one NameNode. If that single server crashed, the entire cluster went blind and stopped working. It was a massive Single Point of Failure (SPOF).

The Fix: High Availability (HA) HDFS using ZooKeeper.

Instead of keeping the persistent EditLog locally on the active NameNode, HDFS utilizes a coordination service called ZooKeeper (specifically forming a Quorum Journal Manager).

How Failover Works:

  • The Active NameNode holds a distributed lock in ZooKeeper and writes all metadata changes to the shared journal.
  • The Standby NameNode constantly reads this shared journal, keeping its own RAM state perfectly synced with the Active node.
  • If the Active NameNode crashes, it loses the ZooKeeper lock. The Standby node immediately grabs the lock, promotes itself to Active, and seamlessly takes over because its memory state is already up-to-date!

✅ 8. Conclusion

By using a rack-aware replication schema and reads, HDFS provides incredibly high availability and throughput for big data workloads.

While the original design was highly susceptible to fault tolerance issues (the SPOF NameNode), implementing ZooKeeper Quorum Journal Managers completely solved this, allowing for instant failovers.

Keep in mind that HDFS is not perfect. As seen in the replication pipeline, application code must be prepared to handle retry logic due to potential data consistency issues when network drops occur mid-pipeline.

Ultimately, using HDFS to store data as the underlying layer for large-scale compute engines (and modern distributed databases like HBase) remains one of the greatest architectural patterns in modern big data technology.
