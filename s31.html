Content Delivery Networks (CDNs)

Complete In-Depth Study Notes & Architecture Breakdown

0. Introduction & Context

Software engineers and internet users consume an estimated 3 to 6 hours of media per day. The underlying technology that has revolutionized the speed at which we can consume this media globally is the Content Delivery Network.

System Design Importance: Understanding CDNs is mandatory for any modern system design interview. When dealing with large-scale web application architecture, CDNs are the standard solution to media delivery bottlenecks.

1. Background: The Problem with Static Content

In large-scale systems, platforms frequently serve Static Content. This includes HTML files, images, videos, or any file where the data is written once and is not expected to be modified, appended to, or truncated frequently.

The Core Bottlenecks:

  • Massive File Sizes: Compared to a simple row of text data in a database, media files are gigantic.
  • Server Overload: Serving these large files directly puts a massive computational and network load on your application servers.
  • Wasted Compute: Because these files do not change, having an application server continually fetch and recompute them for every single user request is a vast waste of resources.
  • Geographic Latency: If your server is in California and the user is in India, physical distance significantly degrades load times.

The Solution: Companies use CDNs to bypass the application server entirely for these specific static files.

2. Storing Static Content: Object Stores vs. HDFS

Before implementing a CDN, we must understand where the original files actually live. Interview Tip: When asked where to store static content, the answer is almost always an Object Store (AWS S3, Azure Blob, GCS).

Why Object Stores over HDFS?

  • Flat Structure vs. Directory Hierarchy: HDFS uses a directory tree managed by a centralized "NameNode". Object Stores use a flat structure (Key-Value pairs), avoiding central bottlenecks.
  • Independent Scaling: With cloud Object Stores, you only pay for what you use. Storage and Compute scale independently. With HDFS, you must provision physical hardware where storage and compute are coupled.
  • Speed via Weaker Guarantees: Object stores don't support atomic file renaming or locks. Writes are immutable (new writes just increment a version number), which translates to highly parallelized, faster read/write speeds.

Downsides to Object Stores

  • No Multi-Object Transactions: You cannot execute complex transactions across multiple files easily without a complex coordination service (like a 2-Phase Commit).
  • Terrible for Batch Jobs / Analytics: HDFS thrives on Data Locality (running compute on the same machine holding the data). Object stores are remote; running analytics requires transferring massive amounts of data over the network to a compute cluster first, which is incredibly slow.

3. Object Stores Aren't Good Enough

Even though Object Stores like S3 are fast and scalable for holding the original files, they do not solve the geographic distance problem.

The Bottleneck: If a user in Asia requests a large 4K video file stored in an S3 bucket in California, making a network call across the globe for every single byte request is highly inefficient and creates unacceptable lag and buffering.

4. Enter Content Delivery Networks (CDNs)

Definition: A CDN is a special type of geographically distributed cache used exclusively to store static content closer to the end-users. Unlike a standard database cache located inside your main data center, CDNs live on the "Edge" (physically close to user populations).

Data Partitioning: You can partition data geographically. For example, American news data is routed specifically to North American CDNs, while Asian news is routed to Asian CDNs.

5. Global Network Architecture

CDNs act as the middleman between the global user base and your centralized origin server. Users are automatically routed to the CDN edge node geographically closest to them.

Handling Stale Data: If the origin server updates a file, the CDN might still hold the old version. This is managed via Expiration Times (TTL - Time to Live) or by manually purging the CDN data.

6. CDNs Analyzed: Trade-offs

Pros (+)

  • Reduces Origin Load: Protects application servers from crashing under the weight of popular media requests.
  • Increases Speed: Geographic proximity drastically lowers latency for end-users.
  • Security & Reverse Proxy: CDNs act as a wall of defense between the client and your server, absorbing and protecting against malicious traffic like DDoS attacks.

Cons (-)

  • Stale Data: Users might occasionally see outdated content (mitigated by TTL/expiration strategies).
  • Cache Miss Penalty: If a file isn't in the CDN, the request takes longer because it has to travel: User → CDN → Origin → CDN → User.
  • Database Overhead: Your database must be updated to store the static URL pointing directly to the CDN instead of the local server path.

7. Conclusion

Adding a Content Delivery Network in conjunction with an object file storage system (like AWS S3) is a virtually guaranteed necessity for any large-scale consumer platform. It takes the heavy lifting off your compute servers and ensures lightning-fast media rendering for a globally distributed user base.
