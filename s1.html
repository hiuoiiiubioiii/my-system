Topic 1

What is a Database?

A database is a system that allows users to access shared data, persist that data over time (even if applications crash or restart), and query it efficiently. Clients interact with a Web Server, which in turn queries the Database Server.

Core Objectives of a Database

  • ⚡ Fast Reads: Retrieving requested data instantly.
  • ⚡ Fast Writes: Storing incoming data without blocking the system.
  • 💾 Persistent Data: Data must survive power outages and crashes.
Topic 2

The Hardware Reality: Disks

To achieve persistence, data goes to a physical disk. Most large-scale, cost-effective databases still rely heavily on Hard Disk Drives (HDDs) rather than SSDs. You must understand how an HDD works to write efficient database engines.

An HDD is mechanical. Data is read by a physical arm moving over a magnetic platter spinning at 5400 or 7200 RPM.

Slow Random Access: The arm must physically lift and move to find scattered data.
Fast Sequential Access: If data is written continuously in a straight line, the arm stays still. Sequential operations are king.
Topic 3

Naive Database Implementations

Why can't we just use basic programming structures? Let's analyze why simple approaches fail at massive scale.

1. Array / List Implementation

O(n) Reads/Writes

Literally just a list. To update ID 3, you must scan the entire array to find it, then overwrite the value in place.

2. Append-Only Log

O(1) Writes O(n) Reads

To leverage fast sequential disk writes, we never overwrite data. We just append the new value to the bottom. But to read the latest value, we must scan the file bottom-up.

3. In-Memory Hashmap

O(1) Reads/Writes

A hashmap is perfect, except it does not scale. A hashmap must reside in RAM. The second your data exceeds your RAM, the hashmap is forced onto the hard disk. Because a hashmap scatters data everywhere, you now incur devastatingly slow random disk operations.

Topic 4

The Real Solution: Indexes

🗺️

To solve the slow read problem of an append-only log, we use Indexes. An index is an extra data structure maintained alongside your actual data. Think of it like the index at the back of a book.

The Universal Index Trade-off

👍 Faster Reads Finds data instantly via the "map".
👎 Slower Writes Every disk write requires updating the index too.

Rule: Only index frequently queried fields. Do not index everything.

Topic 6

SSTables & LSM Trees

Standard architecture for massive, write-heavy databases (like Cassandra).

1. Write to MemTable & WAL

Writes go first to an in-memory balanced binary search tree (the MemTable). Trees keep keys sorted automatically. To prevent data loss if the server crashes, every write is also logged to a sequential Write-Ahead Log (WAL) on disk.

2. Flushing & 3. Compaction

When the MemTable fills up, it writes its pre-sorted contents to an SSTable (Sorted String Table) on disk. Over time, append-only logs create duplicates. A background process merges them (Compaction), keeping the newest values.

4. Reading Data (Sparse Index)

Reads search the MemTable first, then SSTables from newest to oldest. To optimize disk reads, we use a Sparse Index in RAM. We don't store every key, just every 10th or 100th key, to give us the general location block on the disk to start a sequential scan.

Search for Brian? Must be between Bob & Charlie!

Pros High write throughput (goes to RAM buffer). Excellent for range queries due to physical sorting on disk.
Cons Reads can be slow (checking many files). Background compaction consumes CPU/Disk resources.

B-Tree Operations

  • Reads & Updates: Traverse the tree, find the leaf block, and update the value in-place directly on the disk.
  • Writes (Page Splits): Traverse to the block. If there is space, write it. If the block is full, you must split the block into two, and update the parent block to reference the new blocks. This is an expensive operation.
  • Durability: To prevent data corruption during an expensive page split, B-Trees also utilize a Write-Ahead Log (WAL).
Pros Consistently fast reads. High branching factor means trees are shallow (3-4 levels). Great for range queries.
Cons Slower writes compared to LSM trees due to writing to random blocks and managing expensive page splits.
Conclusion

Choosing the Right Engine

In system design, there is no "best" database—only the right tool for your specific read/write throughput requirements.

Hash Indexes Fastest lookups. Only useful for small datasets that fit entirely in RAM (e.g., Cache, Redis).
SSTables / LSM Optimized for high-volume writes. Slower for reading older/missing records (e.g., Cassandra).
B-Trees The standard choice. Highly optimized for fast reads and range queries. Slower writes (e.g., MySQL, Postgres).
