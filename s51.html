Systems Design: Choosing a Database

A continuous vertical feed study guide. Scroll down to master database indexing, replication, and the exact use-cases for SQL, NoSQL, Graph, and Time-Series databases.

1. Background: Database Indexes

A database index speeds up reads conditioned on a specific key. Trade-off: Never overuse indexes, as updating the index slows down database writes.

A. LSM Trees + SSTables (Log-Structured Merge-Tree)

  • Writes: Go straight to an in-memory balanced binary search tree (very fast).
  • Flushing: When the tree gets too big, it is flushed to the disk as a sorted file called an SSTable.
  • Compaction: Background processes merge multiple SSTables together, discarding old/overwritten values of keys.
  • Reads: Checks memory first, then binary searches through the on-disk SSTables.

Reads search newest to oldest. Compaction merges them.

Pro: Extremely fast writes (writing to memory).
Con: Slower reads (may have to search multiple SSTables).

B. B-Trees

  • A binary tree data structure implemented using physical pointers directly on the disk.
  • Writes: Iterates through the tree on disk, overwriting the existing key value OR creating a new page on disk and updating the parent pointer.

2. Background: Replication

Replication is copying data across multiple nodes so if a database goes down, data isn't lost.

Types of Replication

  • Single Leader: All writes go to ONE master node. Reads can come from any node.
    Use when: You need to ensure absolutely zero write/data conflicts.
  • Leaderless / Multi-Leader: Writes can go to all (or a subset of) nodes. Reads pull from all nodes to find the newest data.
    Use when: You need massive write throughput that a single node can't handle. Cost: Potential write conflicts.

3. SQL Databases (Postgres, MySQL)

The traditional standard prioritizing strict correctness over raw distributed speed.

  • Data Model: Relational/Normalized. Changes to one table usually require changes to others (e.g., adding an Author and Books to different tables).
  • ACID Transactions: Guarantees operations fully complete or fully fail.
  • The Distributed Problem: If normalized tables live on different nodes, maintaining ACID requires a Two-Phase Commit. This locks databases to ensure agreement, which is incredibly slow.
  • Under the hood: Typically uses B-Trees.
Conclusion: Use SQL when correctness is more important than speed.
See: Banking applications, job scheduling.

4. MongoDB (Document NoSQL)

  • Data Model: Document data model. Data is written in large, nested JSON-like documents (Denormalized).
  • Advantage: Better data locality. If an Author and their Books are in one document, the disk head only reads one location.
  • Under the hood: Still supports B-Trees and Transactions.

Conclusion: Rarely the "special" answer in an interview. Good if you want SQL-like guarantees but need more schema flexibility.

5. Apache Cassandra (Wide Column)

The heavyweight champion of massive write throughput.

  • Data Model: Wide column store. Has a shard key (for partitioning) and a sort key.
  • Replication: Configurable Multileader/Leaderless.
  • Index: Uses LSM Trees and SSTables (Lightning fast writes to memory).
  • Conflict Resolution: Uses Last Write Wins (LWW) based on timestamps. If clocks are desynced, it might accidentally clobber (overwrite) a newer, valid write!
Conclusion: Great for applications with extreme write volume where strict consistency isn't critical. All writes/reads go to the same shard (no cross-node transactions).
See: Chat Applications (Facebook Messenger), where losing 1 out of 10,000 messages isn't a system failure.

6. Riak (Wide Column)

Very similar to Cassandra (Wide column, Leaderless, LSM Trees), but fixes the "Last Write Wins" data loss issue.

  • The Difference: Supports CRDTs (Conflict-Free Replicated Data Types).
  • Instead of simply overwriting data based on a timestamp, CRDTs provide custom logic to mathematically merge conflicts without losing data.
Conclusion: Similar use case to Cassandra, but specifically better when you need to implement distributed counters or sets (like counting Facebook "Likes") in a conflict-free way.

7. Apache HBase (Wide Column)

Another wide column store, but prioritizes read speed for specific data attributes.

  • Replication: Single leader replication (Built on Hadoop/HDFS). Slower writes than Cassandra, but ensures data consistency/durability.
  • Index: LSM Trees + SSTables.
  • Superpower: Column-Oriented Storage. Stores data by column on disk instead of by row.

Row-Oriented (Disk reads full rows)

[ID|Name|Thumbnail]
[ID|Name|Thumbnail]

Column-Oriented (Disk reads only what you need)

[ID|ID|ID]
[Name|Name|Name]
[Thumb|Thumb|Thumb]
Conclusion: Great for applications that need fast column reads and high column compression (increased data locality).
See: Fetching multiple thumbnails for a YouTube homepage, bulk reading sensor readings.

8. Memcached and Redis (In-Memory)

Key-value stores implemented entirely in RAM (Memory).

  • Under the hood: Uses a standard Hashmap. Lookups are O(1).
  • Redis is generally preferred over Memcached as it is more feature-rich (supports geospatial indexes, sorted sets, etc.).
  • Drawback: Memory is very expensive compared to hard drives, limiting this to smaller datasets.
Conclusion: Useful for data that must be written and retrieved extremely quickly.
See: Caching layers, Geospatial indexes for Uber/Lyft (where driver coordinates update constantly and must be read instantly).

9. Neo4j (Graph Database)

Built specifically for relationships.

  • The SQL Problem: If you use SQL to map relationships (many-to-many), every "hop" to a friend requires searching a B-Tree index O(log N). Deep queries ("friends of friends of friends") become devastatingly slow.
  • The Graph Solution: Neo4j uses direct pointers on disk. The data address of "User A" literally contains a direct memory pointer to "User B". Lookups are O(1).

Conclusion: Only useful for data naturally represented in graph formats.
See: Map routing data, Friends/Connections on social media.

10. TimeScaleDB / Apache Druid (Time-Series)

Niche databases strictly designed for time-stamped streams of data.

  • Under the hood: Uses LSM trees for fast ingestion.
  • The Optimization: Instead of one giant index, it breaks tables into many small indexes chunked by ingestion source and timestamp.
  • Because they are small, entire index chunks can be placed in the CPU Cache for extreme performance.
  • Deletes: Standard databases use "tombstones" to mark deleted data (slow). Time-series databases can just instantly delete the entire small index chunk when the time range is no longer relevant.
Conclusion: Very niche but serve their purpose perfectly.
See: Sensor data, system metrics, server logs (where you query by ingestor and a range of timestamps).

11. "New SQL" (Advanced / Theoretical)

Attempts to solve the slowness of traditional distributed SQL databases. (Great to mention in interviews, but usually too expensive to actually implement).

  • VoltDB: SQL, but runs completely in-memory using single-threaded execution. Because there is only one thread, there are zero race conditions, eliminating the need for slow locking protocols. Very fast, but expensive and limited to small datasets.
  • Google Spanner: Globally distributed SQL. It physically installs GPS and atomic clocks in datacenters to assign undeniable, exact timestamps to determine the order of writes. This bypasses the need for distributed locking. Incredibly expensive.
