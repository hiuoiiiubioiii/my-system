📚 VoltDB Study Notes

Complete breakdown of NewSQL, Single-Threaded Execution, and In-Memory Data Storage

0:34

1. Background & Context

VoltDB is another example of a NewSQL database that uses a variety of optimizations in order to try to gain performance increases over traditional solutions such as MySQL or PostgreSQL.

VoltDB is primarily interesting as a case study in transaction serializability. It takes a highly unconventional, contrarian approach to serializability to optimize its performance compared to traditional relational databases.

0:56

2. Design Philosophy: Abandoning Locks

  • Recall: Traditional SQL databases use two-phase locking in order to implement transaction serializability, which leads to slow throughput (due to locking overhead) and deadlocks.
  • Instead: VoltDB has chosen to use actual serial execution. Each database instance actually executes every operation in a single-threaded manner, hence no locks are needed!
Visualization: Traditional vs. VoltDB Execution
Traditional DB (2-Phase Locking)

Thread 1 ➔ 🔒 Grabs Lock A
Thread 2 ➔ ⏳ Waits for Lock A
Result: Overhead, Blocking, Deadlocks
VoltDB (Single-Threaded)

Queue: T3 T2 T1 ➔ ⚙️ CPU
Result: One at a time. Zero locks. Zero deadlocks.

The Problem with Traditional DBs: Traditional databases use "two-phase locking." This involves grabbing shared and exclusive locks on individual rows for reading and writing. This creates massive overhead. Transactions wait for each other, and it frequently leads to "deadlocks" (causing both queries to freeze).

The VoltDB Solution: Throw away locks entirely. Put all read and write transactions into a single queue. The CPU executes them one by one, in serial order. Because only one transaction is happening at any exact given microsecond, conflicts and deadlocks are impossible.

1:44

3. Supporting Single-Threaded Execution

In order to ensure that each database node is capable of running each operation as fast as possible (in order to avoid bottlenecks), VoltDB does the following:

  • All data stored in memory
  • Transactions must first be stored as stored procedures

The Catch: If you took a standard MySQL database and forced it to run on a single thread, it would be incredibly slow. For single-threaded execution to be a benefit rather than a bottleneck, the individual operations must execute blisteringly fast.

Therefore, VoltDB had to make two massive architectural changes: relying completely on RAM (memory) to eliminate disk wait times, and using Java stored procedures to eliminate network wait times.

2:28

4. Storing Data in Memory & Durability

  • Storing data in memory eliminates disk stalls (no overhead moving pages from disk to page cache).
  • Command Log in conjunction with occasional snapshot ensures durability.
    • Similar to a Write-Ahead Log (WAL), but has lower overhead because it describes the transaction instructions themselves instead of the result.
    • Assuming the command log is updated synchronously, writes can be lost if updated concurrently with memory.
Visualization: Command Log vs Traditional WAL
Traditional WAL

App ➔ DB ➔ Calculates Result ➔
💾 Writes exact final data rows to disk
VoltDB Command Log

App ➔ DB ➔
💾 Writes just the code/instructions to disk ➔ Executes in RAM

Why RAM? Disk I/O is the slowest part of a database. By storing everything in RAM, the database never pauses to fetch data from a hard drive.

Durability (Preventing data loss): If RAM loses power, data is erased. To prevent this, VoltDB uses a Command Log. Instead of saving the results of a query to disk (like a traditional WAL), it saves the instructions (the code itself). This requires writing much less data to the disk.

Synchronous vs. Asynchronous: If done synchronously (disk write happens before RAM update), data is safe but slightly slower. If done asynchronously, a sudden crash could cause a write to be lost before hitting the disk.

4:12

5. Eliminating User Stalls via Stored Procedures

  • Using stored procedures eliminates user stalls.
  • Stalls occur in a read-modify-write transaction when an app reads data, calculates logic, and sends an update. This adds network latency.
  • Sending a single stored procedure ensures that all time spent in the transaction is just CPU cycles, not network latency.
Visualization: The "User Stall" Problem
Standard DB (User Stall)
App ➔ "Get Balance" ➔ DB
DB ➔ "Balance is $100" ➔ App
App ➔ "Set Balance to $50" ➔ DB
(Database halts entirely while waiting for network trips)
VoltDB (Stored Procedure)
App ➔ Sends Java Code Block ➔ DB
DB ➔ Executes everything internally instantly ➔ App
(Zero network waiting during execution)

The Core Issue: In a single-threaded DB, if the database has to wait for data to travel over the internet to the application and back, the entire system freezes. This network latency would destroy performance.

The Solution: VoltDB uses Stored Procedures written in Java. Instead of sending data back and forth, the application sends the logic (the code) to the database upfront. The database executes the entire read-modify-write logic internally, taking only pure CPU calculation time.

5:49

6. Partitioning and Replication

  • Because data is in RAM, datasets will inevitably need to be sharded.
  • Transactions that reach multiple shards use one node as a coordinator node.
  • Small Table Optimization: Keep small, frequently joined tables whole and store them on every node.
  • Multi-Leader Replication: For cross-datacenter conflicts:
    1. Deletes take precedence over inserts/updates.
    2. If two conflicting updates occur, Last Write Wins (LWW) is used.

Sharding is mandatory: RAM is limited compared to disks, so large datasets must be split across machines. You ideally want a transaction to only need data from one shard to avoid network latency.

The "Small Table" Trick: If you have a small table (like a list of zip codes) that is constantly joined with larger tables, do not shard it. Copy that entire small table to every single node. This prevents slow network joins across shards.

Conflict Resolution: In multi-leader setups (e.g., across multiple data centers), conflicts happen. VoltDB has strict, simple rules: A Delete will always overwrite an Update. If it's two Updates, the one with the latest timestamp wins (Last Write Wins).

8:03

7. Conclusion & Interview Utility

By removing locks, VoltDB is able to achieve extremely high performance, even more so than in-memory key-value stores. While VoltDB itself isn’t overly useful for system design interviews, it is a great case study in how actual just using a single-threaded database can be feasible under certain circumstances.

The instructor contrasts VoltDB with Google Spanner. Spanner uses locking, which makes massive read-only queries very inefficient due to the overhead of grabbing locks on thousands of rows. VoltDB proves that an unfamiliar design—putting everything in RAM, using stored procedures, and running a single thread—can completely eliminate lock and network wait times.

💡 Exam / System Design Takeaway

If asked about database locking, deadlocks, or 2-phase commits, use VoltDB as a contrarian example: "While traditional DBs handle concurrency via two-phase locking, systems like VoltDB achieve extreme performance by moving all data to memory, eliminating locks entirely, and forcing single-threaded execution via stored procedures to bypass network latency."
