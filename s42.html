Merkle Trees

Comprehensive Lecture Breakdown: Algorithms, Traversal, Git, and Cassandra

1. Background & Definition

The instructor starts by jokingly referring to them as "Angela Merkle Trees," but clarifies they are a vital algorithmic data structure in computer science.

What is a Merkle Tree? It is an algorithm that heavily relies on hashing. It is extremely useful for efficiently calculating the differences between sets of data or files.

Key Characteristics:

  • Logarithmic Time: Unlike linear searches (checking every file one-by-one, $O(n)$), Merkle trees determine where files differ in logarithmic time ($O(\log n)$).
  • Network Efficiency: They calculate the smallest unit of difference between two sets of records so you only have to send the changed data over the network.
  • Like Bloom Filters: Their high performance is derived from cryptographic hash functions.

Real-World Applications:

  • Git: Revision and version control systems.
  • Leaderless Databases (e.g., Cassandra, DynamoDB): Used in "anti-entropy" processes to sync data replicas.
  • Blockchain/Crypto: Used in decentralized currencies to verify transactions.

2. The Basic Algorithm

Here is how a Merkle Tree is constructed from the ground up, using four basic text files: a.txt, b.txt, c.txt, and d.txt.

  1. The Leaves: Pass each file through a hashing function to generate a unique hash (e.g., a.txt becomes h1z2).
  2. The Parents: Group the leaves into pairs. Concatenate the two hashes together, and hash that combined string to create the parent node.
  3. The Root: Repeat the process moving up the tree until you are left with a single, ultimate hash at the very top.

Important: The Root Hash (po73) represents the exact state of all four files combined.

3. Making a Change & Tree Comparison

What happens when we make a change to a single file? Let's alter c.txt.

When the data changes, its hash changes. That change cascades upward. The parent node recalculates, and ultimately, the Root Node recalculates.

po73
u67r
fdc1
a.txth1z2
b.txty77a
c.txt93pq
d.txts345
Tree is in its original state.

The Traversal Process (How it saves bandwidth):

  1. The system compares the original root to the new root. They don't match, so it looks deeper.
  2. It compares the left child nodes (u67r). They match! It immediately ignores files A and B.
  3. It compares the right child nodes. They don't match, so it looks at their children.
  4. It compares d.txt hashes. They match. Finally, it compares c.txt hashes. Mismatch found!

Result: The system only sends c.txt over the network, completely skipping A, B, and D.

4. Real-World Application 1: Git

The standard Merkle Tree throws out the old hashes when a file is changed. But in Git, we want persistence so we can inspect old commits.

How Git modifies the process:

  • Immutability: Each change to a file is treated as a completely separate file and hashed.
  • Deduplication: If a hash is already in the Merkle Tree (i.e., the file didn't change), nothing happens. It just reuses the pointer.
  • Branching: If it is a new hash, Git adds it as a leaf and creates a new branch/copy of the path from that leaf up to a brand new Root Node.
  • Time Travel: A "Commit" is just a Root Node. To go back in time, Git simply starts reading from the Root Hash of that older commit. The old nodes were never deleted!

5. Real-World Application 2: Cassandra

Cassandra is a leaderless, multi-master database. It uses a background process called Anti-Entropy to ensure all replicas have consistent data.

The Problem: Data is always changing! If you build a Merkle Tree on live memory and send it over the network, the data might change before it even arrives, making the tree instantly invalid and stale.

The Solution: Incremental Read Repair & SSTables

To fix this, Cassandra builds Merkle Trees out of SSTables. SSTables are files written to disk that are immutable (they never change).

  • Repaired vs. Unrepaired: Once an SSTable is verified and synced between replicas, it is marked as "Repaired". Because it is immutable, it will never need to go through anti-entropy again.
  • Smaller Trees: Cassandra only builds Merkle Trees out of the "Unrepaired" SSTables. This means the trees are much smaller, saving massive amounts of network bandwidth and I/O overhead.
  • Size Tiered Compaction: To prevent verified data from mixing with unverified data, Cassandra separately compacts repaired tables and unrepaired tables. If they mixed, the system would have to recalculate everything!

6. Conclusion

Merkle trees are a very clever way of detecting the differences between large sets of records by using a logarithmic tree of hashes. They are extremely applicable in multiple areas of systems design:

  • Detecting the differences in files (Git versioning).
  • Finding the smallest units of difference to propagate over the network (Anti-entropy in databases).
  • Ensuring data integrity efficiently across distributed networks.
