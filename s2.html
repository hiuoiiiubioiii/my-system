1. What is Replication?

Replication is the process of having redundant copies of a database.

  • Fault Tolerance: The application does not fail in the event of a crash. If one goes down, another takes over.
  • Scalability: Reduces the load on each database serving requests.
🎙 Analogy: Cloning Yourself "If I have a heavy load from school homework, and a heavy load from working out at the gym, making a clone of myself would allow me to spread that work out. Similarly, spreading requests across multiple databases stops one machine from getting overwhelmed."

2. Single Leader Architecture

There are three main types of replication: Single Leader Multi Leader Leaderless. This lecture focuses exclusively on Single Leader.

  • All writes go to one Master database.
  • The Master sends a list of writes to other databases via a Replication Log.
  • Reads can come from ANY database (Master or Slaves).

3. Increasing Availability & Handling Crashes

How do we keep the system running smoothly when adding machines or dealing with infrastructure failures?

A. Adding a New Follower (Without Downtime)

You can't just copy data while the database is live because data is constantly changing.

  1. Take a consistent snapshot of the leader database at a specific point in time.
  2. Note the exact index/position in the replication log associated with that snapshot.
  3. The new follower downloads the snapshot, then asks the leader for all changes that occurred after that specific index.

B. Dealing with a Follower Crash

Followers write to durable storage (hard drives). At the time of a crash, the follower knows exactly where it was in the replication log.

  1. On reboot, the follower fetches all new changes from the leader node.
  2. It starts implementing changes from the exact index where it previously failed to catch up.

C. Dealing with a Leader Crash (Failover)

When the Leader dies, the system must perform a Failover to promote a Follower to the new Leader.

  1. Determine a new leader via consensus (usually the replica that is the most up-to-date).
  2. Configure all clients to send writes to the new leader.
  3. Configure all other followers to pull replication logs from the new leader.
⚠️ Problems with Failover
  • Lost Data: Writes from the old leader might not have propagated to replicas before the crash.
  • Accidental Failovers: Network congestion can make a healthy leader look dead. Forcing a failover consumes heavy resources, making the network congestion even worse.
  • Split Brain: If the old leader comes back online, it might still think it's the leader. If two databases accept writes simultaneously, data becomes severely corrupted.

4. Synchronous vs. Asynchronous Replication

Synchronous

  • The client does not receive a success message until ALL replicas complete the write.
  • Tradeoff: Provides Strong Consistency (data is always exactly up to date everywhere), but writes take a long time due to global network transport times.

Asynchronous

  • The client receives a success message the exact second the Master completes the write. Master updates replicas in the background.
  • Tradeoff: Writes are much faster, but results in Eventual Consistency. Clients might make "stale" reads if they hit a lagging replica.

5. Eventual Consistency Anomalies

While Eventual Consistency is fast, it creates three specific, frustrating scenarios for user experience. Here is how to fix them.

Anomaly 1: Reading Your Own Writes

The Problem: You make a change, read from a lagging replica, and don't see your change. It looks like the save failed.

🎙 Analogy "You update your Facebook profile to say you are 6'0" instead of 5'11" to impress people. You hit save, the page reloads from a replica, and it still says 5'11". You get angry."

The Solutions:

  • Always route reads from the user directly to the Leader for areas of the app they can edit themselves (like their own profile).
  • Always read from the Leader (or a confirmed up-to-date replica) for a set timer (e.g., 30 seconds) immediately after the client makes a write.

Anomaly 2: Monotonic Reads (Time Traveling)

The Problem: Reads look like they are going backward in time because subsequent refreshes hit different replicas.

Refresh 1 (Hits Replica A - Up to date)
1:05Kate: What's up
1:10Jordan: Not much playing video games
1:15Kate: Wanna come over?

Refresh 2 (Hits Replica B - Lagging behind)
1:05Kate: What's up
1:10Jordan: Not much playing video games
Message from 1:15 suddenly disappears!

The Solution: Fix this by having each user read from the same replica every time (e.g., hash the User ID and map it to a specific replica).

Anomaly 3: Consistent Prefix Reads (Causality Lost)

The Problem: The causal relationship between writes is lost because a preceding write takes longer to replicate than the response (usually due to database partitioning/sharding). Replies appear before the original message.

User C views the feed (Hits lagging Partition 1, fast Partition 2)
6:22 - (Person A) Oh no the Knicks blew another game. [DELAYED IN NETWORK]
6:25Person B: Screw me!!
6:27Person C: On my way!

*Person C sees the reply out of context because the original message is stuck on a slower partition.*

The Solution:

  • Put causally related events on the same exact partition.
  • If not possible, keep track of causal dependencies (give the reply an ID pointing to the original message, so the database knows to hold it until the original arrives).

6. Replication Log Implementation Options

How do we actually format the data sent from the Master to the Slaves? We have three options:

  • 1. Copy over SQL statements (Bad)
    SQL statements are nondeterministic. If a query uses NOW() to get the current time, the Master and the Follower will execute it at different exact milliseconds, resulting in mismatched data.

  • 2. Use Internal Write Ahead Log (WAL) (Bad)
    This tells the disk exactly which literal bytes changed. It is not scalable if you change database engines (e.g., MySQL to Postgres). Different engines store data in different locations on disk, so byte-level instructions will break the replicas.

  • 3. Use Logical Log (Best)
    Describes exactly which rows were modified (by Primary Key) and how. It is decoupled from the storage engine, allowing for future-proofing if the underlying database engine changes in the future.

7. Conclusion & Tradeoffs

Pros:

All writes go through one single machine. This ensures consistency across replicas and eliminates data conflicts (you have one definitive source of truth).

Cons:

Since all writes have to go through one machine, write throughput may not be acceptable at scale. The leader becomes a bottleneck.

⏭ Next Steps in System Design To solve the write bottleneck of Single Leader replication, we either need to partition (shard) the dataset and have a single leader per partition, or look toward other replication strategies (Multi-Leader or Leaderless), which will be covered in upcoming materials.
