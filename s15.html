Batch Processing

Comprehensive Video Lecture Notes & Systems Design Concepts

1. Background & The Problem

Most modern companies hold an immense volume of unstructured data about user activity and usage patterns. They need to run massive, long-running computations on this data to gain insights.

Why not just use a Data Warehouse?

  • Structured Data Requirement: Data Warehouses require data to be pre-formatted and highly organized.
  • The SQL Bottleneck: You are inherently limited by the SQL query language. You cannot easily run arbitrary, custom algorithms (like machine learning models) directly inside a traditional Data Warehouse.
💡 The Solution: Batch Processing
Batch computing allows companies to dump tons of unstructured data, write arbitrary code to process it, and generate outputs. Examples include: Building search indexes, Machine Learning aggregations, and ETL (Extract, Transform, Load) pipelines.

2. Distributed File Systems (DFS)

To process massive data, you first need to store it. You can't fit petabytes of data on a single laptop hard drive.

  • A Distributed File System acts exactly like a normal file system (folders, sub-folders, files), but it is spread across hundreds or thousands of different computers (nodes).
  • Redundancy & Replication: Data is copied across multiple machines. If one node catches fire, your data is safe.
  • Hadoop (HDFS): The industry-standard open-source version, originally based on the Google File System.

3. MapReduce: The Core Concept

MapReduce is a simple programming model designed to run on top of Hadoop (HDFS). To use it, developers only need to write two functions: a Mapper and a Reducer.

Key Principles:

  • Immutable Inputs: MapReduce never modifies your original files. It reads them and outputs brand new files.
  • Data Locality (Crucial!): Moving data over a network is extremely slow and expensive. Instead, MapReduce moves the code to the data. Mapper functions are run directly on the physical node where the input file lives.
  • Fault Tolerance: If a single server crashes (or the task is killed because the server is needed for higher-priority work), MapReduce only restarts that specific node's task, not the entire massive job.

  • Step 1 & 2: The Mapper breaks the file into records and extracts a Key and a Value.
  • Step 3 (Shuffle/Sort): A Hash Function is applied to the Key. This ensures that every record with the same exact Key gets sent to the exact same Reducer. Keys are sorted locally.
  • Step 4: The Reducer receives the sorted data. Because all identical keys are grouped together sequentially, it requires very little RAM. It iterates over the values for one key, runs custom logic, and saves.

5. Joins in MapReduce

Often, you need to associate data from two different input files. Doing this via a live database network call is too slow. Here are three ways MapReduce handles joins locally:

A. Sort Merge Joins (Reduce-Side)

Mappers process both large datasets. Because they use the same hash function, identical keys from both datasets are routed to the same Reducer, which merges them together.

// Mapper 1 (Browser Data) & Mapper 2 (Age Data) output:
jordan: youtube.com
jordan: 21

// Reducer Merges them into:
jordan: youtube.com, 21

B. Broadcast Hash Joins (Map-Side)

Used when one dataset is massive, but the second dataset is tiny. You don't need a Reducer. The entire tiny dataset is loaded into the RAM (as a Hash Map) of every single Mapper. The join happens instantly on the Mapper node.

C. Partitioned Hash Joins (Map-Side)

If both datasets are too large for RAM, but they are partitioned the exact same way (e.g., all 'A' names are in partition 1 for both sets). The Mapper only loads the matching partition of the smaller dataset into memory, skipping the Reducer phase entirely.

6. Pitfalls of MapReduce

While revolutionary, MapReduce has significant performance flaws when chaining multiple jobs together (Job A → Job B → Job C).

⚠️ The 3 Major Flaws:
  1. Stragglers: Job B cannot start until Job A is 100% finished. If one node in Job A is slow/glitchy, the entire pipeline is frozen waiting for that one "straggler".
  2. Materialized Intermediate State (Disk I/O): MapReduce writes the output of every single job to the physical hard drive (Disk). Writing to Disk is incredibly slow and a massive waste of resources if those files are only being used to pass data to the next job.
  3. Unnecessary Sorting: MapReduce always sorts keys. If Job B doesn't actually need sorted data, the system wastes massive CPU power doing it anyway.

7. DataFlow Engines (Apache Spark)

To solve the pitfalls of MapReduce, the industry created modern DataFlow engines like Apache Spark. Instead of isolated mappers and reducers, Spark uses "Operators".

Legacy MapReduce

Treats every job as a black box. Reads from disk, processes, writes to disk. Slow and requires heavy I/O operations.

Modern Spark

Understands the entire data flow from beginning to end before executing. Maximizes data locality and optimizes operations perfectly.

Why Spark is Superior:

  • In-Memory Processing: Spark does not write intermediate states to disk. It keeps the data flowing in RAM (memory), making it exponentially faster.
  • Fault Tolerance Trade-off: Because data is kept in RAM, a server crash wipes the data. MapReduce could just read the hard drive, but Spark handles crashes by recomputing the lost data from the last checkpoint.
  • Deterministic Requirement: Because Spark relies on recomputing data upon failure, your functions must be deterministic (the same input must always yield the exact same output, without time-based randomness).

8. Conclusion & Next Steps

Batch processing is fundamentally about taking unformatted data on a Distributed File System and transforming it to find insights. Dataflow engines like Spark have proven vastly superior to MapReduce by keeping data in memory and avoiding unnecessary disk writes and sorting.
