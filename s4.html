Database Architecture Study Guide

Leaderless Replication

Context: This is the third type of database replication covered in this series (following Single-Leader and Multi-Leader). In this continuous feed, we break down exactly how Leaderless Replication works, how data is kept consistent, the mathematical formulas behind quorums, and the edge-case flaws inherent in this system.

1. Core Overview

Unlike Single or Multi-leader replication where writes are bottlenecked through specific leader nodes, Leaderless replication completely removes this hierarchy.

  • Any replica can accept a write directly from a client.
  • Writes and Reads are sent to all nodes in parallel over the network.
  • Once a predefined threshold of nodes returns a success value, the client is notified that the operation was successful.

The Trade-off: This structure requires a massive amount of simultaneous network calls compared to leader-based methods.

2. Keeping Data Up to Date

Because writes are sent to many nodes over a network, some requests will inevitably fail or be delayed (e.g., Node 1 gets the write, but Node 2's network packet drops). Databases use two primary mechanisms to fix out-of-sync data:

Mechanism A: Anti-Entropy

A continuous background process that looks at multiple nodes, compares the stored version numbers of specific keys (in a Key-Value store), and replaces outdated values on lagging nodes with newer data.

Mechanism B: Read Repair

This happens on the fly during a read operation. When a client reads from multiple replicas in parallel, it finds the most up-to-date piece of data and immediately propagates it back to the replicas that had outdated data.

3. Quorums (The Math of Consistency)

In the Read Repair example, how can we guarantee that at least one of the replicas we read from actually *has* the Version 20 data? If we only randomly read from DB2 and DB3, we wouldn't even know Version 20 existed. To solve this, Leaderless databases use Quorums.

The Quorum Variables:

  • n = total number of database replicas.
  • w = minimum nodes written to for a successful Write.
  • r = minimum nodes read from for a successful Read.

The Golden Rule: If w + r > n, the read and write sets must overlap. At least one read node will have the newest copy.

Standard Configuration

Typically, n is set to an odd number (e.g., 7). You calculate w and r using: w = r = (n + 1) / 2.

For n = 7, w = 4 and r = 4. This setup can tolerate up to 3 node failures ((n - 1) / 2) because you only need 4 out of 7 nodes online to satisfy reads and writes.

4. Why Quorums Are Not Perfect

Quorums provide an illusion of strong consistency. However, there are 4 critical edge cases where the system falls out of sync:

1. Failed Writes Do Not Roll Back
If w=4, but your write only reaches 3 nodes, the client receives a "Write Failed" error. However, the database does not undo/delete the data from those 3 successful nodes. This unconfirmed data is now floating in the cluster and could be accidentally propagated via Read Repair.
2. Node Restoration Drops the Quorum
If a node with the newest data crashes, and you restore it from an older backup, it loses its up-to-date versions. This effectively lowers the true w count in the cluster without the system realizing it, meaning future reads might miss the newest data.
3. Sloppy Quorums & Hinted Handoff
Example: A Facebook cluster has 99 Profile nodes (w=50, r=50). All 99 nodes crash! To maintain app uptime, the system temporarily routes Profile data writes to Facebook Messaging nodes.

Because data isn't on its native nodes, Profile reads will return blank/old data (hence "sloppy"). Once the 99 Profile nodes reboot, a background process called Hinted Handoff transfers the profile data from the messaging nodes back to the profile nodes.

4. Write Conflicts (Race Conditions)

Because network speeds vary, concurrent writes can arrive at databases in different orders.

Conclusion Summary

Pros

  • High Write Availability: Works incredibly well for cross-datacenter solutions.
  • Low Latency Writes: If configured correctly, a write only has to reach nodes within a single local datacenter to be considered successful.

Cons

  • Slower Reads: Must query multiple nodes across a network and potentially execute Read Repairs.
  • Write Conflicts: Requires complex logic to resolve race conditions.
  • Illusion of Consistency: Due to edge cases, Quorums do not guarantee a perfectly consistent state 100% of the time.
