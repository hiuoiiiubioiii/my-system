1. Introduction & Context

🎙️ Instructor's Audio Intro

  • Video Motivation: Much of the information online about HBase is plagiarized or lacks depth. This guide provides clear, accurate architectural depth.
  • Historical Context: HBase is heavily modeled after Google's Bigtable (which runs on the Google File System). HBase essentially copies this architecture but builds it on top of Hadoop's HDFS.
2. Background & The HDFS Problem

🖥️ Slide Content

  • HBase is a wide-column storage database designed for transactions or analytics, built on HDFS.
  • HDFS provides high latency writes/reads (direct from disk) and only allows sequential writes.
  • HBase uses an LSM Tree architecture to mitigate this, providing low-latency writes/reads for random accesses.

🎙️ Instructor's Deep Dive

  • The Problem with HDFS: HDFS is designed for massive files and batch processing. It writes sequentially to disk, making random reads and writes very slow.
  • The HBase Solution: To fix this, HBase sits on top of HDFS and uses an LSM (Log-Structured Merge) Tree and SSTables. It absorbs random read/write requests, organizes them in memory, and then writes them sequentially to disk, resulting in high-performance operations.

  • Each row has a single row key.
  • Columns are grouped into "Column Families" (consisting of multiple values).

🎙️ Instructor's Deep Dive

  • Comparison to Cassandra: Looks similar but functions differently. Cassandra has specific clustering keys for internal sorting.
  • Row Keys in HBase: Row keys are just single strings. To achieve sorting, you format the row key into multiple parts separated by a delimiter (e.g., PartA:PartB:PartC).
  • Design Warning: You must be very clever when designing your row keys to ensure the data sorts exactly the way you need it to upon retrieval.

🎙️ Instructor's Deep Dive

  • This architecture is nearly identical to Google's Bigtable.
  • HMaster: The central coordinator that tells clients where to go.
  • Region Servers: The workhorses handling the actual data reads and writes.
  • ZooKeeper: Used for coordination, keeping track of server health, and managing backups.
5. Master Server (HMaster)

🖥️ Slide Content

  • Stores all file metadata and locations of chunks of files (similar to HDFS NameNode).
  • Uses Range-based partitioning that splits based on row key if it gets too big.
  • Clients reach out to the master server first to figure out file locations.
  • Used with ZooKeeper to replicate its Write-Ahead Log (WAL) to support a backup master node.

🎙️ Instructor's Deep Dive

  • The Hotspotting Problem: Because HBase uses range-based partitioning, it is bad for time-series data. If your row key is a timestamp, every single new write goes to the exact same partition (the current time). This overloads one server while the rest sit idle.
  • Solution to Hotspotting: You must hash the key or alter it so writes are distributed evenly.
  • Fault Tolerance: The master server pushes its logs to ZooKeeper. If the primary master dies, a backup master reads the logs from ZooKeeper and takes over seamlessly.

🖥️ Slide Content

  • Runs as a separate thread on HDFS datanode servers.
  • Sends heartbeats to ZooKeeper so master knows they are alive.
  • Uses an LSM tree style in-memory data structure. Writes are sent here first (plus a Write-Ahead Log for safety).
  • Once MemStore is large, writes flush to sorted HFiles (SSTables) in a column-oriented format.

🎙️ Instructor's Deep Dive

  • Crucial Misconception Cleared: A Region Server is usually not a standalone machine! It is a software program running on the same physical computer as the HDFS DataNode program.
  • Read Process: Reads check the memory (MemStore) first. If data isn't there, it checks the HFiles on disk. HBase uses optimizations like Bloom Filters to speed this up.
  • Column-Oriented Advantage: Storing data by columns means if you run analytics on just one specific column, the system pulls only that data without reading irrelevant info, massively increasing throughput.
7. Replication

🖥️ Slide Content

  • Since the region node is running on an HDFS replication node, it uses the HDFS replication pipeline to synchronously replicate the HFile to other nodes.

🎙️ Instructor's Deep Dive

  • Strong Consistency: Because it relies on HDFS, writes must succeed across the HDFS replication pipeline before being marked successful.
  • Handling Failures: While one Region Server handles reads/writes for a specific chunk of data, the underlying data files are backed up across multiple HDFS nodes. If the Region Server crashes, another server can take over because the data is safe in HDFS.
8. Analytics Perspective

🖥️ Slide Content

  • Column-oriented storage makes for very high read throughput of one column.
  • Being built on HDFS allows good integration with MapReduce and dataflow engines (Spark, Tez).
  • Joins are not supported in HBase! Use batch processes to perform joins.

🎙️ Instructor's Deep Dive

  • The Hadoop Ecosystem: The biggest advantage of HBase is that your data is sitting exactly where tools like Spark and MapReduce expect it to be (inside HDFS).
  • No SQL Joins: Because it is NoSQL and unstructured, you cannot just write a SQL JOIN command. You have to write complex batch processing jobs (using multiple mappers and reducers) to manually combine the data associations.
9. Conclusion: HBase vs. Cassandra

Cassandra

  • Leaderless (Dynamo style)
  • Real-time transactions
  • Super fast writes
  • User-facing apps

HBase

  • Master-oriented
  • Great as a Data Lake
  • Batch/streaming processes
  • Fast reads/writes via LSM

🎙️ Instructor's Deep Dive

  • Cassandra: Built for user-facing applications where you need instant writes and eventual consistency without a single point of failure.
  • HBase: Won't write quite as fast as Cassandra, but it shines as a Data Lake. A Data Lake is a place to dump massive amounts of unstructured/semi-structured data to eventually run heavy analytics on later.
  • Data Lake vs. Data Warehouse:
    • HBase (Data Lake): Unstructured, No SQL, great for dumping massive data.
    • Data Warehouse: Star/Snowflake schemas, highly structured Fact and Dimension tables, meant for complex SQL querying. (If you need advanced queries, don't use HBase, opt for a SQL Data Warehouse).
