📚 Database Isolation Levels

Continuous Study Guide • Scroll to learn

🎬 The Core Dilemma

Transactions are a great abstraction for databases, ensuring data is perfectly safe. For perfect safety, a database must use Serializable Isolation (processing transactions one-by-one, strictly sequentially).

The Problem: True serializability is incredibly slow.

The Solution: Databases offer "Weaker Isolation Levels." They sacrifice perfect safety for speed, meaning the database allows certain concurrency bugs to happen in exchange for high performance.

🛑 1 & 2: Dirty Reads & Dirty Writes

Dirty Read: A client reads data written by another transaction that hasn't committed yet. If that transaction aborts, the client just read data that never truly existed.

Dirty Write: A transaction overwrites data that was written by another uncommitted transaction.

Visualizing Dirty Read Prevention
Database State: kim's_man = "Kanye"
Transaction A (Uncommitted): UPDATE kim's_man = "Pete"
Transaction B (Reads): Reads "Kanye" (Ignores uncommitted "Pete")

🛡️ The Fix (Read Committed Isolation):

For writes: Put a row-level lock on the object so only one writer can touch it at a time.

For reads: Do NOT use locks (too slow). Instead, the database remembers the old value ("Kanye") and serves it to readers until the new value ("Pete") is fully committed.

⏱️ 3: Read Skew (Non-Repeatable Reads)

This happens during long "read" queries (like analytics). While the query is running, other transactions are changing data. The long query reads different pieces of data at different points in time, resulting in an inconsistent view.

The Venmo Anomaly (Read Skew)
Start: You = $100 | Friend = $100 Total = $200
Step 1 (Query reads You): Sees $100
Step 2 (Friend transfers $50 to you): You = $150 | Friend = $50
Step 3 (Query reads Friend): Sees $50
Final Query Math: $100 + $50 = $150 Wait, where did $50 go?!

📸 The Fix (Snapshot Isolation):

Every transaction gets a monotonically increasing Transaction ID (TxID). When reading, a transaction can only see data written by TxIDs less than or equal to its own.

Snapshot Isolation in Action
Record Value Written By TxID
Jordan "brollic" 14
Bieber (Old) "uglier_than_jordan" 12
Bieber (New) "ok_fine_good_looking" 19

👉 Transaction 15 wants to read both.
It reads Jordan (14 < 15).
It cannot read the new Bieber (19 > 15), so it reads the old Bieber (12 < 15).

💸 4: Lost Updates

The classic "Read-Modify-Write" bug. Two threads read an object, modify it, and write it back at the exact same time. One thread overwrites the other, destroying data.

Lost Update Example
Initial State: Jordan_net_worth = $1,000,000
Thread 1 & 2 Read: Both read $1,000,000
Thread 1 & 2 Modify: Both add $1,000,000 (Math: $1M + $1M = $2M)
Thread 1 & 2 Write: Both write $2,000,000 to the DB.

Result: $2M instead of $3M. You've been conned out of a million dollars!

🛠️ Solutions for Lost Updates:

  • Atomic Writes: DB built-in math functions (e.g., `compare_and_set`).
  • Explicit Locks: App code uses `FOR UPDATE` in SQL (can cause bugs if developers forget to release locks).
  • Automatic Detection: DB uses Snapshot Isolation to detect if the row changed between your read and write, forcing a rollback/retry.

Note: In distributed multi-leader databases, these fail. You must store conflicts as siblings and use conflict resolution logic (like CRDTs).

👯 5: Write Skew

Two transactions read the same set of objects to check a condition (predicate). They both decide it's safe to write, but they modify different objects in the set, accidentally breaking the rule.

The VIP Table (Max 3 People)
IDNameTable
1Jordan1
2Lindsay Lohan1

Timeline:
1. Thread A & B read Table 1. They see 2 people. Rule: Max 3.
2. Thread A decides it's safe to add Kate Upton.
3. Thread B decides it's safe to add Megan Fox.

Result: Kate and Megan are both added. Table 1 now has 4 people. Invariant broken!

🔒 The Fix (Predicate Locks):

You apply a lock to all the rows you read to make your decision. Thread B is blocked from reading the VIP table until Thread A finishes inserting Kate.

👻 Phantoms & Materializing Conflicts

A "Phantom" is a Write Skew bug where both transactions are trying to create a brand new row. The problem? You cannot put a Predicate Lock on a row that doesn't exist yet!

Example: Two people check if a meeting room is empty at 2:00 PM. No row exists. They both insert a booking row. Overbooked!

Materializing Conflicts
Without Materialization: Locking fails because 2:00PM row = NULL
With Materialization: Background script generates empty dummy rows for the whole week.
Result: Transactions can now lock the "empty" 2:00PM dummy row!

📝 Final Summary & Instructor Advice

  • Read Committed: Prevents dirty reads & writes. Fastest, least safe.
  • Snapshot Isolation: Prevents read skew via TxIDs. Detects lost updates. Great middle ground.
  • Predicate Locks & Materialization: Prevents write skew and phantoms.

💡 The Ultimate Takeaway:

Using weak isolation keeps databases fast because locking is minimal. However, if your application can afford the performance hit, simply using true Serializable Isolation is best. It entirely removes the need for developers to reason about (and potentially mess up) concurrency logic!
