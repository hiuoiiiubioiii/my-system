1. The Threat of Unreliable Nodes

Even when you control every single server, database, and node in your distributed system, it is highly possible for individual nodes to become unreliable.

These nodes usually have good intentions. They are trying to work with the system, but they might be operating on outdated information. They believe something to be true, while the majority of other nodes disagree.

The "Rule of One" Problem:
Unreliable nodes become a massive threat when your system requires strictly one of something. For example:
  • One Leader: In a single-leader replication setup.
  • One Distributed Lock: Only one process should hold a lock to edit a file at a time.
  • One Unique Username: Ensuring only one account can claim a username like "poonslayer69".

2. Why do Nodes Fail? "Process Pauses"

How does a node get out of sync in the first place? Often, it's due to a Process Pause. A node will be running a process, and suddenly it gets preempted by a "stop the world" background task. The active process freezes completely until the background task finishes.

Three common examples of Process Pauses:

  • Garbage Collection Algorithms: Common in languages like Java. Java uses "stop the world" garbage collection. It will literally freeze all ongoing processes (sometimes for 10 to 15 seconds!) to clean up memory.
  • Virtual Machine Migration: If a VM is physically moved from one host server to another, all processes must pause during the transit.
  • User Actions: Something as simple as a user closing their laptop lid or hitting the power button on their mobile phone.

3. The Disaster Scenario (Without Fencing)

Because distributed systems don't have shared memory, they rely on "Leases" for locks. A lease grants a node a lock, but it expires after a certain amount of time.

If a node pauses while holding a lease, disaster strikes. Below is the visualization of a Split-Brain / Corrupted Data scenario.

Visualization: Unfenced Lock Collision
What just happened?

Client 1 was about to execute code, but experienced a 15-second GC pause. During this pause, its lease expired. Client 2 claimed the lock and wrote its data. When Client 1 woke up, it didn't check if its lease was still valid. It immediately wrote its data, overwriting Client 2 and corrupting the storage.

If this was a leader election, the nodes would assume the paused leader is dead and elect a new one. When the old leader wakes up, you get a Split-Brain situation where two nodes think they are the single leader!

4. The Solution: Fencing Tokens

To fix this, we implement Fencing Tokens.

The Core Idea: Every time the consensus service (majority of nodes / Lock Service) makes a decision, it assigns a monotonically increasing ID (a number that strictly goes up). This ID is passed along with every write request to the storage.

Visualization: Fencing Tokens Rejecting Outdated Nodes
How it saves the system:
  1. Client 1 requests the lock and receives it along with Token 33.
  2. Client 1 enters a GC Pause, and its lease expires.
  3. Client 2 requests the lock, receives it, and is granted the next ID: Token 34.
  4. Client 2 writes its data, passing Token 34. The storage system remembers: "The highest token I've seen is 34."
  5. Client 1 wakes up and attempts to write its data, passing Token 33.
  6. The storage service checks the token and says: "Wait, 33 is lower than 34. You are an outdated node." It rejects Client 1's request, keeping the data perfectly safe!

5. Conclusion

Because process pauses and unreliable client nodes are an unavoidable reality in distributed systems, we can only trust decisions made by a majority of nodes in our cluster. A single node can be faulty without even realizing it.

Fencing tokens are an incredibly easy-to-implement safety mechanism. They act as a guard rail ensuring that once a consensus is reached by the system, no single faulty, out-of-sync node can overwrite it, completely protecting you against multiple lock holders and split-brain scenarios.
