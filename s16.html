🌊 Background & Core Concepts

Often, we will have lots of data coming in that we want to process asynchronously in the background.

  • Streaming works on an unbounded set of data (infinite flow), as opposed to just a limited set of files/records (which is Batch Processing).
  • Messages are created by a Producer node, and eventually handled by at least one Consumer node.
🎙️ Instructor's Insight While these messages can be sent directly from producer to consumer (via protocols like UDP multicast), it is common practice to use an intermediary called a Message Broker to buffer these messages in a centralized place. This protects data if a consumer crashes!

📦 Message Brokers & Delivery Patterns

A Message Broker is a special type of database that handles streaming. Producers push to it, and consumers pull from it. Some brokers keep messages durably, while others delete them after successful consumption.

Two Main Delivery Patterns:

  • Fan-out Send the exact same message to all consumers.
  • Load Balancing Send one message per consumer to maximize throughput and processing speed.

The "In-Order" Problem

Even though a queue holds messages in the order they were received, if a consumer crashes or takes a while to handle a message, the broker may have sent the next message to a different consumer which processed it first!

Messages can reliably be delivered in order, but it comes with a performance penalty.

⚡ Types of Message Brokers

1. In-Memory

Can be done on something like a Redis instance.

  • No persistence (data held in RAM).
  • Messages are permanently deleted the second a consumer acknowledges them.
  • Best for: Maximum speed, when messages take a long time to process, and their order does not matter.

2. Log Based (On Disk)

Messages sent to an append-only log on disk.

  • Highly persistent and replicable.
  • Keeps messages around after processing so you can potentially replay them.
  • Best for: Fault tolerance and ordering message processing within a partition.
🎙️ Deep Dive: Log-Based Partitions The Log can be partitioned to improve performance. However, there is a golden rule: One consumer per partition. This ensures ordered delivery. The downside? A single hard-to-process message stops the rest of the partition from proceeding forward!

📊 Common Uses: Logging & Metrics

We often want to aggregate certain events into time windows, such as application logs or metric loads. This is challenging because an event can arrive to a stream any amount of time after the window closes due to out-of-sync server clocks.

Types of Fixed Time Intervals:

  • Tumbling Windows: Non-overlapping intervals of fixed length (e.g. every minute starting at 0 seconds).
  • Hopping Windows: Overlapping intervals of fixed length (e.g. all 5-minute intervals starting at 0 seconds). You aggregate tumbling windows to create hopping windows.
  • Sliding Windows: Fixed duration, but no set start point. Created using an in-memory buffer of events, dropping them once they age out.

🔄 Change Data Capture & Event Sourcing

Change Data Capture (CDC)

Writes to a database are converted into a stream. They can then be consumed by derived data sources (caches, search indexes, data warehouses) to keep them up to date without network-heavy custom code.

Assuming logs are persistent, they can be compacted by only holding the most recent value of keys.

Event Sourcing

Similar to CDC, but instead of streaming database writes, all events of the user are put into a streamed, append-only log.

  • Pros: Highly future-proof. You aren't stuck with a specific data schema and can derive entirely new databases from it.
  • Cons: The log cannot be compacted, because everything is an "event", not a final state write.
🎙️ Instructor's Example: Event Sourcing Instead of a database holding `watch_count: 42`, Event Sourcing logs specific events:

"Jordan clicks the watch button. Jordan is the 40th watcher of Kate Upton's post. Jordan is the 41st watcher of Kate Upton's post. Jordan is the 42nd watcher of Kate Upton's post."

🔗 Stream Joins

Just like batch processing, we need to associate data. To reduce network calls to an actual database, we often keep some amount of "state" local to the stream broker.

  • Stream-Stream Joins: Joining two live event streams (e.g., a search query stream + a click stream). The broker keeps a local index. If a click arrives within a certain time window, it joins it with the search index.
  • Stream-Table Joins: Enriching a stream event with database data. To avoid making a network call for every single event, you subscribe to the CDC of the database table and maintain a live local copy of the table inside the stream broker!
  • Table-Table Joins: Subscribing to multiple sources of CDC to keep multiple local tables up to date. The result of the joined streams acts as a pre-computed cache of an actual SQL query.

🛡️ Fault Tolerance (Exactly Once Processing)

We want to ensure that each message is processed exactly once (no less, no more).

1. Ensuring processed AT LEAST once:

  • Occasionally checkpoint stream state to disk to restart from the most recent state on crash.
  • Run microbatches: let a few stream events accumulate and run a batch workload on them (batch jobs are easy to restart if they fail).

2. Ensuring NOT processed more than once:

  • Use Idempotence: Keep track of seen messages via a unique message ID (idempotency key) to ensure they are ignored if sent twice.
  • Use Atomic Transactions (like Two-Phase Commit). This guarantees delivery and processing simultaneously, but comes with a massive penalty—it is much slower.
