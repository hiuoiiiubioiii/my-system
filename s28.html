Time Series Databases

Complete Architecture & System Design Notes

🎬 Introduction & Core Premise

🧠 Instructor's Premise

The goal of this lecture is not to teach you how to use a specific database like TimescaleDB or InfluxDB. Instead, it focuses on the abstract design principles and architecture that all time-series databases share. This is highly useful for system design interviews, where you are expected to understand the *concepts* rather than specific software implementations.

1. Background

👁️‍🗨️ Slide Notes

  • In many applications, there is a pattern of “write-once, read many times” data inserts corresponding to a range of time - things like logs, sensors, or any other consistent data being ingested from a stream follows this rule.
  • As a result, many databases specifically tailored for this use case, such as InfluxDB, TimescaleDB, and Apache Druid have been created. They allow both great throughput for reads and writes on time series data.
🧠 Instructor's Insights
  • The Universality of Time-Series Data: Almost every modern company will eventually have to store time-series data. This usually takes the form of server logs or sensor telemetry.
  • The Problem with Traditional DBs: Time-series data arrives as a continuous, massive stream. Traditional relational databases aren't optimized for this specific "firehose" ingestion.
  • The Solution: Specialized databases were built specifically to handle incredibly high write/read throughput specifically for ordered time data.

2. Time Series Operations

👁️‍🗨️ Access Patterns

  • Writes: Primarily inserts to a recent time interval, adjacent values likely similar.
  • Reads: Generally from the same timestamp/data source combo over one column of data. Usually looking at the most recent time interval.
  • Deletes: Common scenario is bulk deletion of data older than a specific date prior to the present.

🧠 Designing around Access Patterns

To design a good system, you must design around how the data is actually accessed:

  • Write Patterns: You are writing to the end of the timeline. You aren't inserting data from 7 days ago; you are inserting data from right now. We never update this data; we just "write once."
  • Read Patterns: Users usually want to see a specific metric (one column) from a specific source, over a recent timeframe (e.g., "Show me Server A's CPU usage over the last 3 hours").
  • Delete Patterns: People don't keep log data forever. The standard practice is bulk deletion based on a threshold (e.g., "Delete everything older than 6 months").

3. Optimizing Writes

👁️‍🗨️ Slide Notes

  • Should likely build a compound index first sorted on source ID, and then within that sorted on timestamp.
  • This allows writes from the same data source over similar intervals of time to be on the same node.
🧠 Instructor's Insights

Compound Indexing & Sharding: The most natural way to organize this data is a tuple: (Data Source ID, Timestamp). If you build your database using this compound index, it guarantees that data coming from the same source at roughly the same time will be grouped together. If you are sharding across multiple servers, this ensures high-speed localized writes, rather than scattering a single sensor's data across a dozen different machines.

4. Optimizing Reads (Columnar Storage)

👁️‍🗨️ Slide Notes

  • Storing data in a column-oriented manner is much faster.
  • Timestamps and metrics are sequential/similar and can be compressed a ton.
  • Having all column values in one file reduces disk I/O and allows faster aggregations (we generally only need 1 or 2 columns at a time for graphing).

🧠 Instructor's Insights
  • Why Column-Oriented? Standard databases store data in rows. But users usually only want to graph one specific metric at a time. If stored in columns, the DB only pulls that specific column from the disk, drastically reducing Disk I/O.
  • Massive Compression: Because adjacent data points are highly similar (e.g., timestamps change by 1 second, temperature stays at 72.1° for hours), this data is highly compressible (e.g., Run-Length Encoding). This shrinks the data footprint immensely.

5. The "Chunking" Concept

👁️‍🗨️ Slide Notes

  • Even within one node, it is better to treat each (source, time interval) tuple as its own chunk table (abstract these away via a large "Hypertable").
  • Since most writes go to only a couple of these, we achieve much better performance by caching their entire index in memory.
  • Otherwise we would have many index pages occasionally swapped in and out of memory, incurring processing overhead.

🧠 The Memory/RAM Trick

This is the most important architectural concept:

  • The Problem with Giant Tables: If you put all time-series data into one massive table, the B-Tree index becomes so large it can no longer fit in the server's RAM. The computer has to constantly swap index pages between the slow Hard Drive and fast RAM (disk-swapping/thrashing), creating massive lag.
  • The "Chunk" Solution: Time-Series DBs fix this by breaking the giant conceptual table into many tiny, physical "Chunk tables" defined by source and time block (e.g., "Sensor A, Jan 1st").
  • Because 99% of reads and writes happen on recent data, the DB only needs to keep the indexes for the recent chunks in RAM. The older chunks' indexes stay safely on the hard drive. This eliminates disk-swapping.

6. Optimizing Deletes

👁️‍🗨️ Slide Notes

  • Breaking the main table into chunk tables greatly optimizes delete performance.
  • In LSM trees, a delete is a write (adds a tombstone) and requires waiting for compaction. In B-trees, you must traverse the tree to set pointers to null.
  • Instead, to delete a bunch of old data all at once, we can just delete the corresponding chunk files/indexes.

🧠 Instant Deletion

Why normal deletes are slow: Deleting a million old rows requires a million individual operations (writing tombstones or nullifying pointers).

Why Chunking makes deletes instant: Because our old data is already organized into neat, isolated files based on time intervals, we completely bypass traditional DB delete mechanisms. The DB simply asks the operating system to delete the entire chunk file. It's a single, nearly instantaneous operation with zero database overhead.

7. Conclusion & Interview Summary

👁️‍🗨️ Slide Notes

Using a database specifically made for time series allows huge gains in performance by:

  • Creating separate indexes for each data source and time interval combination.
  • Using Column-oriented storage for fast aggregations and compression.
🎯 Interview Cheat Sheet

If you get a system design interview question about dealing with logs, telemetry, or metrics, bring up these specific concepts to impress the interviewer:

  1. Acknowledge the access pattern: High write volume, appending to the end, querying recent single metrics, bulk dropping old data.
  2. Suggest Columnar Storage: Explain how it reduces storage size via compression and speeds up single-metric graphing by reducing Disk I/O.
  3. Explain the "Chunking" Architecture: Detail how breaking the index into time-and-source-based chunks prevents memory bloat, keeps active indexes in RAM, and makes bulk data deletion instantaneous file drops.
  4. Distributed Scaling: Note that this chunking design scales beautifully, as chunks can easily be hashed and distributed to different nodes.
