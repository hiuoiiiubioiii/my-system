Two-Phase Commit (2PC)

Complete Study Notes & Visualizations from the Distributed Systems Lecture

1. Introduction & Core Concepts

Welcome to the study feed for Consensus Algorithms in Distributed Systems. In this lesson, we explore Two-Phase Commit (2PC), a simple consensus algorithm used to solve the problem of atomic commits.

Definitions to Know:

  • Atomic Commit: In a distributed transaction touching multiple partitions (nodes), every single write must either succeed completely or fail completely (roll back). There is no middle ground.
  • The Challenge: Achieving atomicity on a single machine is easy. But when a transaction spans multiple nodes over a network (where power can fail or connections drop), all nodes must agree on the outcome. This requires consensus.

2. Why Do We Need Atomic Commit?

Atomic commits prevent databases from getting out of sync due to partially completed transactions.

  • Cross-Partition Transactions: e.g., On LinkedIn, writing a job listing that spans multiple partitions. All must update simultaneously.
  • Global Secondary Indexes: Every time you write to a primary partition, the secondary index (living on another node) must also be kept in sync.
  • Derived Data Consistency: Keeping primary databases synced with data warehouses or caches.

💡 Instructor's Real-World Example:

Imagine sending an email notification every time a database write is successful. If the write fails but an email is still sent, and then the system auto-retries the write and sends another email, the user gets duplicate emails. Atomic commit ensures the email only triggers if the write is 100% committed everywhere.

3. How Two-Phase Commit Works

The architecture consists of a Coordinator Node (acts as the single leader/manager) and two or more Database Partitions (participant nodes).

Phase 1: The Prepare Phase

The Coordinator sends a "Prepare" request to all nodes. The partitions process the request and respond.

🤝 The "OK" Promise:

When a node replies with "OK", it means: "I am prepared to do this. No matter what happens—disk failure, network loss—I guarantee I will commit this transaction as soon as you tell me to."

Phase 2: The Commit/Abort Phase

The Coordinator tallies the responses from Phase 1:

  • Success: If ALL partitions respond with "OK", the Coordinator sends a result=commit request to all nodes.
  • Failure: If even ONE partition fails to respond or responds with an error, the Coordinator sends a result=abort request, and everything rolls back.

4. Under the Hood: Why It Works

How does the system keep track of all this without making mistakes?

  • Unique Transaction IDs: The Coordinator assigns a monotonically increasing, unique ID to each transaction so nothing gets mixed up.
  • Local ACID Properties: Each participant node executes its portion of the transaction locally, giving it local atomicity and serializability.
  • The Commit Point: The exact moment a node replies "OK" during Phase 1. It is legally bound by the algorithm to hold that transaction ready.
  • Coordinator Write-Ahead Log: Before sending the final Commit/Abort, the Coordinator writes its decision to an internal log. If the Coordinator crashes right after deciding, it can read its log upon reboot and finish sending the commands.

5. The Fatal Flaws of 2PC

🚨 High-Yield Concept: Two-Phase Commit is NOT fault-tolerant!

Flaw 1: Partition Failure

If a participant partition fails after the prepare phase (after it said "OK"), the Coordinator must try to reach it forever until it comes back online.

Flaw 2: Coordinator Failure (Single Point of Failure)

If the single Coordinator node crashes after nodes replied "OK", but before sending the final Commit/Abort command, the entire system breaks down.

  • "In Doubt" Nodes: Partitions are stuck. They promised to hold the transaction, but don't know the final verdict. They cannot talk to each other to figure it out.
  • System Freezes: Because nodes are "in doubt," they hold exclusive locks on the relevant rows (via Two-Phase Locking). Other transactions cannot read or write to those rows. The database effectively halts.

6. Heterogeneous vs. Database Internal 2PC

The instructor clarifies how 2PC is actually implemented in practice:

Heterogeneous Transactions (The XA API)

This is 2PC running across completely different types of software (e.g., a MySQL DB and an Apache Kafka queue). It uses a universal API called XA. Because the systems are so different, there is no shared protocol for optimizations. It is inherently slow and blocking.

Database Internal Transactions

This is 2PC running within the same software ecosystem (e.g., a distributed PostgreSQL cluster). Because the nodes run the same code, they can share lock info, detect deadlocks, and use optimistic concurrency controls (like Serializable Snapshot Isolation), making it much more efficient and less blocking.

7. Summary & What's Next

To summarize: 2PC uses a single Coordinator to ask all nodes if they can commit. If yes, everyone commits. If no, everyone aborts. While it guarantees consistency, it sacrifices availability and lacks fault tolerance.

Foreshadowing: Because 2PC freezes if the Coordinator crashes, modern distributed systems use superior consensus algorithms like Paxos, Raft, or ZAB.

⏩ Why are Raft/Paxos better?

Unlike 2PC which requires 100% agreement, these algorithms use Quorums (a majority vote). If a minority of nodes crash, the system continues to function perfectly, providing true fault tolerance.
