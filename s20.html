Study Notes: Coordination Services

A complete vertical feed covering the role, architecture, and consistency trade-offs of coordination services in distributed systems.

1. The Background & The Problem

In any distributed system containing a cluster of nodes, there is a strict need for a centralized, reliable source of truth for shared state and configuration metadata.

What is "Shared State"?

  • IP Addresses: Where to route traffic for specific nodes in the cluster.
  • Partitions: Which data partitions currently reside on which nodes.
  • Liveness Tracking: Which nodes are alive and which have crashed.
  • Distributed Locks: Preventing multiple nodes from modifying the same resource simultaneously.

Instead of building complex consensus logic from scratch, engineers use existing Coordination Services inside their tech stack. The most popular ones are Apache ZooKeeper and etcd.

Crucial Workload Characteristic: Coordination services are designed specifically for read-heavy workloads (e.g., ZooKeeper is optimized for a 10:1 read-to-write ratio). They are not meant to be general-purpose databases.

2. What Exactly Are Coordination Services?

By definition, they are highly available (replicated) Key-Value stores built on top of a consensus layer.

  • File System Vibe: In ZooKeeper, data is stored similarly to a file system (where files can have child files), but fundamentally, it acts as a key-value store.
  • The Consensus Layer: They utilize algorithms like Paxos, Raft, or ZAB (ZooKeeper Atomic Broadcast) to maintain a shared, replicated log across servers.

Why you don't use them as a standard Database:

Writes are extremely slow. Because of the consensus layer, every single write operation must be routed through the Leader node. The Leader then has to broadcast the write to the followers and wait for a quorum (a majority) of nodes to acknowledge it before committing. You cannot use this for high-write-volume tasks.

3. Reads & Eventual Consistency

Because coordination services use consensus algorithms, people often mistakenly assume that reads are automatically Strongly Consistent.

The Reality: To achieve their signature high read throughput, coordination services default to Eventual Consistency.

Why? Because you are allowed to read from any replica in the cluster, not just the leader. Since replication takes time to travel over the network, reading from a follower replica means you might receive outdated (stale) data.

4. Preventing "No Monotonic Reads"

A Monotonic Read anomaly happens when a client makes a read and time appears to move backward.

Example: You read from an up-to-date replica and see value "X". Then, you request data again but hit a lagging replica, and it returns an older value "W".

The Solution: Replicated Log IDs

Coordination services avoid this using the IDs (positions) inside their replicated log:

  1. Every time a client makes a read/write, the server replies with the data AND the ID of the last position in the replicated log it has seen.
  2. The client stores this ID locally.
  3. If the client asks another replica for data, the replica compares its own log length against the client's ID.
  4. If the replica's log is shorter (it is lagging behind the client), it refuses to return data. It either makes the client wait until it catches up, or tells the client to try a different replica.

5. Ensuring Predicate Validity

What is a Predicate? It is information you read from a database before deciding to make a subsequent read or write.

🩺 The Doctor Example:
Imagine you are a doctor at a hospital. Rules state you can only clock out if another doctor is in the room. You check the system (read the predicate), see "Doctor B" is there, and proceed to click "clock out".

But what if "Doctor B" clocked out a millisecond before your write went through? Your predicate is now invalid, and the hospital has zero doctors.

The Solution: "Watches"

In coordination services, you can attach a "watch" to any key or file you read. The replica tracks this watch. If another client changes that key before your transaction finishes, the coordination service sends you a notification/event informing you that your read is outdated.

You can then cancel your write and retry the process. (Instructor notes this is conceptually similar to Serializable Snapshot Isolation or optimistic concurrency control).

6. Achieving Strong Consistency

While default reads are eventual, sometimes Strong Consistency is strictly required (e.g., you cannot route traffic to an IP address that no longer exists). Coordination services offer ways to force strongly consistent reads, though it severely hurts performance.

Method 1: Read from the Leader

Because the leader handles all writes, its data is always perfectly up to date.
The Cost: The leader is already bogged down with massive network calls coordinating writes across the cluster. Forcing all reads to the leader destroys the system's high-read throughput capability.

Method 2: Sync

  • From a client node, write the special command "sync" into the replicated log.
  • Because it's a write, it goes to the leader. The leader responds with the exact current up-to-date ID of the log.
  • The client takes that ID and makes a read from any replica.
  • The replica will only return data if the sync operation has been fully propagated into its local log.

The Cost: You are still generating a write operation (the sync) which puts excess load on the leader.

7. Quorum Reads & The Race Condition

Since Methods 1 and 2 overload the leader, an area of active research is Quorum Reads.

The Theory: Consensus algorithms require a majority of nodes to accept a write. Therefore, if you read from a majority (quorum) of nodes, math dictates that at least one node must have the most up-to-date write. This bypasses the leader entirely!

The Reality (Why it's not perfect): It is susceptible to a specific race condition where the quorum read occurs between the "accept" phase and the "commit" phase.

What happens here?
1. The Leader (R1) processes the write x=4. It sees R2 and R3 have accepted the write into their logs.
2. The Leader commits x=4 locally and sends a message telling R2 and R3 to commit.
3. THE GLITCH: Before the commit message arrives, our Quorum Read queries R2 and R3.
4. Because x=4 is uncommitted on both, both replicas return the last committed value: x=3. We get stale data despite reading a quorum!

Proposed Fixes: The system could check the uncommitted write-ahead log. If it sees `x=4` uncommitted, it tells the client to wait and retry. However, there's no 100% guarantee `x=4` will successfully commit (e.g., the leader could crash), making this an imperfect research area.

8. Conclusion

Coordination services act as the "brain" for metadata in modern distributed data storage systems.

  • Contrast with Gossip Protocols: Unlike Gossip Protocols (where nodes randomly whisper data to each other), Coordination Services act as a strict, centralized authority built on a consensus algorithm.
  • The Golden Rule: They are built for read-heavy workloads and default to Eventual Consistency out of the box to maintain high performance. While there are ways to achieve strongly consistent reads, doing so comes at the cost of significantly reduced performance.
