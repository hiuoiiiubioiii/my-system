Multi-Leader Replication

Episode 3 Study Notes • Continuous Feed

1. Architecture Overview

Unlike single-leader architectures where all writes funnel into one node, Multi-Leader Replication allows many nodes to accept writes simultaneously.

When a leader accepts a write, it acts as a client to the other leaders, syncing the new data across the network. This is incredibly powerful for global applications where you want a database geographically close to the user to reduce latency.

2. Replication Topologies

How do the leaders actually communicate? There are three main network patterns (topologies) used to sync writes across multiple datacenters.

Circular

Nodes are connected in a ring.

Poor fault tolerance. If one crashes, the chain breaks.

Star

One central node connects to all others.

If the center node crashes, replication stops completely.

All-to-All

Every node is directly connected.

Best fault tolerance, but causes Race Conditions.

3. The Core Tradeoffs

Why choose this architecture over a standard single-leader database?

  • Global Reliability: Having a leader in each datacenter means a massive regional outage won't break your application.
  • Write Scalability: You are not limited to the write throughput of a single machine.
  • Write Conflicts: The biggest problem. If two users update the exact same row in two different datacenters simultaneously, the databases will conflict when they try to sync.

4. Conflict Resolution Strategies

Because conflicts are inevitable in a Multi-Leader setup, the database must have rules on how to handle them.

1. Conflict Avoidance

Use a routing function so all writes for a specific item (e.g., "User A's profile") always go to the exact same leader. Flaw: Fails if that specific leader goes offline.

2. Last Write Wins (LWW)

Each write gets a timestamp. When syncing, the database just keeps the write with the newest timestamp and deletes the other.

Critical Flaw: Clock Skew. Distributed servers drift out of sync. If Server A is 3 seconds behind Server B, Server B's write will "win" even if Server A's write actually happened later in real life. This leads to silent data loss.

3. On Read

The database doesn't guess. It stores both conflicting writes. When a user requests the data, the DB returns both "sibling" values, forcing the frontend application or user to manually merge them.

4. On Write (CRDTs)

The database detects the conflict and automatically runs a background function to merge them. This function must be commutative and associative (order doesn't matter). This is the basis for Conflict-Free Replicated Data Types (CRDTs).

5. Defining "Concurrency"

Two writes are concurrent if each client did not know about the write of the other when they made the write.

Concurrency has nothing to do with physical time. It is entirely about knowledge and causality (dependencies).

If you write X, and I write Y without knowing about X, they are concurrent.
If you write X, I read X, and then I write Y, they are NOT concurrent. My write "happens-after" yours.

6. Version Vectors Masterclass

How does the database track who knew what? Using Version Vectors. A Version Vector is an array tracking how many updates each replica has processed: [ReplicaA_Count, ReplicaB_Count].

Example: Two replicas holding a Shopping Cart array.

Initial State

Both replicas are empty. No writes have happened.

Replica A [0, 0] Data: [ ]
Replica B [0, 0] Data: [ ]

Client 1 adds a PS5 (Hits Replica A)

Replica A processes the write and increments its index (the first number).

Replica A [1, 0] Data: [PS5]

Client 2 adds an Xbox (Hits Replica B)

Client 2 knows nothing about the PS5. They write to B. Replica B increments its index (the second number).

Replica B [0, 1] Data: [Xbox]

Client 1 adds Lotion (Hits Replica A)

Replica A increments its index again. It still knows nothing about the Xbox on B.

Replica A [2, 0] Data: [PS5, Lotion]

The Sync & Conflict Detection

The datacenters finally communicate over the network. They compare vectors.

Comparing [2, 0] and [0, 1]

Rule: If neither vector is strictly greater than the other in all positions (2 > 0, but 0 < 1), the writes are Concurrent.

The Merge

The database takes the maximum number at each index to create the new merged vector. It saves both arrays as "siblings" to be resolved by the user "On Read".

New Vector = [ Max(2,0) , Max(0,1) ] = [ 2, 1 ]
Synced Datacenters [2, 1] Data Siblings: [PS5, Lotion] AND [Xbox]

7. Summary: Vector Semantics

  • Reading: A client reads data -> it gets the latest version vector from the database.
  • Writing: A client writes data -> it MUST pass the version vector it last read back to the DB, so the DB knows what knowledge the client is acting on.
  • Happens-Before: Vector A is strictly newer than Vector B if every element in A is greater than or equal to the corresponding element in B. (e.g., [3,4] replaces [3,3]).
  • Concurrency: All other vectors are concurrent and must be merged (e.g., [4,2] conflicts with [3,3]).
