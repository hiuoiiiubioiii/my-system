Real-Time Client-Server Communication

In-Depth Systems Design Study Notes & Visualizations

1. Background & Context

In standard web architecture, HTTP requests are strictly initiated by the client. However, modern systems design interview questions frequently ask how to provide real-time updates pushed from a database/server back down to a client.

Common Use Cases Taught in the Video:

  • Chat Applications: WhatsApp, Facebook Messenger (needs instant bidirectional flow).
  • Ride-Sharing Applications: Uber or Lyft (continually seeing where your driver is located on a map).
  • Notification Services: Getting instant push notifications on your phone or browser.
The Core Goal: We need a strategy to propagate updates from the server to the end-user device efficiently, without crashing our backend infrastructure.

2. Naive Polling

How it works: The client makes a standard HTTP request to the server at a fixed interval (e.g., every 5 seconds) asking, "Do you have new data?"

Client
Server

Visual: Client continuously pings. Server returns "empty/red" response 99% of the time.

Why this is bad: You will overload your servers with unnecessary requests!

The Drawbacks (Cons):

  • Server Overload: If millions of smartphones are hitting the server every 5 seconds, it will completely DDOS your own servers.
  • Wasted Compute & Bandwidth: Most of the time, the server has no new data (e.g., waiting for an Instagram DM). The server wastes compute power opening a connection just to return an empty response.

3. Long Polling

How it works: The client makes a typical HTTP request. If the server does not have new data, it keeps the request open (holds the connection). Once new data is present, it responds and closes the connection. The client then immediately issues another long-polling request to start the cycle over.

Client
Server

Visual: Client sends packet. It waits at the Server until data is ready, then returns. Client immediately fires again.

Pros:

  • Easy to implement: Relies on standard HTTP requests.
  • Universal support: Works on all browsers and legacy infrastructure.

Cons:

  • One-Directional: Any information sent from the client to the server must be done through a separate, normal HTTP request.
  • Excess Server Load: Constantly tearing down and re-establishing HTTP connections consumes server resources and compute.
  • Race Conditions: If a user has two browser tabs open, they establish two separate long polls. This can cause messages to arrive out of order, requiring deduplication logic.

4. WebSockets

How it works: WebSockets provide a fully bidirectional, persistent communication channel. An initial HTTP request is "upgraded" to a WebSocket connection, allowing data to flow freely in both directions simultaneously without sending HTTP headers every time.

Client
Server

Visual: Continuous, simultaneous two-way traffic over a single persistent connection.

Architecture Detail (Important):

WebSockets are registered to a specific port on the server. Because ports are limited, a given application service can hold a maximum of ~65,000 active connections before needing to be horizontally scaled (adding more backend servers).

Pros:

  • True Real-Time: Perfect for high-frequency bidirectional data like chat apps.
  • Lower Network Overhead: Heavy HTTP headers are only sent once during the initial connection setup.

Cons:

  • The Thundering Herd Problem: If an application server crashes, all 65,000 WebSocket connections drop. They will all try to instantly reconnect to the remaining servers simultaneously, creating a massive traffic spike that can crash load balancers and backup servers.
  • Overkill: Keeping a connection permanently open is a waste of server memory if data changes are infrequent (e.g., a dashboard that updates every 30 minutes).
  • Browser Support: Lesser support on very old browsers.

5. Server-Sent Events (SSE)

How it works: A persistent one-way connection from servers to clients. The client-side code registers to listen for events, and the server pushes data down the open pipeline whenever it wants.

Client
Server

Visual: Persistent connection where data flows exclusively from Server to Client.

Pros:

  • Persistent Connection: Unlike Long Polling, it uses a persistent HTTP connection (no constant killing and reconnecting).
  • Auto-Reconnect: Unlike WebSockets, SSE handles re-establishing failed connections automatically out-of-the-box via the browser.
  • Great for feeds: Perfect for stock tickers, live news feeds, or notifications.

Cons:

  • Single Directional: The client cannot send data back up the pipeline. It must make a separate request to send data.
  • Connection Load: Holding a lot of concurrent connections open takes up server memory and adds load if the data updates are not actually needed frequently.

6. Conclusion & Interview Strategy

While mentioning WebSockets is generally enough to pass standard systems design interviews for real-time applications, showing distinction is key for senior roles.

The "Senior" Answer:
Interviewers want to see that you understand trade-offs. If they ask you to design a Stock Ticker, saying "WebSockets" is technically correct, but pointing out that SSE is better because it is one-way, has automatic reconnections, and lower overhead demonstrates deep architectural understanding.

In actual tech companies, Long Polling and SSE are still heavily utilized. Why? Due to their widespread support, ease of understanding, seamless integration with existing HTTP load balancers, and lower overhead for infrequent events compared to WebSockets.
