1. Background (What is Aurora?)

📺 On the Screen

  • AWS Aurora is a NewSQL database.
  • Claims a 35x performance boost over running traditional MySQL in the cloud.
  • Uses concepts like quorum writes for replicated logs and clever caching.

🧠 Instructor’s Deep Dive

What is NewSQL? It’s a database that provides the familiar SQL interface (like MySQL or Postgres) but is built on a modern, distributed, cloud-scale architecture.

The goal of this breakdown is to understand exactly how Amazon achieves that massive 35x performance improvement by breaking down traditional bottlenecks.

2. Traditional RDBMS Design (The Old Way)

Memory (Page Cache)
Disk (B-Tree)

📺 On the Screen

  • Pages from the disk that need to be read/modified are loaded from disk into memory.
  • If modified, an entry is appended to the write-ahead log (WAL).
  • Changes are eventually written back to the disk.

🧠 Instructor’s Deep Dive

In a traditional database (like standard MySQL), compute (memory) and storage (disk) are tightly coupled on the same server.

The Workflow: Data is stored on disk in a B-Tree structure. When queried, parts of the B-Tree are loaded into the Page Cache (Memory). If you modify data, the change is written to a Write-Ahead Log (WAL). *Eventually*, the whole modified memory page is written back to the disk.

3. Aurora Design (The Core Innovation)

Memory Server (Page Cache)
↓ Log Entries (WAL) ↓
Disk Server (B-Tree)

📺 On the Screen

  • Aurora decouples the cache and disk onto two different servers!
  • Takes too long to send entire modified pages over the network (typically 8kb).
  • Instead, it just sends new write-ahead log entries. Let the disk server derive state locally.

🧠 Instructor’s Deep Dive

The Problem: Sending an entire modified memory page (typically 8 Kilobytes) over a network to a storage server takes too much bandwidth. Every single write sending 8KB is highly inefficient.

The Aurora Solution: Do not send the whole page. Instead, only send the log entry (Write-Ahead Log). The log entry is tiny. The storage (disk) server receives this tiny entry and applies the update locally in the background.

Because Memory and Disk are completely decoupled, they can now be scaled and replicated independently.

Protection Group (6 Nodes)

📺 On the Screen

  • Storage is replicated across 6 nodes, split across 3 Availability Zones.
  • Writes require a quorum of 4 out of 6 nodes to be successful.
  • Ensures high availability if an entire availability zone goes down.

🧠 Instructor’s Deep Dive

Storage is grouped into a "Protection Group" of 6 disk servers. Why 3 different Availability Zones? Fault tolerance. If a whole data center burns down, you still have data in two other locations.

How a Write Works: The Database Node sends the tiny log entry to all 6 storage nodes simultaneously. It waits for a positive "I got it" response from just 4 out of the 6. Once 4 nodes confirm, the write is considered committed. This prevents the system from hanging if a node is slow.

5. Lightning Fast Reads

📺 On the Screen

  • Reads do not have to use a quorum!
  • The caching layer simply looks at its log to see what the log position is.
  • It then issues a read from just one up-to-date replica.

🧠 Instructor’s Deep Dive

Because the Database Node tracks its own Write-Ahead Log, it knows exactly what the most current "log position" is. When a read request comes in, it looks at its 6 storage replicas, finds just one that is up-to-date with the current log position, and reads directly from it. This makes reads incredibly fast and bypasses the network overhead of checking multiple nodes.

6. Scaling the Caching Layer

📺 On the Screen

  • Can add read-only cache replicas (up to 15 nodes).
  • The main leader cache asynchronously propagates log changes to them.
  • Read-only replicas issue Eventually Consistent reads (reads can be slightly stale).

🧠 Instructor’s Deep Dive

If your application has heavy read traffic, you can scale the *Memory* layer independently. Only one node is the Leader (handles writes). The Leader asynchronously sends its Write-Ahead Log to the replicas.

Because this happens asynchronously, a replica might be a split-second behind. Therefore, reading from a replica is "Eventually Consistent." If you need absolute perfect consistency, you route the read back to the primary Leader node.

📺 On the Screen

  • If a new node takes over as leader, it must ensure uncommitted writes are reverted.
  • It uses a Quorum Read to figure out the greatest committed log position.

🧠 Instructor’s Deep Dive

Imagine the Leader node sends out writes (indexes 101 and 102) but crashes before getting the 4 responses needed to officially commit them. A new node is promoted to Leader, but it doesn't know what was committed.

The Fix: The new Leader performs a Quorum Read (asks 4 out of 6 nodes for their highest index). It realizes that the highest log index that successfully reached a majority is 100. It commands the storage nodes to delete/revert anything higher than 100 (discarding 101 and 102). The system is safely synced.

8. Failure Scenario 2: Storage Node Crashes

📺 On the Screen

  • Data is partitioned into 10GB chunks.
  • To avoid network bottlenecks when replacing a failed node, shards are sent to a variety of nodes to parallelize copying.

🧠 Instructor’s Deep Dive

If you have a 1-Terabyte database, Aurora chunks it into 100 segments of 10GB each. If a storage server holding hundreds of these chunks dies, copying all that data sequentially to a new server would take hours and clog the network.

The Fix: Aurora uses consistent hashing. When a node dies, the remaining nodes work together in parallel to recreate those 10GB chunks across the network. Because many servers share the workload, recovery is lightning fast.

9. Conclusion & Interview Tips

📺 On the Screen

  • Aurora decouples storage and caching while maintaining high availability and strong consistency.
  • Highly useful to mention in a SQL vs. NoSQL interview discussion.

🧠 Instructor’s Deep Dive

The SQL vs. NoSQL Debate: In interviews, candidates often say, "SQL doesn't scale well horizontally; use NoSQL."

The Counter-Argument: You can drop AWS Aurora in the interview! It proves that by using NewSQL concepts (decoupling storage/compute, sending WAL logs instead of full B-Tree pages, using quorum writes), you can scale relational databases horizontally while keeping strong consistency, normalized data, and ACID guarantees.
