Raft Consensus Algorithm

Complete In-Depth Lecture Breakdown

1. Introduction & Background

Raft is a consensus algorithm introduced around 2014 designed to display fault tolerance in distributed systems.

  • Purpose: Used to build a replicated log so that every node (server) in a cluster eventually has the exact same log.
  • Result: Achieves Total Order Broadcast (eventual consistency across databases).
  • Why is it popular? Compared to older, notoriously complex algorithms like Paxos (or Multi-Paxos), Raft was explicitly designed to be easy to understand.

Goal: Ensure all databases eventually share the exact same table/log, achieving eventual consistency.

2. Core Vocabulary

Three crucial terms define the mechanics of Raft:

Quorum: A majority of nodes in the cluster (e.g., 3 out of 5).
Rule: It is impossible for two different quorums to make conflicting decisions because at least one node must overlap between the two groups, preventing a conflicting vote.
Fencing Token (Term Number): An increasing number used to resolve conflicts like "split brain." In Raft, this is called the Term Number.
Rule: The higher Term Number always wins.
Leader: A single node through which all writes are sent and propagated. If the leader fails, the system must switch to a new leader safely.

3. Raft Overview: High-Level Flow

How does data actually get saved into the database layer?

  1. Client Request: The Leader receives a write request.
  2. Propagation: The Leader sends all writes to the "Follower" nodes.
  3. Quorum Agreement: Once the Leader receives positive responses from a majority (quorum) of followers accepting the write, it is considered safe.
  4. Commit: The Leader tells the followers to "commit." This moves data from the local Log into the actual Key-Value Store/Database layer, allowing clients to read it.

4. Deep Dive: Leader Election

This is the most critical part of Raft: ensuring there is only one leader at a time per Term. Every node is in one of three states:

The Heartbeat & Election Trigger

  • The Leader periodically sends Heartbeats to all followers to say, "I am alive."
  • If a follower doesn't receive a heartbeat within a specific timeout window, it assumes the leader is dead and becomes a Candidate.

The Election Process

  1. The Candidate votes for itself.
  2. It increments its internal Term Number by 1.
  3. It asks all other nodes for their vote.
🧠 GENIUS MECHANIC: Randomized Timeouts
If all followers realized the leader died at the exact same time, they would all vote for themselves, resulting in a tie where nobody gets a majority. To fix this, Heartbeat Timeouts are randomized. One node will always wake up slightly faster, declare itself a candidate, and start the election before others.

Voting Logic

When a node is asked for a vote, it checks three conditions:

  • Term Check: Is the Candidate's term number higher than my local term number?
  • Log Check: Is the Candidate's log more up to date than my local log? (Candidates cannot win if they are missing data).
  • Vote Check: Have I already voted for someone else in this term? (Nodes get exactly 1 vote per term).

5. Broadcasting Messages (ReplicateLog)

Once a Leader is elected, it uses the ReplicateLog function to keep followers in sync. This acts as both a heartbeat and a data sync mechanism.

When sending data, the Leader mentally splits its log into two parts:

  • Prefix: All the older messages the Leader assumes the follower already has.
  • Suffix: The brand new messages the follower needs.

The Payload: The Leader sends the Follower the Suffix AND the Term Number of the very last message in the Prefix.

6. The Prefix Invariant

This mathematical rule is what makes Raft completely safe from data corruption.

The Raft Invariant: If the logs on two replicas have the same term number at the same index in the log, they MUST be identically the same up to and including that index.

Why is the right side impossible?

If Index 1 matches (Op: B, Term: 3), Index 0 MUST match. A leader could never have been elected in Term 3 if its earlier logs (Index 0) were missing or different, because it wouldn't have possessed the "most up-to-date log" required to win the election.

7. Conflict Resolution

Because of the Prefix Invariant, followers can easily verify if they are in sync.

Scenario A: In Sync (Happy Path)

The Follower matches the Leader's Prefix Term Number. It copies the Suffix over, overwriting any garbage data, and sends a "Success" acknowledgment.

Scenario B: Out of Sync (Conflict!)

The Follower checks the Prefix Term Number, and it does not match its local log. The Follower rejects the write.

Leader's Fix (Shrinking the Prefix): The Leader realizes its assumption was wrong. It steps backward one index in the log, expands the Suffix, and tries again. It loops backward until it finds the exact index where the logs match, and overwrites the Follower from that point forward.

Scenario C: Deposed Leader

If a Follower responds to the Leader with a higher Term Number, the Leader realizes it has been deposed. It immediately gives up leadership and reverts to a Follower state.

8. Conclusion & Trade-offs

Why use Raft?

  • Highly Fault Tolerant: The whole network doesn't need to be online. You only need a Quorum (majority) alive to process writes.
  • Understandable: Vastly easier to implement, trace, and debug than older algorithms like Paxos.

When is Raft NOT ideal?

  • Network Heavy: It requires constant network calls (heartbeats, voting phases, and acknowledgments).
  • Cross-Partition Transactions: Compared to "2-Phase Commit" (2PC), Raft is amazing for replicating a single log, but not ideal for atomic transactions that span across multiple different database partitions.
