📚 Data Serialization

Mastering distributed systems data transport & schema evolution.

1. Background & The Core Problem

In distributed systems, we constantly need to move data between computers or save it to disk. However, data lives inside a computer's RAM differently than how it travels over a network.

In-Memory Object
class Person {
  name: "Jordan"
  pointer: 0x12ab
}
→
Byte Stream (Network)
[01001010, 01101111...]
"You cannot send a memory pointer over a network. That memory address means absolutely nothing to the receiving computer. We must translate this in-memory representation into a flat, transportable 'sequence of bytes'. This is called Serialization."

Why format matters: How we structure these bytes impacts our network efficiency, storage costs, and our ability to safely update our software without crashing our systems (Schema Evolution).

2. The Naive Approach: Language-Specific Libraries

Most programming languages have built-in serialization (e.g., Pickle for Python, Marshal for Ruby, Serializable for Java). While easy to use, they are terrible for production distributed systems.

Why you shouldn't use them:

  • Language Lock-in If you serialize with Python's Pickle, a Java microservice cannot read it.
  • Security Flaws Malicious actors can craft byte streams that, when deserialized, instantiate arbitrary objects and execute malicious code on your server.
  • No Versioning They handle changes to data structures (adding/removing fields) very poorly.
  • Bad Performance They are slow and create bloated, massive byte streams.

3. Standardized Encodings: JSON & XML

JSON and XML are ubiquitous. They are fantastic for public web APIs because they are human-readable and universally understood. But for internal cross-service communication, they have major flaws.

"JSON is not particularly compact. Every single time you send a Person record, you have to send the string 'userName' over the network. If you send millions of records, you are sending the field names millions of times, wasting massive network bandwidth."
The Network Bloat Problem

{ "userName": "Jordan", "favoriteNumber": 24 }

In JSON, the red text (keys) must be transmitted every single time. This is heavy metadata!

Other issues with JSON/XML:

  • Type Ambiguity: XML struggles to differentiate numbers from strings. JSON cannot specify the precision of floating-point numbers.
  • No Binary Support: They only support Unicode strings. Binary data must be heavily encoded (like Base64) to be sent via JSON.

4. Binary Encodings: Thrift & Protocol Buffers

Created by Facebook (Thrift) and Google (Protocol Buffers / Protobuf), these frameworks solve the bloat problem by using Schemas and Field Tags.

Protobuf Schema Example
message Person { required string userName = 1; optional int64 favoriteNumber = 2; }
"The Magic is Field Tags. Look at the numbers 1 and 2. Instead of sending the string 'userName' over the network, Protobuf just sends the number 1. The receiving computer looks at its schema, sees 1, and knows that means 'userName'. This drastically reduces file sizes!"

Code Generation: Once you write this schema, a compiler generates classes (in Java, Go, Python, etc.) automatically. You just use these generated classes in your code.

5. Schema Evolution & Compatibility

In the real world, "rolling updates" are standard. Server A might run Code V1, while Server B runs Code V2. They must still understand each other. This requires compatibility.

🔙 Backwards Compatibility

Newer code can read data written by older code.

(V2 code reads V1 data)

🔜 Forwards Compatibility

Older code can read data written by newer code.

(V1 code reads V2 data)

Rules for Thrift/Protobuf Evolution:

  • NEVER change existing field tags. If you change "userName" from tag 1 to tag 4, old code looking for tag 1 will break.
  • Adding fields: You can add new fields, but they must have new, unique tags, and they cannot be required. If old V1 code receives a new tag '4', it simply ignores it (Forwards Compatibility). If a new field was required, old code wouldn't send it, causing a crash.

6. Avro: The Alternative for Big Data

Avro was built for Hadoop and massive data files. Unlike Protobuf, Avro has NO field tags. It relies entirely on the order of the fields in the schema and matches fields by their Name.

"To decode Avro, you need to look at both the schema used to write the data (Writer Schema), and the schema the current code expects (Reader Schema)."

Writer vs. Reader Schema Resolution

Writer's Schema (Old Data)
Field Name
userName
photoURL
⇨
Reader's Schema (New Code)
Field Name
userName (Matched!)
userID (Uses Default)

photoURL is ignored by the reader. userID is missing from the writer, so the reader assigns a default value.

Network Optimization: Because Avro lacks tags, the full schema must accompany the data. To save bandwidth:
1. In Hadoop files, put the writer schema once at the top of a file containing millions of records.
2. Over a network, send a Version Number with the record. The receiving server looks up that version's schema in a database.

7. Why Field Tags are Hard for Databases (ETL)

If you want to automatically export your SQL database to a data lake every night (an ETL process), Protobuf is highly annoying.

"SQL databases do not have 'Field Tags' (numbers). A human would have to manually assign tags to every SQL column to map it to Protobuf. With Avro, because it relies on field names, you can write a script that automatically looks at SQL column names and generates an Avro schema instantly."

Data Outlives Code Remember, a user profile saved in your database 5 years ago was written with an old schema. The serialization framework (Avro/Protobuf) acts as the translator, ensuring your modern code doesn't crash when reading 5-year-old database rows.

8. Final Conclusion

Serialization libraries like Protobuf, Thrift, and Avro are critical for modern backend and distributed engineering because they provide three major benefits:

  • 🗜️ Compression: They strip away structural metadata (like field names in JSON), making network transfers and disk storage incredibly fast and cheap.
  • 📜 Documentation: Keeping a database of Schemas acts as a historical record of exactly what your data looked like at any point in your application's history.
  • ⚙️ Code Generation: In statically typed languages, they automatically generate the boilerplate classes needed to interact with the data, preventing human error.
