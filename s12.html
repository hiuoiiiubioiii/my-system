Linearizability and Ordering

A deep dive into Strong Consistency, Lamport Timestamps, and Total Order Broadcast.

1. What is Linearizability?

In distributed systems, Linearizability is synonymous with Strong Consistency.

Once a write is successfully committed to any single replica for a given object, all subsequent reads (regardless of which replica they hit) will show the results of that write.

  • No stale reads: Users will never see an outdated version of an object after a write completes.
  • The Single Copy Illusion: To a user, the distributed database behaves exactly as if there is only one central copy of the data.
Crucial Distinction: Do not confuse Linearizability with Serializability.

Linearizability is a guarantee about a single object (recency/stale reads).
Serializability is a database transaction concept guaranteeing operations across multiple objects prevent race conditions. It has nothing to do with transactions!

2. When is Linearizability Useful?

You need linearizability when your system absolutely requires just ONE absolute truth of something.

  • Distributed Locks: Only one node can hold a lock at a time. If a node reads a stale state saying a lock is free when it isn't, catastrophe ensues.
  • Uniqueness Constraints: Imagine claiming a username in a database. If you and a friend concurrently try to claim "goldenshowerlover100" on different replicas, strong consistency ensures only one person gets it, and no subsequent writes for that username succeed.

Pros and Cons

  • Pros: Always up-to-date results. The developer logic is much simpler.
  • Cons: Huge performance hit. Implementing this requires network consensus, which adds massive latency.

3. Ordering: Total vs. Partial

If we want to act as if there is only one copy of data, there must be a Total Order of operations. The state of the database must be expressible by executing operations one at a time in a sequence that all replicas agree upon.

This order preserves causality (if A happens before B, A is before B in the order). If A and B are concurrent, their relative order doesn't matter, but every replica must apply them in the exact same order.

The problem with Partial Order: If replicas don't force a total order on concurrent events, Replica 1 might process A then B, while Replica 2 processes B then A. This leads to inconsistent data.

4. How to Create an Ordering

  • Single Leader Replication: The leader's Write-Ahead Log (WAL) does this natively.
  • Leaderless / Multi-leader: Because writes enter from anywhere, we need specific tools. Version Vectors work but take O(n) space. The alternative is Lamport Timestamps.

Deep Dive: Lamport Timestamps

A Lamport Timestamp provides a total order using a simple tuple: (Counter, NodeId).

Rule: Nodes track the highest counter they've seen. If they see a higher one during an interaction, they skip to that number. NodeId breaks ties.

Node 1 writes to itself, counter becomes 5.

Final Total Order (sorted by Counter, then NodeId):
(1, 1) → (1, 2) → (2, 2) → (3, 2) → (4, 2) → (5, 1)

5. Lamport Timestamps: Tradeoffs & Pitfalls

Feature Lamport Timestamps Version Vectors
Space Complexity O(1) - Just two numbers. O(n) - Scales with number of nodes.
Concurrency Forces Total Order. Cannot tell if events were concurrent. Expresses Partial Order. Can identify concurrent siblings.
Data Loss / Merging Arbitrary tie-breaking leads to silent data loss. No custom merging. Allows custom merging logic for concurrent edits.

The Retroactive Pitfall

Lamport Timestamps provide a total order, but they only give us this order retroactively. They are insufficient for real-time linearizability.

Example: You and a friend concurrently write a claim for username fupa_sniffer_21 to different replicas. Both replicas accept it and return a "Success" message to both users immediately.

Only later, when replicas sync and compare Lamport Timestamps, is the tie broken. One user's "successful" write is quietly rolled back. You didn't get real-time strong consistency.

6. Total Order Broadcast (TOB)

To fix the retroactive ordering problem, replicas must receive operations in the correct order as they happen. This requires a protocol called Total Order Broadcast.

  • Reliable Delivery: Messages are delivered to every node without being lost (requires constant retries).
  • Ordered Delivery: Messages are delivered in the exact same sequence to every single node.

Replicated Log via TOB

Write: Claim User X (Node A)
Write: Claim User X (Node B) [REJECTED]
Read: Get User X
Write: Update User X settings

Because all nodes process this log strictly left-to-right, the second concurrent claim is naturally rejected. No retroactive rollbacks needed.

Achieving Linearizability with TOB

  • Writes: Append to the log. Replicas perform writes strictly in log order. (First username claim wins, subsequent ones in the log error out).
  • Reads: To ensure reads aren't stale, you must also append reads to the log. The database only returns the value to the user when it reaches that read request in the log, ensuring all prior writes are processed.

7. Summary & The Consensus Problem

While TOB can create linearizable storage, and linearizable storage can create TOB, how do we actually build these?

Ultimately, doing this requires solving the Consensus Problem (getting independent, unreliable nodes to agree on a single state/order). This requires complex algorithms like Paxos or Raft.

Final Takeaway: Linearizability acts as if there is only one copy of data. Ordering is easy with a single leader, but requires tools like Lamport timestamps otherwise. Because Lamport timestamps are retroactive, we need Total Order Broadcast for real-time consistency.

If you can avoid having to deal with strong consistency, eventual consistency often suffices and takes a far lesser toll on performance.
