Systems Design Interview Blueprint

A comprehensive, scrollable study guide based on the 6-step framework for tackling any technical systems design interview.

Introduction

Why Structure Matters

Possessing systems design knowledge is not enough. If your design on the whiteboard is a jumbled mess, you will appear disorganized to the interviewer.

The Goal: You must convey your knowledge in an effective, concise, and organized manner. There is a universally accepted "proper way" to tackle these interviews, broken down into the vertical feed below.

Step 1

Requirements Clarifications

Interview prompts are usually very broad (e.g., "Design Uber" or "Design Twitter"). You cannot design all of Uber in 45 minutes. You must immediately narrow down the scope by asking the interviewer questions.

📋 The Interviewer Checklist:

[ ] Functional Requirements (Features): What exactly are we building?
    ↳ Ex (Twitter): Can we post tweets? Do we have DMs? Images?

[ ] Scale / Users: How many total users? Daily active users?

[ ] Performance Goals: What is the main bottleneck to solve?
    ↳ Ex (Twitter): Are we optimizing for fast tweet posting, or extremely fast home-page reads?

At the end of the day, your interviewer is only interested in a small, specific part of the application. Find out what it is before you start drawing.

Step 2

Capacity Estimations

You need to do some math on the whiteboard to prove you understand the scale of the data you will be storing. This dictates the hardware, caching, and partitioning you will need.

  • Ratio of Reads to Writes: A "read-heavy" application requires a completely different architecture and database choice than a "write-heavy" application.
  • Number of Operations: How many operations per day? What is the size of each?
🧮 Whiteboard Math Cheat Sheet:

1 Character = 2 bytes
Average 140-char Tweet = ~300 bytes
Average Image Upload = ~10 Megabytes
High-Quality Video = ~1 Gigabyte

Why do this math? If you calculate you need to store 10 Terabytes of data, and a standard server holds ~10TB of disk space, you know immediately that you must partition/shard your database across multiple machines.
Step 3

API Definitions

Do not waste too much time here. Keep it basic. You just need to show how the client (e.g., the mobile app) communicates with your backend server using REST endpoints.

// 1. User Account Creation Endpoint
POST create_user(email, password_hash)

// 2. Core Feature Endpoint (e.g. Twitter)
POST send_tweet(user_id, text)

Note: In the real world, these endpoints take in much more data (headers, auth tokens), but for an interview, just write out the bare minimum function calls to show you understand the concept.

Step 4

Data Tables / Schemas

You do not need to commit to a specific database type (SQL vs NoSQL) just yet. You simply need to outline what information you will be storing.

/* Tweets Table Schema */
{
  "User_id": int,
  "Text": string,
  "Timestamp": date,
  "Metadata": json
}

Instructor Tip: It is completely fine (and expected in a microservices architecture) to have multiple different database tables holding different pieces of the pie.

Step 5

Get Designing! (The Architecture)

Key Concepts to Include/Discuss on the Whiteboard:

  • Load Balancing: Distributes traffic so no single server gets overwhelmed.
  • Replication: Copying data across databases so it isn't lost if a machine dies.
  • Sharding: Splitting data across multiple machines based on your capacity math.
  • Microservices: Avoid "monoliths." Break the app into smaller, dedicated services (e.g., an Auth service, a Notification service).
Step 6

You Will Get Grilled (Defending Tradeoffs)

The interviewer is not just going to nod along. They will look at your architecture and say, "What if this server crashes? Why did you do this? What about this edge case?"

Your job is to prove you didn't just memorize a diagram from a textbook. You do this by explaining TRADEOFFS. You don't need a 100% flawless design, you need a feasible solution that you can logically defend.

Technology Choice Pros (Why you chose it) Cons (The Tradeoff)
WebSockets (vs Long Polling) Real-time, persistent connection (Great for chat apps). Harder to scale, keeps connections open consuming server memory.
Microservices (vs Monolith) Can scale individual parts of the app independently. Complex network latency, harder to debug system-wide issues.

Example Defense: "No one cares if you can design Facebook Messenger unless you can explain *why* you chose WebSockets over Long-Polling for that specific scenario."

🏁 Ready to Practice

This concludes the conceptual framework. Now that you know the 6 steps (Requirements -> Math -> API -> Schema -> Design -> Tradeoffs), apply them to specific system design problems.
