Microservices Architecture

Comprehensive Study Notes & Visualizations based on Systems Design breakdown

1. Background

Slide Text

In many systems design questions, you will have to design a good amount of functionality for some hypothetical use case. Typically, it makes sense to split each piece of function into separate microservices, which act as self contained units of servers that can interact with other microservices as if they were external services.

Instructor Breakdown

  • The Problem Setup: In an interview, you might be asked to design a complex app (e.g., Tinder). This involves several distinct functionalities: messaging, matching, account creation/deletion, etc.
  • The Microservices Solution: Instead of building one massive application, it's usually better to separate these disjointed functionalities. In a microservice architecture, these become self-contained services. Individual teams can work on these services independently.

2. Original Approach: Monoliths

Slide Text

All code contained in the same repository, same set of nodes in a cluster responsible for handling any possible application request.

Instructor Breakdown

  • Historical Context: This is the older approach, common 10-20 years ago when sites didn't operate at the massive scale requiring complex distributed systems.
  • How it Works: All logic (handling users, business analytics, data management) is packaged together into one centralized backend application. All of this code lives in a single Git repository.
  • Execution: A cluster of server nodes will all run this exact same, single application. Everything touches one centralized database.

3. Benefits of a Monolith Approach

Slide Text

  • Easy to implement: Do not need to think about how to split up functionality. All code goes in the same repository.
  • Easy to test: As opposed to using blackbox testing with other microservices, can transparently make sure that everything works properly via unit tests in one repository.
  • Easy to deploy and scale: Only one repository to deploy. If we need more compute we simply redeploy the same repository on other nodes.

Instructor Breakdown

Despite their drawbacks at scale, monoliths are still used because they have distinct advantages:

  • Implementation: There is no added complexity of figuring out how to logically divide services. You just write code in one place.
  • Testing: End-to-end testing is much simpler. Because all the code is present, you don't have to rely on mocking external services to see if a feature works; you can run full tests on the actual codebase.
  • Deployment & Scaling: Deployment is straightforward—you just deploy the one repository. To scale, you just take that entire application and put it on more server nodes.

4. Cons of a Monolith Approach

Slide Text

  • Large application codebase makes it harder to onboard new developers.
  • Deployments are huge and likely take a while.
  • Harder for multiple teams to work on at the same time (Merge conflicts, blocking bugs).
  • Pieces of the service that differ in popularity cannot be scaled independently.
  • Wanting to use a new technology or framework requires rewriting the entire monolith.

Instructor Breakdown

The speaker emphasizes that these cons are significant, especially as a company grows:

  • Maintainability: A massive codebase with different developers using varying coding styles is difficult to maintain and very daunting for new developers to learn.
  • Deployment Bottlenecks (Biggest Issue): Even a tiny code change (e.g., 2% of the company's code) requires redeploying the entire monolith, causing massive downtime for the 100% of the service.
  • Team Collaboration: It's very difficult for multiple teams to work concurrently. Merge conflicts are frequent. A bug merged by one team can break the build and block another team from compiling unrelated work.
  • Inefficient Scaling (Key Example): Imagine Facebook is a monolith, and the "Facebook Shopping" feature suddenly gets very popular. With a monolith, you can't just scale up the shopping feature. You have to scale the entire application linearly, wasting a massive amount of computing resources.
  • Tech Stack Lock-in: If your monolith is in Java, and you decide Kotlin is better, you must rewrite the entire application. Agile development prefers rewriting small chunks at a time, which is impossible here.

5. Microservice Architectures

Slide Text

Teams each manage a few self-contained services that can be deployed and scaled independently, using a tech stack of their choice. This is far more scalable and solves all of the major issues mentioned regarding monoliths in the previous slide, at the cost of adding some complexity to your application layout.

Instructor Breakdown

  • How it Works: The architecture features one centralized User Interface on the front end. On the back end, there are multiple "mini" applications (microservices).
  • Communication & Data: These microservices communicate with each other (typically using REST/HTTP endpoints or SOAP). Crucially, they often have their own separate databases containing only the data relevant to that specific service.
  • Solving Monolith Problems: Each service acts as an independent unit of compute. This means:
    • Teams can develop, deploy, and scale their specific service completely independently of other teams.
    • Teams can choose the tech stack that works best for their specific service (e.g., Service A uses Java, Service B uses Kotlin).

6. Quick Aside: Docker

Slide Text

  • Applications are packaged into "containers": Holds all dependencies/versions so environments (local laptop, test, production cloud) run exactly the same.
  • Deploying to a cloud: Docker allows running multiple containers on just one virtual machine (acts like a lightweight OS). Gives each container the ability to take the resources they need in a given moment, leading to far less waste of compute power.

Instructor Breakdown

The shift to microservices made tools like Docker essential. Docker is a DevOps tool for deploying and managing code.

  • Containers vs. Bugs: Docker packages your microservice code along with all its dependencies, libraries, and binaries into a "container." This guarantees that the code runs exactly the same way on a developer's laptop as it does in the cloud. It eliminates platform-specific bugs ("it works on my machine" problem).
  • Efficient Resource Usage:
    • The Old Way: Usually, you partition servers into Virtual Machines (VMs) and guess how much capacity each VM needs. If you have 10 microservices, you might allocate too much space for one (wasting money) and too little for another (causing crashes).
    • The Docker Way: Docker acts like a lightweight operating system. You configure one VM and throw all your Docker containers onto it. The containers dynamically scale up and down, taking only the compute resources they need at any given moment, preventing wasted compute and saving money.

7. Conclusion

Slide Text

While it generally makes sense for most new applications with a smaller team to start out under a monolith design to develop quickly and reduce added complexity, using a microservice architecture at scale is virtually a necessity. It allows teams to develop quickly on their own without a need for a unified codebase, and provides them with the ability to deploy and scale their service as they see fit. Finally, a bug introduced in the code by one team will not affect the services of another, and failures will be siloed.

Instructor Breakdown

  • When to use Monoliths: Perfectly fine for small startups or new applications where the goal is to develop quickly without the overhead of designing a complex distributed system. At small scale, you don't need microservices.
  • When to use Microservices: For large companies operating at massive scale (millions/billions of users like FAANG), decoupling functionality is mandatory. It speeds up development by allowing independent teams to work, deploy, and scale individually.
  • Siloing Failures: Crucially, microservices "silo" failures. If one small microservice crashes, it doesn't bring down the entire platform for all users.
