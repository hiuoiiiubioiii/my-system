Comprehensive Study Guide & Visual Notes

1. Background & The Need for HTTPS

📝 From the Whiteboard

In previous lessons, we mentioned network protocols (TCP/UDP) used to send byte streams reliably. However, how do we ensure the byte streams are going to the right place and are secure?

Certificate transparency is used in HTTPS, which takes traditional HTTP requests and adds an additional security layer (known as SSL/TLS).

🎙️ Instructor's Deep Dive

TCP and UDP are great for ensuring data gets from point A to point B, but they don't ensure that data is encrypted. If data falls into the wrong hands, anyone can read it. To fix this, we use HTTPS, built on top of HTTP. HTTPS utilizes an SSL/TLS security layer to guarantee that the data sent to the other party is encrypted and secure from interception.

📝 From the Whiteboard

  • A third party reads and intercepts HTTP packets, and can see any of the information being sent.
  • The only way to prevent data from being stolen is by encrypting it such that only the server can understand it!

🎙️ Instructor's Deep Dive

A Man-in-the-Middle attack occurs when a user believes they are sending requests directly to a web server, but a perpetrator has disguised their IP address to sit in the middle of the connection. If you send passwords or location data in plain HTTP, the perpetrator reads it all. The fix is encryption—scrambling the data so the middleman just sees gibberish.

3. RSA (Public Key) Encryption

📝 From the Whiteboard

  • The server generates a Public Key and a Private Key.
  • It broadcasts the Public Key over the network to the client.
  • The client encodes any messages it wants to send using the Public Key.
  • The server keeps the Private Key to itself—it is the only way to decrypt the message.

Caveat: The client has to make sure they are using the public key from the actual server, and not from a man in the middle!!

🎙️ Instructor's Deep Dive

RSA is an asymmetric encryption system. The server shares its public key openly. You use it to encrypt your data. Once encrypted, only the private key can reverse the math to read the data. Because the server never shares its private key, a MITM attacker cannot decrypt your data. However, the main vulnerability is trust: How do you know the public key you just received actually belongs to the server, and not an attacker handing you their own public key?

4. Digital Certificates & CAs

📝 From the Whiteboard

Certificates are used to verify that a server is actually who it says it is. Each server must register with a Certificate Authority (CA).

The certificate provides information about the server: its owner, IP address, its public key, and the authority that registered it (verifiable via a digital signature).

🎙️ Instructor's Deep Dive

To solve the "fake public key" problem, we use Certificates. Think of a certificate as a digital passport issued by a highly trusted centralized entity called a Certificate Authority (CA). The CA investigates the server and issues a certificate saying, "We verify that this specific Public Key legitimately belongs to Netflix.com."

5. The TLS Handshake Process

  1. Browser Connects: Asks server for identity.
  2. Server Responds: Sends its Digital Certificate (signed by CA).
  3. Browser Verifies: Checks CA's digital signature. If valid, browser generates a random "Session Key", encrypts it with the Server's Public Key, and sends it.
  4. Server Decrypts: Uses its Private Key to decrypt the session key. A secure session is established!

🎙️ Instructor's Deep Dive

How do we verify the CA's signature? CAs have their own public/private key pairs. The CA uses its private key to digitally sign the server's certificate. Your web browser (Chrome, Safari) comes pre-installed with the public keys of all trusted CAs. Your browser uses the CA's public key to decrypt the signature on the certificate. If it matches, you know a trusted CA issued it!

6. The Fatal Flaw: Can we trust CAs?

📝 From the Whiteboard

What if a CA got hacked? Then a malicious actor could start giving themselves certificates for domain names that they don’t own.

What we need is Certificate Transparency.

🎙️ Instructor's Deep Dive

The entire SSL/TLS system relies heavily on centralized Certificate Authorities. If a malicious actor hacks a CA, they can issue a perfectly valid, signed certificate that says, "I am Netflix.com" but attaches the hacker's public key. Your browser will trust it, and the MITM attack succeeds. We need a system where CAs are held publicly accountable.

7. Certificate Transparency (Logs, Auditors, Monitors)

📝 From the Whiteboard

Certificate Transparency is a way of publicly broadcasting all issued certificates so that bogus certificates can be quickly noticed and thrown out.

  • Certificate Logs: Append-only lists where CAs must publish all certs.
  • Auditors: Web browsers that query the logs to ensure a cert is in it.
  • Monitors: Programs run by domain owners (e.g., Netflix) that poll the logs looking for bogus certificates issued in their name.

🎙️ Instructor's Deep Dive

CAs are now forced to add every certificate they issue to public, append-only logs. When your browser (the Auditor) receives a certificate, it checks the Log to ensure it's published. Meanwhile, companies run Monitors. Netflix's Monitor constantly scans the logs. If it sees a certificate issued for Netflix that it didn't authorize, it immediately knows a CA was hacked and revokes the bogus certificate.

The Vulnerability: What if the server hosting the log lies? It could show the Auditor (your browser) the fake entry, but hide it from the Monitor (Netflix) so they never catch it.

8. Securing Logs with Merkle Trees

📝 From the Whiteboard

How can we detect if the log server isn't lying? Merkle Trees!!

  • Generate a Merkle Tree from the log to get the root hash.
  • To ensure a sequence of elements is in the tree, recreate it on the browser and see if root hashes match.
  • Optimized to only need log(n) elements (just the hashes from opposite branches).

🎙️ Instructor's Deep Dive

To prevent logs from lying, we use cryptographic proofs called Merkle Trees. A Merkle tree hashes pairs of data all the way up to a single Root Hash. To prove a certificate is in the log, your browser doesn't have to download the entire multi-gigabyte log. It only needs the hashes of the opposite branches to calculate its way up to the Root Hash. Because hashing is deterministic and collision-resistant, faking this proof is mathematically near-impossible.

🎙️ Instructor's Deep Dive

To prove `c1` is in the tree, the log server provides `c1`, plus the hashes `45ty` and `99j7`. The browser hashes `c1` to get `a12x`. It concatenates `a12x` with the opposite branch `45ty` and hashes them to get `143c`. Finally, it concatenates `143c` with `99j7` and hashes them to get `tyg4`. Because `tyg4` matches the known Root Hash, the browser is 100% certain the certificate is in the log!

10. Fork Consistency & The Gossip Pool

📝 From the Whiteboard

  • Fork Attack: Log shows monitors and browsers different versions of the log.
  • To prevent this, browsers exchange Merkle tree roots with monitors to see if they differ (Gossip Pool).
  • Merkle trees can be used to prove one tree is a prefix of the other (showing how the root of the earlier tree combines with new elements to output the new root).

🎙️ Instructor's Deep Dive

A corrupt log could try a "Fork Attack"—maintaining two versions of the truth. It shows a fake log to your browser, and a clean log to the Monitor. To catch this, browsers and monitors use a "Gossip Pool." They constantly talk to each other in the background, comparing the Root Hashes they are seeing. If the hashes don't match, they know the log server is forked and corrupt. Furthermore, "prefix proofs" ensure that as new certificates are appended, no historical certificates were secretly deleted.

11. Conclusion & Limitations

📝 From the Whiteboard

Certificate Transparency is not perfect. While cryptographic proofs are useful for pointing out a bad actor, they can’t do so instantly. A malicious CA can still get away with stealing passwords for a small period of time until a monitor catches it.

🎙️ Instructor's Deep Dive

This system relies on "eventual consistency." It takes a short amount of time for the Monitors to scan the logs, find the bogus certificate, and revoke it. During that tiny window, a hacker might succeed. However, Certificate Transparency introduces the core concept of dealing with decentralized, distributed systems where we cannot trust the actors. We use cryptographic proofs and gossip networks to establish truth—which is the exact foundation we will use when learning about Blockchains and Bitcoin!
