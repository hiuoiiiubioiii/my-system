1. Background: The Purpose of Caching

When designing large-scale distributed systems, caching is essential. Redis and Memcached are the industry standards for highly performant data access.

The Core Concept: Both services live in Random Access Memory (RAM), allowing you to build a Distributed Hash Map.

Time Complexity: Lookups (accessing keys) and setting keys happen in O(1) time (constant time), which is incredibly fast and impossible for traditional disk-based databases.

2. Memcached: The Bare Minimum

Memcached is an extremely simple, bare-bones distributed string-to-string hashmap. It is basically a limited subset of what Redis can do.

  • Node Ignorance: Memcached servers do not know about each other.
  • Client-Side Routing: Because nodes don't communicate, routing logic lives in the client library. The client uses a consistent hashing ring to route requests.
  • LRU Eviction: If an instance gets full, it uses the Least Recently Used (LRU) algorithm to evict the oldest data.
  • No Built-in Failover: There is no built-in failure handling or replication.
Real-World Workaround (Facebook): Because there is no failover, Facebook uses an empty "Gutter" Memcached instance. If a primary node fails, requests are routed to the Gutter to take its place. Data isn't copied over; the Gutter populates organically over time as new reads/writes occur.

3. (Single Node) Redis: The Multifunctional Upgrade

Redis is essentially a modification of Memcached that adds a massive amount of functionality, turning it from a simple cache into a versatile tool.

Strings Sets Maps Lists Atomic Operations Transactions Range Queries
  • Beyond Strings: Supports complex data structures and atomic operations (e.g., safely appending to a string).
  • Transactions: Perform multiple writes serially to a single node as an atomic unit.
  • Hash Tags: Allows hashing of only part of a key, giving control over partition routing.

Disk Persistence Strategies

Redis can write to disk, making it a viable actual database, not just a volatile cache.

Checkpointing

Takes intermittent snapshots of memory. Pros: Faster. Cons: High risk of data loss (losing writes) if a crash happens between snapshots.

Write-Ahead Log (WAL)

Every write goes to disk before being written to memory. Pros: Prevents data loss. Cons: Slower write speeds.

4. Redis Cluster: High Availability

The distributed version of Redis, designed to provide high availability and consistency out-of-the-box without needing external management.

  • Single Leader Replication: Writes go to the leader, which asynchronously replicates to followers. Warning: If the leader crashes before replication finishes, those writes are permanently lost.
  • Gossip Protocol & Failover: Nodes share "heartbeats" conveying node states and partition mapping. If a leader dies, a quorum of master nodes votes on a new leader.
  • Split-Brain Prevention: Redis uses an "Epoch" number (election version number) to ensure a quorum cannot make conflicting leadership decisions.
  • Partitioning: Does not use consistent hashing! It uses exactly 16,384 fixed hash slots. Admins map these slots to nodes to prevent server hotspots. Replicas can handle read requests to reduce master load.

5. When NOT to use Redis Cluster?

If Redis is just Memcached with more features, why would you ever use Memcached? The answer lies in strict architectural requirements. Redis forces you into its predefined paradigms (Single Leader, Gossip).

Revert to Memcached and build a custom architecture if you:
  • Need strongly consistent data: You absolutely cannot tolerate lost writes on a failover event.
  • Want alternate replication: You require a Master-Master or Leaderless (DynamoDB style) replication pattern.
  • Prefer centralized coordination: You want to use a dedicated coordination service (like ZooKeeper) for configuration and partition management instead of relying on the Gossip protocol.

6. Conclusion

Caching greatly reduces database load and latency, though it increases system complexity (especially regarding write consistency). Despite the complexity, it is a must-have in real-world systems.

💡 Pro Tips for System Design

Cost vs. Benefit: Caching should be used whenever possible until it prices you out (RAM is expensive). If you can afford it, use it.

The Facebook Metric: At Facebook, 99% of read requests are handled by the cache. This entirely saves their underlying databases from crashing under immense global load.

Interview Strategy: Anyone can say "I will use a cache." To stand out, articulate the subtle trade-offs: Client-side Consistent Hashing (Memcached) vs. 16k Hash Slots (Redis); Single Leader vs. Leaderless; Gossip Protocol vs. ZooKeeper.
