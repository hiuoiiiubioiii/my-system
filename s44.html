Big Data Columnar Storage Data Engineering

1. Background & The Core Problem

To understand Parquet, we must understand the problem it solves. Previously, data serialization formats like Avro, Thrift, and Protocol Buffers were discussed. These formats store data in a row-oriented manner.

The Problem with Row Storage

  • Row-oriented storage keeps all fields of a single record (row) adjacent on the disk.
  • This is excellent for finding a specific user's entire profile.
  • However, Big Data Analytics (using MapReduce or Spark) rarely needs whole rows. It usually performs aggregations across specific columns (e.g., "Find the average age of all 10 million users").

🎙️ Instructor's Deep Dive

If you use row-based storage for analytics, the system wastes massive amounts of time reading irrelevant columns just to find the ones you need. By switching to column-oriented storage, we gain data locality. If all "ages" are stored physically next to each other on the disk, reading them becomes incredibly fast. Furthermore, we want data to be stored as compactly as possible so transferring it over the network to compute nodes is efficient.

2. Hybrid-Oriented Storage

While pure column storage is great for analytics, it suffers from a major flaw: Lack of disk locality between elements of the same row. If you ever *do* need to reassemble a full row, it requires jumping all over the disk to find Column A, Column B, etc.

Parquet's Solution: A Hybrid Approach.

Parquet takes a massive table and slices it horizontally into "partitions" (e.g., Row 0 to Row 100,000). Inside that partition, the data is stored by column. This creates a perfect balance: you get the analytical speed of columnar storage, but because columns are bounded within a partition, reassembling a row doesn't require scanning the entire disk.

3. Parquet Introduction & Architecture

Parquet data is highly structured hierarchically. The dataset is split into Parquet files containing the following internal hierarchy:

  • Parquet File: Contains multiple Row Groups and a footer with metadata.
  • Row Groups: Logical horizontal partitions of the actual data.
  • Column Chunks: Inside a row group, data is split into chunks for each column.
  • Pages: The chunks are further divided into pages (typically 1MB in size).

The default value for a Row Group is usually 128 Megabytes of row data. In a database that is Terabytes in size, you will have many partitions. The lowest level of storage is the Page. Pages are crucial because they contain metadata (the Min, Max, Count, Dictionaries, and Bloom Filters) regarding the specific 1MB of data encoded inside them. This metadata enables massive query optimizations.

4. Data Encoding & Extreme Compression

Because columns naturally contain repeated data (e.g., a "Country" column will have "United States" repeated thousands of times), Parquet utilizes advanced compression algorithms:

  • Plain Encoding: If data is fixed length, store raw data. If varying length, store an integer representing the length, followed by data.
  • Dictionary Compression: Replaces repeating strings with small integer keys.
  • Bit Packing: Reduces the bit-size of the integer keys.
  • Run Length Encoding (RLE): Compresses adjacent duplicate values into a mathematical tuple.

This pipeline shrinks files exponentially. First, long strings are swapped for integers via the Dictionary. Next, Bit Packing comes in. If your dictionary only has 4 unique countries, you don't need a massive 64-bit integer—you only need 2 bits to represent numbers 0 through 3! Finally, RLE handles adjacent duplicates. If "United States" (key 0) appears 3 times in a row, Parquet doesn't write `0, 0, 0`. It writes the tuple (0, 3).

5. Optimal Chunk Size Trade-offs

Choosing the correct Chunk Size is a vital engineering decision when generating Parquet files.

  • If chunks are too big: You will not be able to fit all unique values in the dictionary. Parquet will be forced to drop dictionary compression and fallback to inefficient "plain encoding" for the rest of the data.
  • If chunks are too small: You incur massive storage overhead because you have to store the metadata section for every single tiny chunk, dwarfing the actual data size.

6. Predicate Pushdown (The Speed Secret)

Predicate Pushdown is the ability to filter data at the storage level, skipping massive amounts of irrelevant data before it even enters memory or network bandwidth.

How it skips data:

  • Min/Max Stats: If a query asks for `x > 69`, Parquet reads the page metadata. If the page's Max value is 50, the entire page is skipped.
  • Dictionaries: If querying for `x == "Japan"`, and "Japan" is not in the page's dictionary, skip the page entirely.
  • Bloom Filters: Space-efficient probability data structures recently added to Parquet that definitively state if a value is not present.
  • File Partitioning: Saving files in folders by date/category allows skipping entire files (e.g., reading only `/2023/January/` files).

This is why pre-sorting your data before writing to Parquet is a pro-tip! If data is pre-sorted, the Min/Max values of pages are tight and sequential (e.g., Page 1 is 0-20, Page 2 is 21-40). This allows Predicate Pushdown to skip huge continuous blocks of data instantly. If your data is entirely random, the Min/Max ranges will be vast (e.g., Min 1, Max 100 on every page), rendering this optimization useless.

7. Conclusion & Big Data Architecture

Parquet provides a hybrid approach combining column chunks with rich metadata to allow for highly efficient querying. However, it is not a direct replacement for row-based serialization.

🎙️ Final Instructor Takeaways

Parquet doesn't replace Avro, Thrift, or Protobuf. It works in conjunction with them.

In a modern Data Lake (like Databricks), you use row-based formats (Avro) for ingesting live, single-record data (e.g., a single user clicking a button). But for Batch Processing, ETLs, and Analytics, you convert that data to Parquet.

In Big Data, the primary bottlenecks are CPU serialization/deserialization cycles and Network Latency. Because Parquet is so tightly compressed and Predicate Pushdown prevents transferring irrelevant data, it drastically reduces compute costs and query times.
