Unreliable Clocks in Distributed Systems

1. The Background: Why We Want to Use Clocks

Imagine a distributed database with multiple "leader" nodes accepting writes. Because separate nodes cannot communicate instantly, they often have conflicts regarding the order of operations.

The "Simple" Idea: From a theoretical perspective, using a timestamp seems like a great way to resolve conflicts. If every event is stamped with the current time, you just pick the event with the later timestamp. It avoids the complexity of tracking concurrent "sibling" events (like Version Vectors do).

The Reality: We cannot rely on physical clocks. While simple in theory, they are inherently flawed for distributed ordering.

2. How Clocks on Computers Actually Work

To understand why physical clocks fail in distributed systems, we must understand how a computer actually tells time.

  • NTP Synchronization: Most computers don't have atomic clocks or direct GPS receivers. They get their time by occasionally connecting over the network to an NTP (Network Time Protocol) server.
  • Quartz Oscillation: Between these occasional NTP syncs, the computer relies on an internal, physical quartz crystal to measure elapsed time and keep the clock ticking forward.

3. The Three Major Issues with Clocks

The instructor highlights three critical reasons why this NTP + Quartz setup makes computer clocks unreliable for ordering events:

  • 1. Network Latency: Synchronizing with NTP requires a network call. The server replies with a timestamp, but the message takes an arbitrary amount of time to travel back. By the time your machine receives it, the timestamp is already outdated.
  • 2. Quartz Clock Drift: Quartz crystals aren't perfectly precise; they oscillate slightly faster or slower based on temperature and manufacturing. According to Google's measurements, a computer clock can drift by an average of up to 7 seconds per day without an NTP sync. In computing, 7 seconds is an eternity.
  • 3. Untrustworthy Clients: You cannot trust timestamps generated by a client device (like a phone). Users can manually change their system time (often done to cheat in mobile games), rendering client timestamps useless for determining true order.

4. Measuring Elapsed Time Correctly

If physical clocks drift and delay, how do we measure the time between events (elapsed time) on a single machine?

Do NOT use "Time-of-Day" Clocks!

  • If an NTP sync happens while you are measuring, your clock might literally jump backward or forward in time to correct itself.
  • Time-of-day clocks often ignore "leap seconds" (tiny adjustments made to global time), causing further inaccuracies.

The Solution: Use a Monotonic Clock.

A monotonic clock is essentially a dumb counter that only ever goes forward. The specific value (e.g., "10045") means absolutely nothing and doesn't map to real-world time. Its only purpose is to safely measure deltas (differences) between two events without the risk of the clock jumping backwards.

5. Ordering Events & Data Loss

How do we solve the original problem of ordering events across different machines?

If you use physical timestamps to order events across nodes (a strategy known as "Last Write Wins" or LWW), out-of-sync clocks will literally cause you to drop user data.

The Data Loss Scenario:

  • Server A's clock is drifting 5 seconds ahead of Server B.
  • A user updates a record on Server B.
  • A second later (in the real world), another user updates the same record on Server A.
  • Because Server B's clock is behind, Server A's timestamp appears newer, even though Server B's write actually happened last. Server A's data "wins", and Server B's data is permanently dropped. It looks as if data was sent backwards in time!

The Real Solution: Instead of physical time, use Logical Timestamps. These are mechanisms (like Version Vectors or Lamport Timestamps) whose sole purpose is to measure the relative ordering of events, rather than tying them to a flawed physical clock.

6. Summary & Real-World Applications

The Core Concept: In distributed systems, a physical clock reading is not absolute truth. It can only generate a confidence interval (e.g., "I am 90% sure the time is between X and Y"). Due to NTP network delays and quartz drift, clocks are ultimately unreliable.

Yet, many databases still use them. Why? Because it's simple.

  • Cassandra: A popular NoSQL database that still uses physical timestamps for conflict resolution. They admit it isn't perfect and heavily rely on system administrators configuring strict NTP synchronization to minimize data loss.
  • Google Spanner: Relies heavily on "True Time." To make this work without data loss, Google doesn't rely on standard NTP and Quartz. They use highly specialized hardware, attaching GPS receivers and atomic clocks directly to every single server to keep the confidence interval incredibly tight, allowing for consistent global snapshots.

Conclusion: Unless you are Google building specialized hardware, relying on physical clocks introduces complex, time-based bugs. Use logical ordering instead!
