Collaborative Text Editing

Systems Design Deep Dive: Overcoming concurrency conflicts in real-time document editing.

1. The Core Problem

Platforms like Google Docs and VSCode allow multiple users to edit the same data at once. The ultimate system goals are to avoid conflicts and ensure every local copy of the document converges to a consistent state that makes sense.

Why is this hard? (Array writes are not commutative)

If we treat a text document like a simple array of characters, concurrent edits break the system. Commutative means the order of operations doesn't matter. But with arrays, inserting an item shifts all subsequent indexes. The order does matter.

💡 Instructor Explanation Because array indices shift when a new character is added, an insertion at "index 3" means something entirely different after someone else has inserted a character at "index 0". Without a reconciliation system, user documents fall completely out of sync.

2. Solution 1: Operational Transform (OT)

Operational Transform (OT) fixes the shifting index problem using a complex mathematical transformation function (Function T).

💡 Instructor Explanation OT accepts relative changes and pushes them through Function T. If User A types at index 0, Function T tells User B: "Hey, User A just shifted everything right by 1. Your planned insertion at index 2 must be transformed to index 3."

The Pitfall: Function T requires a Total Order of updates. This means every edit must route through a central server (single leader) to get time-stamped in exact order. This creates a massive bottleneck and high latency for systems with millions of users.

3. Solution 2: Text Editing CRDTs

CRDT (Conflict-Free Replicated Data Type) is an alternative that allows peer-to-peer merging without a central server. Since array inserts aren't commutative, CRDTs change the data model entirely: A document becomes a Set of (char, location) tuples.

Fractional Indexing

Instead of integers (0, 1, 2), every character gets an absolute, fractional floating-point coordinate.

💡 Instructor Explanation: Engineering Intricacies Because coordinates are absolute, inserting text at the beginning of the document doesn't change the coordinates of the text at the end. Operations can arrive out of order! But there are massive edge cases to handle:
  • Concurrent Exact Inserts: Two users type in the exact same spot simultaneously and generate the same float (e.g., both get 0.25). We need arbitrary tie-breakers (like comparing User IDs).
  • Floating Point Precision: Dividing space by 2 continuously (0.5, 0.25, 0.125...) eventually exceeds computer floating-point precision limits. The document must occasionally be re-indexed.
  • Interleaving Text: If User A pastes "HELLO" and User B pastes "WORLD" at the same spot, the system must group the blocks so it doesn't print "HWEOLRLLOD".
  • Deletions: You can't delete a character you don't know exists yet. Deletes require Causal Broadcast (using Lamport timestamps) to ensure the "insert" message is processed before the "delete" message.

4. Conclusion & Review

Building real-time collaboration is incredibly difficult. While Operational Transform is the battle-tested standard (used by Google Docs), CRDTs are the active field of research solving the latency and scaling bottlenecks of central servers.

Quick Review Checklist

  • The Core Problem
    Array writes are non-commutative; the order of application changes the final array state, leading to document divergence.
  • Operational Transform (OT)
    Uses Function T to transform shifting indices. Requires a central server to establish a Total Order, which limits scalability and adds latency.
  • Text CRDTs & Fractional Indexing
    Treats the document as a Set of characters with absolute floating-point locations. Allows peer-to-peer, commutative merging without a central server.
  • CRDT Edge Cases
    Must handle floating-point precision exhaustion, block-text interleaving, concurrent exact-location inserts, and require Causal Broadcast (Lamport clocks) for deletions.
