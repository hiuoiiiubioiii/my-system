TCP vs. UDP

Comprehensive Transport Layer Protocol Study Guide

1. Background & The Network Layer

At its core, a network is a highly complex, chaotic system of cables and routers sending unstructured 1s and 0s. To make sense of this chaos, we use Protocols.

A protocol is an established standard for formatting and sending data so all devices can extract useful information. TCP and UDP operate at the Transport Layer on top of an IP network. Their primary job is to deliver packets from Node A to Node B, but how they go about it is what separates them.

2. Transmission Control Protocol (TCP)

TCP is the meticulous protocol. It provides reliable, ordered, and error-checked streams of bytes between two computers.

  • Ordered: Achieved via Sequence Numbers.
  • Error-Checked: Achieved via Checksums.
🎤 Instructor's Insight: Checksums "Every self-contained packet includes a checksum (a mathematical sum of the bytes inside it). When the receiving computer gets the packet, it does the same math. If a bit accidentally flipped during transit due to network noise, the checksums won't match, and TCP knows the packet is corrupted and throws an error."

3. The Three-Way Handshake

Unlike UDP, TCP nodes cannot just start blasting data at each other. They must formally establish a connection first.

Client (Host P)
1. SYN (seq=x)
2. SYN (seq=y), ACK (x+1)
3. ACK (y+1)
Server (Host Q)

Note: Every acknowledgement (ACK) value tells the other computer what sequence number it expects to receive next.

🎤 Instructor's Insight: Why Random Sequence Numbers? "If we always started at 1, malicious users could guess the sequence number, spoof packets, and inject malicious data into the connection. Starting with a random number prevents this."

4. Reliable Transmission

Because TCP guarantees ordered delivery, it has built-in mechanisms to handle lost packets.

Duplicate ACKs (Dup Ack)

If the Sender sends packet #99 (which gets lost), and then sends packet #100, the Receiver will get #100. However, the receiver knows it is missing #99. It cannot accept #100 out of order. It will reply with "ACK 99" (meaning "I still need 99!"). The sender realizes the mistake and resends packet 99.

Timeout-Based Estimates & Exponential Backoff

If a sender doesn't receive an ACK within a certain time (based on the estimated Round Trip Time - RTT), it will resend the packet.

🎤 Instructor's Insight: Protecting Against DDoS "If a receiver goes offline, the sender shouldn't just spam the network every millisecond trying to resend data. Instead, it uses an exponential backoff. It waits 1 second, then 2, then 4, then 8... decreasing the frequency of retries. This ensures broken nodes don't accidentally cause a DDoS attack on the network."

5. Flow Control (Protecting the Receiver)

TCP regulates flow so one device doesn't overwhelm another device with too much data.

Receiver's Memory Buffer (Queue)

85% Full

Every computer has a buffer in memory to hold incoming packets before the CPU processes them. If this buffer fills up, packets are dropped. In every ACK message, the receiver tells the sender exactly how many bytes of free space it has left. If the receiver has 0 room left, the sender pauses, waiting for a tiny "probe" packet to see when space opens up.

6. Congestion Control (Protecting the Network)

While Flow Control protects the receiver, Congestion Control protects the actual network links (cables/routers) from congestive collapse.

🎤 Instructor's Insight: The Danger of Congestion "If routers get overwhelmed, packets drop. Dropped packets trigger TCP retries. Retries mean MORE packets on the network, making the congestion even worse."

Mitigation: AIMD Strategy

TCP uses a "Congestion Window" to limit packets in-flight. It uses Additive Increase, Multiplicative Decrease (AIMD):

  • 📈 Additive Increase: Window size grows linearly and cautiously when transmissions are successful.
  • 📉 Multiplicative Decrease: The millisecond it detects congestion (dropped packets), it cuts the window size completely in half to instantly relieve network pressure.

7. Connection Termination

Terminating a TCP connection uses a Four-Way Handshake because both halves of the connection must terminate independently.

  • Initiator sends FIN ➔ Receiver sends ACK.
  • Receiver sends FIN ➔ Initiator sends ACK.
🎤 Instructor's Insight: The "TIME_WAIT" State "After sending the final ACK, the port goes into a specific timeout period. If you immediately close the port and open a new connection, an old packet from the previous connection (delayed in a slow router) might suddenly arrive. The timeout lets all these ghost packets 'die' before the port is reused."

8. User Datagram Protocol (UDP)

UDP is the total opposite of TCP. One node indiscriminately starts sending data to another node without any consideration for flow control, dropped packets, ordering, error detection, or congestion control.

💻 💨 🔫🤬
🎤 Instructor's Insight: The Scarface Analogy "The presentation uses a picture of Scarface firing a machine gun. This is a perfect analogy. UDP is the 'rat-a-tat-tat' protocol. No handshakes, no checking if the receiver is ready, no caring if the network is full. It just blasts packets at the destination. If they arrive out of order, or don't arrive at all, UDP does not care."

9. TCP vs. UDP: Comparison & Use Cases

UDP (User Datagram Protocol)

  • ⚡ Speed: Much faster (no handshakes or ACKs).
  • 📉 Reliability: None. Packets drop and arrive out of order.
  • 🛑 Protections: None. Can overwhelm receivers/network.
  • 🎯 Best For: Real-time apps where absolute speed matters and dropped data is acceptable.
  • Use Cases: Video chatting (Zoom), Live Multiplayer Gaming, DNS requests, Live Stock Tickers.

TCP (Transmission Control Protocol)

  • 🐢 Speed: Slower (overhead for setup and ACKs).
  • 📈 Reliability: High. Guarantees delivery and exact ordering.
  • 🛡️ Protections: Flow Control and Congestion Control.
  • 🎯 Best For: Accuracy-focused apps where every single byte of data must be present.
  • Use Cases: REST APIs, Web Browsing (HTTP/HTTPS), File transfers, Emails, Banking apps.
🎤 Instructor's Insight: Video Chat vs. Banking "Why UDP for Video Chat? If a frame of video gets delayed, by the time it arrives, the conversation has moved on. It's better to drop the frame (a quick screen glitch) than pause the whole call waiting for TCP to resend it.

Why TCP for Web/Banking? If you load a bank balance, you cannot afford to drop digits or have packets arrive out of order. Accuracy is paramount over latency."

10. Conclusion & System Design

As a modern software engineer, you likely won't program TCP/UDP handshakes directly (the OS handles this). However, knowing the difference is crucial for high-level system architecture.

The Golden Rule: TCP is the industry standard default. Trying to program reliable transmission, flow control, and congestion control into your own application code from scratch is a nightmare. But, understanding when to strip away TCP's guarantees to get the raw, low-latency speed of UDP is a hallmark of excellent system design.
