Geospatial Indexes

Complete study notes detailing how modern databases handle 2D geographic data efficiently for location-based queries.

1. Background & The Use Case

In many modern applications, it is crucial to perform queries based on the location of a given point. The most common pattern is finding all objects within a certain distance (radius) of an original point.

  • Uber/Lyft: Finding nearby drivers within a few miles.
  • Yelp: Locating restaurants around your current coordinate.
  • Tinder: Filtering potential matches by geographic proximity.

The Challenge: Quick querying in a database is usually accomplished by creating an index (like a B-Tree). However, indexing geolocation data requires sophisticated work because traditional indexes are 1-dimensional, while geographic coordinates are 2-dimensional (Latitude & Longitude).

2. Coordinate Data: The Problem

Coordinates are expressed as a 2D tuple: (Latitude, Longitude). Let's look at why a traditional database index fails here.

Target point (Yellow) and search radius. Traditional indexes force a full table scan.

Why traditional B-trees fail:

  • If you index only on Latitude, you can quickly find points within the vertical height of your circle.
  • However, the database still has to fetch all of those points (which could be millions across the globe at that latitude) and manually calculate the Longitude to see if they fall inside the circle.
  • In the worst case, this degrades into a massive table scan, which is far too slow for real-time applications.

3. The Solution: Geohashing

To solve the 2D problem, we use Geohashing. Geohashing maps areas of a 2D plane to a single, 1D string value.

The map is split into regions (a, b, c). Each region is recursively split into subregions (aa, ab, ac), appending characters per depth.

How it works:

  1. Divide the map into a grid and assign each section a character (e.g., 'a', 'b', 'c').
  2. Zoom into a cell (like 'a') and divide it into a sub-grid. Assign new characters appended to the parent (e.g., 'aa', 'ab', 'ac').
  3. Repeat this process. The deeper you go, the longer the string, and the smaller/more precise the geographic area becomes.

Crucial Concept - Lexicographical Sorting: Because of this hierarchical structure, if you sort all geohashes alphabetically in a database, points that are geographically close to each other will sit right next to each other in the database index.

5. The Geospatial Indexing Process

Recall that B-trees and LSM Trees allow us to execute extremely fast range queries. Here is how we use them with Geohashes to find points within a radius:

  1. Convert: Convert your original starting coordinate into a long, precise geohash.
  2. Determine Depth: Figure out the geohash depth that encapsulates the distance you want to search (using the size table above).
  3. Truncate: Chop off the end of your original geohash string to match that depth. This creates a "bounding box".
  4. Range Query: Use the database index to find all points inside that bounding box (e.g., `WHERE hash > 'box_start' AND hash < 'box_end'`).
  5. Filter: The bounding box is a rectangle, but your radius is a circle. Calculate the exact math distance of the returned points to filter out the corners of the box.

6. Example Query execution

Goal: Find all points < 1km from coordinate (42.167, 12).

1) Original Geohash: 91vz62q

2) Get Bounding Box:
   a) We know length 5 has a size of ~4.9km².
   b) Truncate to length 5: We want all points in "91vz6".

3) Execute Range Query:
   Find all points (p) where: 91vz6 <= p < 91vz7
   *This is an indexed B-tree query, making it super quick!*

4) Filter:
   Calculate actual distance of returned points and drop any > 1km.

Note on Edge Cases: In reality, if your target point is near the border of the bounding box, the 1km radius might spill into the neighboring geohash grid. Therefore, a robust system may need to check the ranges of the surrounding bordering bounding boxes as well (up to 9 separate range queries total).

7. Distributed GeoSpatial Indexes

How do applications with massive datasets (Uber, Tinder) scale this across multiple servers? Partitioning (Sharding).

Because geohashes group geographically close data by string prefixes, you can easily assign large chunks of an index to specific database nodes.

  • Node 1: Assigned box "a" and all its children (aa, ab, ac...).
  • Node 2: Assigned box "b" and all its children.

This guarantees that all nearby points exist on the same physical server, avoiding expensive cross-network queries when searching within a local radius.

8. GeoSpatial Indexes in Practice

Lyft & Redis

Lyft heavily utilizes Redis' built-in geospatial indexes. Redis uses geohashing under the hood and keeps the index entirely in-memory. This provides the massive performance and low-latency required to track moving cars in real-time.

Uber & H3 (Hexagons)

Uber created their own open-source geospatial indexing system called H3. It is similar to standard geohashing, but instead of dividing the world into squares/rectangles, it divides it into a grid of Hexagons.

Why Hexagons? A hexagon is much closer to the shape of a circle than a square is. This makes calculating distances to neighboring cells highly uniform and efficient, reducing the complexity of edge cases.

9. Indexing Shapes? (R-Trees)

Geohashing is perfect for points (like a person or a car). But what if you need to index full polygon shapes (like the borders of a park, a voting district, or a building footprint)?

For shapes, databases unlikely use geohashes. Instead, they use a more complicated tree structure known as an R-tree.

Root Box
R-trees work by creating nested bounding boxes. The outer root box contains smaller child boxes, which eventually contain the distinct geometric shapes.

Conclusion

GeoSpatial indexing is a must-have for any application that heavily relies on finding objects with similar locations (real estate services, hotel finders, dating apps).

The Core Takeaway: By converting each two-dimensional geographic coordinate into a one-dimensional hash with a hierarchical structure, we can quickly perform area range queries on geospatial data using standard, highly-optimized database indexes.
