Chain Replication & CRAQ

What is it? Chain Replication is a database replication scheme invented in 2004.

The Goal: To achieve both Strong Consistency (linearizability) and High Availability, while still maintaining a decently high throughput.

Instructor's Fun Fact: The paper for this architecture was actually written by the instructor's Operating Systems professor!

The Evolution: The original Chain Replication architecture was later modified into a system called CRAQ (Chain Replication with Apportioned Queries). CRAQ uses the exact same chain layout but drastically improves read speeds.

Vanilla Chain Replication: Writes

Nodes are organized into a strict chain: a Head, middle Replicas, and a Tail.

  1. The Client sends a write request only to the Head node.
  2. The Head passes the write to the next Replica, continuing down to the Tail.
  3. Once the Tail receives it, the write is committed. The Tail passes an Acknowledgement (ACK) back up the chain.
  4. When the Head gets the ACK, it tells the client the write was successful.

Vanilla Chain Replication: Reads

The Golden Rule: All read requests are made only from the Tail node.

Why? Because writes are only acknowledged as successful once they reach the Tail node. By forcing all clients to read from the Tail, the system guarantees Strong Consistency. You will never read stale or uncommitted data because the Tail is the ultimate source of truth.

Handling Node Failures (Failover)

The system uses an external coordination service (like ZooKeeper) running a consensus algorithm to monitor node health, avoiding split-brain scenarios common in gossip protocols.

1. Head Fails

Promote the next replica to Head. May lose uncommitted writes from the old head, but the client will safely retry since they never got an ACK.

2. Tail Fails

Promote the replica right behind it to be the new Tail. Zero data loss, as this node contains all committed data.

3. Middle Fails

Remove from chain. The predecessor node updates to point to the successor node and passes any unacknowledged writes to bridge the gap.

Analysis of Vanilla Chain Replication

Pros

  • Strong Consistency: Achieved via total ordering of writes.
  • Cheap Writes: Unlike single-leader (Raft/Paxos) where the leader messages all nodes at once, the Head only messages one node. Prevents Head overload.

Cons

  • Read Bottleneck: Read throughput is limited to exactly one node (the Tail) unless database is sharded.
  • Write Bottleneck: The chain is only as fast as its slowest link. One slow middle node cripples the entire write speed.

CRAQ: Fixing the Read Bottleneck

CRAQ modifies the system to allow read queries from any node while keeping strong consistency. It adds two variables to the data:

  • Version Number: Increments with each write.
  • State: Either Clean (committed by Tail) or Dirty (uncommitted, waiting for ACK).

How CRAQ Writes Work:

  1. Client writes a new value. Head increments version and marks it Dirty.
  2. The Dirty write propagates down the chain.
  3. Once the Tail receives it, it is committed! Tail saves it as Clean.
  4. Tail sends ACKs backward. As nodes get the ACK, they switch local state from Dirty to Clean and delete old versions.

CRAQ Reads: The Tricky Part

Clients can now query any node. The node checks its internal state to decide how to answer:

Scenario 1: Data is Clean

If the client asks for data and the replica's highest version is labeled Clean, the replica safely returns the value immediately.

Scenario 2: Data is Dirty

If the highest version is labeled Dirty, the replica doesn't know if the Tail has committed it yet. Returning it could break Strong Consistency.

The Solution: The replica pauses and sends a tiny network request to the Tail asking: "What is the current committed version number?" The Tail answers, and the replica returns that specific version's value to the client.

Final Evaluation & Interview Tip

Why CRAQ scales: In read-heavy workloads, most reads hit "Clean" data, meaning read throughput scales linearly with the number of replicas. Even when dirty, asking the Tail for a tiny integer (version number) is way faster/cheaper over the network than having the Tail send the actual file/data payload to the client.

Note: CRAQ still suffers from the write bottleneck (one slow node slows all writes).

💡 Systems Design Interview Tip

Chain Replication and CRAQ aren't the most common architectures, but they are incredible aces up your sleeve. If an interviewer asks you to design a system needing Strong Consistency while minimizing the network load on the leader node, explaining Chain Replication will make you look highly knowledgeable and stand out.
