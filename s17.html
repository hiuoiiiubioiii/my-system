Part 2

Background: Why do we need it?

Consistent hashing is a way to uniformly and randomly distribute load over nodes (servers) in a cluster.

The Core Benefit

It minimizes the amount of "reshuffling" (moving data around) when you add or remove nodes. The majority of already-hashed items stay exactly where they are!

Use Case 1: Sharding

When partitioning a database, you don't want to rebalance the entire dataset when adding one server. Consistent hashing ensures minimal rebalancing, reducing network stress.

Use Case 2: Load Balancing

Routing requests to the same application servers repeatedly allows for better cache performance (more cache hits, fewer misses).

Part 3

Visualizing the Hash Ring

To understand the algorithm, we visualize the hash function's output as a ring.

  1. Pick a Hash Function: A one-way function that takes input and gives a predictable number.
  2. Imagine the Range: For this example, the hash function outputs 0 to 999.
  3. Build the Ring: After 999, the number line loops back to 0, creating a circle.
Part 4

Adding Nodes (Virtual Nodes)

If we simply hashed each server (Node 1, Node 2) onto the ring once, they might clump together, causing one node to take 70% of the load. This creates a hotspot.

The Solution: Virtual Nodes

Instead of placing one point per server, we place many virtual copies of each server randomly around the ring.

This ensures the load is divided up fairly. No single node gets stuck with a massive, unfair chunk of the ring.

Part 5

Data Placement

How do we know which server holds a specific piece of data?

1

Hash the Key: Take your data key (e.g., "jordan") and hash it.
hash("jordan") = 261

2

Locate on Ring: Find where 261 lives on the ring (just past the 250 marker).

3

Move Clockwise: Move clockwise around the edge of the ring until you hit the very first node.

4

Store/Retrieve: The node you hit is the one responsible for that data! (In this case, Node 3).

Part 6

What happens when a node fails?

If a server crashes or is removed (e.g., Node 3 goes offline):

  • All the "virtual nodes" for Node 3 disappear from the ring.
  • The data that was pointing to Node 3 simply shifts clockwise to whatever node is next in line.
  • Crucial Concept: Nodes 1 and 2 keep all of their existing data! Only the data chunks that belonged to Node 3 are reassigned.

This is why it's called consistent hashing—the system avoids a massive reshuffle, saving network bandwidth and computing power.

Summary

Key Takeaways

Even Distribution

A solid algorithm for distributing load evenly among servers utilizing virtual nodes.

Minimal Movement

On exits or joins, the vast majority of items stay in their exact same place.

Real-world Use

The backbone of load balancers and database partitioning methods like Apache Cassandra.
