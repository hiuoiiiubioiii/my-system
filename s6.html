Database Transactions

Part 1: Strong Isolation & Serializability

1. What are Transactions?

Transactions are an abstraction used by some databases that provide ACID guarantees about queries.

Each write in the transaction will either be committed or aborted entirely without any side effects.

Client Bundle
Read(A)
Write(B)
Write(C)
→
DB

🗣️ Instructor's Notes

Think of a transaction as a "bundle" or "package" of operations (reads and writes) sent to the database.

  • The database treats this bundle as a single unit.
  • There is no partial success—either every operation in the package works, or the whole package fails and leaves no trace behind.
2. ACID Explained

Atomicity

If a client makes several writes, but a fault occurs, the existing writes are rolled back. Half a transaction will never exist.

Consistency

The application can rely on the database to ensure invariants hold (in the face of faults). The DB behaves exactly as expected.

Isolation

Concurrently executing transactions are isolated from one another. Each pretends it is the only one running on the database.

Durability

Once a transaction is completed, the data will never be forgotten, even in the face of faults (e.g., power outages).

🗣️ Instructor's Notes

Isolation is the hardest part! This eliminates race conditions. If multiple threads are hitting the database at the exact same time, the database forces them to act as if they are standing in a single-file line.

Because forcing threads to wait in line is slow, Isolation is what causes major database performance bottlenecks.

Durability: Once it's committed to the hard drive, it's safe (unless someone literally smashes the hard drive with a hammer!).

3. Implementing Isolation: Actual Serial Execution

Idea: Implement all database queries on a single CPU thread.

Strict Conditions Required:

  • In-memory database (No slow disks)
  • Use a stored procedure (No slow network calls)
  • Attempt to keep all transactions limited to one partition
Pros: Very simple to implement. Race conditions are mathematically impossible.
Cons: Throughput limited to a single core. Stored procedures are hard to version control and use weird languages.

🗣️ Instructor's Notes

How it works: Literally just use one thread. If there's only one thread, race conditions cannot happen.

Why the strict requirements? A single CPU thread is fast, but waiting for network calls or hard-drives to spin is incredibly slow. To make a single thread fast enough, the data must live in RAM (like Redis or Memcached), and the code must live directly on the database (Stored Procedures).

The Reality: Most highly scaled applications cannot survive on a single CPU core, and developers hate writing stored procedures. It is an annoying after-thought for many DBs.

4. Implementing Isolation: Two Phase Locking (2PL)

Idea: Each object has a lock on it. Threads must grab the lock before touching data.


  • Shared Lock: Multiple transactions can read a row concurrently.
  • Exclusive Lock: Required to write. You can only get this if NO ONE else is holding a shared or exclusive lock.

🗣️ Instructor's Notes

This is the most popular solution historically used by databases.

  • Shared Lock (Read-Only): Think of this as looking at a painting in a museum. Many people can look at it at the same time.
  • Exclusive Lock (Write-Only): Think of this as taking the painting off the wall to repaint it. If you want to do this, everyone else must leave the room. If even one person is still looking at it (holding a shared lock), you must wait.
5. The Phantom Bug & Index Range Locking

The Phantom Bug: How do you lock a row that doesn't exist yet? (e.g., checking if a room is booked, then booking it).

Attempt 1: Predicate Locks. Lock all objects matching a search condition. (Issue: Terrible performance. DB has to check every row ever added).

Solution: Index Range Locking. Use an index to put a lock on a "superset" of the data quickly.

🗣️ Instructor's Notes

Instead of telling the database to monitor the abstract concept of "The Oval Office on Tuesday" (which takes massive computing power via Predicate Locks), you just lock the Index for the Oval Office.

Trade-off: By locking the index for "Oval Office," you accidentally lock all days for the Oval Office (Monday, Wednesday, etc.), not just Tuesday. However, because indexes are pre-built and fast, it is mathematically much faster for the database to lock a "superset" of data than to use Predicate Locks.

6. The Big Problem with 2PL: Deadlocks

Terrible performance occurs due to frequent deadlocks. The DB must detect this, abort one transaction, let the other finish, and then retry the aborted one.

T1: Wants to add +1
T2: Wants to add +1
⬇️ Grabs Shared Lock
⬇️ Grabs Shared Lock
Jordan's Attractiveness: 8
⬆️ Needs Exclusive Lock (WAIT)
⬆️ Needs Exclusive Lock (WAIT)
🚨 DEADLOCK DETECTED! Both holding Shared, both waiting for Exclusive.

🗣️ Instructor's Notes

2PL is famous for creating deadlocks. Here is the example from the slides:

  1. T1 reads jordan_attractiveness: 8 (Gets Shared Lock).
  2. T2 reads jordan_attractiveness: 8 (Gets Shared Lock).
  3. T1 wants to add 1 to make it 9. Needs an Exclusive Lock. Wait! T2 has a shared lock.
  4. T2 wants to add 1 to make it 9. Needs an Exclusive Lock. Wait! T1 has a shared lock.

Result: DEADLOCK! Neither can move forward. The database acts as a referee, realizes they are stuck, and completely aborts T2. This lets T1 upgrade its lock, change the value to 9, and finish. Then, the DB automatically retries T2 from scratch (it reads 9, writes 10).

7. Implementing Isolation: Serializable Snapshot Isolation (SSI)

Idea: Let everything run concurrently as if there was no locking, and only revert/abort a transaction if a concurrency bug has been detected.


All reads occur from a snapshot of the database:

  • If an uncommitted write occurred before your read, you need to check if that write has since been committed by the time you make your write. (Abort if so).
  • Keep track of which transactions have read an item. If another transaction writes to said item, abort all transactions that read it.

🗣️ Instructor's Notes

This is a relatively new, optimistic algorithm.

  • 2PL is pessimistic (it locks everything assuming things will go wrong).
  • SSI is optimistic (it assumes things will be fine, and asks for forgiveness rather than permission).

Performance Impact: Because threads aren't sitting around waiting for locks to release, throughput is insanely fast. Unless... your app has a ton of users trying to edit the exact same row at the exact same time. If that happens, SSI crashes and burns due to a high volume of aborts and retries, making it slower than 2PL.

8. Conclusion

Transactions are a potentially very useful abstraction of multiple operations that need to be bundled together.

They either all succeed, or all fail. However, they come at a great performance cost.

As a result, many databases have chosen to implement weaker forms of isolation, which will be discussed in the next video.

🗣️ Instructor's Notes

Real-world, highly scaled applications usually cannot afford the performance hit of true Serializability (Strong Isolation).

To make apps run fast, database engineers accept "weak isolation"—meaning they purposefully allow some minor concurrency bugs to exist in exchange for blazing-fast performance. Stay tuned for Part 2!
