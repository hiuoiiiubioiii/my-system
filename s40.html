Apache Flink

Comprehensive Study Notes & Visualizations

1. Background & Core Definition

What is Flink? Flink is a stream processing framework. It is software designed to be run on a cluster of nodes acting as consumers.

Crucial Distinction: Flink is not a message broker (like Kafka or RabbitMQ). It relies on message brokers to provide the data stream, and Flink does the actual computation.

Key Features

  • True Real-Time: Unlike Spark Streaming which uses "micro-batching" (waiting to gather a small chunk of messages before processing), Flink processes events continuously in true real-time as they arrive.
  • Stateful Computations: It persists "state" (memory of past events) across the nodes in the cluster to perform meaningful aggregations or joins on incoming data.

3. Why is "State" so Important?

If streams are just continuous flows of independent events, why does Flink need to remember them (keep state)?

  • Grouping Events by Time (Windowing): Processing data like "Show me all clicks in the last 5 minutes". Flink holds events in state until the time window closes, then processes them together.
  • Stream Enrichment: Flink can hold a local copy of a database table. When a minimal stream event arrives (e.g., just a UserID), Flink checks its local state table, enriches the event with demographic data, and passes it forward.
The Tinder Example (Pattern Matching):
1. Every "swipe right" (like) is put into a stream.
2. Flink processes the swipe and stores it in its local state.
3. If a swipe comes in, and the opposite directional swipe is already sitting in the state memory, Flink declares a Match.
4. Flink then forwards this match data into a second output stream.

4. Fault Tolerance

Because Flink nodes store vital state locally (to process things quickly without querying a database every second), a node crash is dangerous. If a node dies, the state dies with it.

The Solution: Flink occasionally takes Checkpoints of the local state on every node and saves them to durable, highly available distributed storage (like HDFS or AWS S3).

If a node crashes, Flink spins up a new node, pulls the last saved checkpoint from S3, and resumes processing the messages the node hadn't gotten to yet.

  • Single Job Manager Node: Takes your application code and automatically translates it into a parallelized "dataflow graph". It tells Task Managers what to do and initiates cluster-wide checkpoints.
  • High Availability (HA): The Job Manager is a single point of failure. Flink uses a coordination service (like ZooKeeper) to track which nodes are alive. If the Job Manager dies, a standby node uses ZooKeeper to instantly take over.
  • Task Managers: The worker nodes that actually execute the sub-tasks, process the data, and maintain local state.

6. Log-Based Message Brokers (Kafka)

Flink is designed to consume data from log-based message brokers (like Apache Kafka). These are persistent, disk-based queues.

Instead of deleting messages once read, the broker keeps them. Flink consumers keep track of an Offset (a specific index number). This allows Flink to "rewind" and replay messages if a node crashes.

7. Checkpointing Mechanism

Flink uses the Chandy-Lamport algorithm for distributed snapshots, which is consistent with causality. Here is the step-by-step flow:

  1. The Barrier: The Job Manager occasionally injects a special "Checkpoint Barrier" into the stream.
  2. In-Order Flow: Because log-based brokers deliver in order, the barrier flows through the stream just like a normal message.
  3. Snapshot: When a Task Manager receives the barrier, it pauses, notes its current broker offset, and saves its local state to HDFS/S3.
  4. Async Background: This snapshot is non-blocking. It happens in the background. The checkpoint is only considered successful when all nodes report to the Job Manager that they successfully saved their state.

Crash Recovery Consequence: If a crash happens, all nodes revert to the last successful checkpoint and resume from those saved offsets. Because some messages might have been processed between the checkpoint and the crash, reverting may result in certain messages being processed more than once (At-Least-Once processing). Developers must use idempotent operations or two-phase commits to ensure exactly-once semantics.

8. Changing Cluster Size (Scaling)

What happens when you add or remove nodes? Flink must redistribute the state.

Similar to consistent hashing, Flink aims to move as little state as possible over the network while balancing the load. To make this easy, Flink saves checkpoints as simple lists of Keys and Values (rather than complex monolithic objects representing the whole node).

By using lists of K/V pairs, Flink can easily slice and re-partition the state using range schemas when sending state to new nodes.

9. Conclusion: Flink vs. The Competition

  • Vs. Spark Streaming: Spark processes in micro-batches (small time intervals). Flink is a fully real-time stream processing engine.
  • Vs. Apache Storm / Samza: Older engines required you to manually build the dataflow graph and handle parallelization yourself. Flink automatically calculates the dataflow graph and parallelizes the work for the developer, significantly improving the developer experience.
