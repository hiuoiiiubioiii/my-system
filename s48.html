Bitcoin Design Breakdown

Continuous scrolling study feed based on lecture slides & verbal explanations

Topic 1

Background & Motivation

Originally, the motivation behind Bitcoin was to act as a decentralized currency. The goal was to eliminate the need for all transactions to be approved and monitored by a centralized third party (like a bank).

However, with decentralization comes a lack of trust. Malicious actors on the network may try to trick others into believing they have more money than they really do. Bitcoin solves this by leveraging a blockchain alongside cryptographic proofs.

🎙️ Instructor's Insight:
Because we cannot trust the parties running the network (the local computers/nodes), we cannot use traditional centralized databases. We have to adapt typical "ordered log" algorithms (like distributed consensus models such as Raft or Paxos) to work in a "trustless" environment where literally anyone could be a bad actor.
Topic 2

Coins & Tracking Ownership

In Bitcoin, a "coin" is simply comprised of a chain of transaction records showing every transfer of that coin. The latest record in the chain shows the public key of the current owner.

How the Cryptography Works:

  • Signing: Jordan uses his Private Key to sign the block, proving he is initiating the transfer.
  • Verifying: The receiver can use Jordan's Public Key to decrypt the message. It should mathematically match the hash of the previous block, proving Jordan was the one who signed it.
🎙️ Instructor's Insight:
If Jordan wants to buy something, he creates a new transaction record. Anyone on the network can take Jordan's Public Key (which is publicly visible on the previous block) to decrypt his signature. This mathematical proof guarantees Jordan is the true owner and authorized the transfer.
Topic 3

The Double Spending Problem

While the previous design is useful for tracking ownership, there is a fatal flaw: Nothing stops an owner from creating two transaction blocks, sending the exact same coin to two different people.

If we had a publicly visible ordered log of all transactions, the receiver could look back, see the coin was already spent, and reject the payment. The challenge? Ordering events typically requires a central server!

🎙️ Instructor's Insight:
The core design challenge of Bitcoin is figuring out how to create this globally ordered log without a central authority dictating what happened first.
Topic 4

The Blockchain Structure

The solution to Double Spending is the Blockchain: A set of blocks acting as a public ledger (record of transactions) for all coins in the network.

What is inside a Block?

  • Previous Hash: A hash of the previous block in the chain (creates the link).
  • Transactions: A bundled list of valid transactions.
  • Nonce: A special number used to prove the validity of the block.

Every node keeps a copy of the blockchain. When a node wants to add a block, it broadcasts it. Because all valid transactions are inside this shared ledger, no one can double spend.

Topic 5

Adding a Block (Proof of Work)

When a peer receives new transactions, it bundles them into a block. But to officially add it to the chain, the peer must mine the block using CPU power.

Find x such that:
hash(transactions, prev_block_hash, x) = 000000...
  • The peer must use CPU power to determine the nonce (x). This is called Proof of Work.
  • The resulting hash must start with n leading zeros.
  • Parameter n adjusts over time to keep the rate of mined blocks relatively constant.
🎙️ Instructor's Insight:
This is a critical concept. You cannot just add a block; you must solve a highly intensive CPU puzzle. The node guesses a random number (x), hashes everything together, and checks if it has enough zeros. If not, it guesses again. It requires massive brute-force computing.

Difficulty Adjustment: The network wants exactly 1 block created every 10 minutes. If more computers join and solve puzzles too fast, the network increases n (requires more leading zeros), making the puzzle harder.
Topic 6

Validating Blocks

Once peers receive a newly broadcasted block, they do not just trust it. They run a checklist to ensure it is valid.

✔
Check the Hash: Does the new block actually have the proper number of leading zeros?
✔
Check the Chain: Does the previous block hash point to the actual existing block in the chain?
✔
Check Transactions: Are they valid? Ensure there is no other record of the coin being spent (no double spending) and that signatures are mathematically valid.

Peers "vote" that they have accepted a block by adding it to their ledger and starting to mine the next block using this new block's hash.

Topic 7 & 8

Forks & Resolving Conflicts

Sometimes, the blockchain splits into two paths, creating a Fork. This happens if:

  • Two honest peers find the nonce at the exact same time (network lag causes nodes to receive them in different orders).
  • One malicious peer intentionally sends one block to half the network and a different block to the other half.

Resolving the Conflict: The Longest Chain Rule

Peers will continue to add blocks to whichever branch they received first. However, if a peer sees that another branch has gotten longer, it will abandon its current shorter branch and switch to the end of the longer branch.

🎙️ Instructor's Insight:
Double spending is technically possible during a fork. This is why you wait for confirmations. If you sell a car for Bitcoin, do not hand over the keys the second the transaction hits the blockchain. Wait until 5 or 6 blocks have been built on top of your transaction to ensure it is on the accepted "longest chain" and won't be erased by a resolved fork.
Topic 9

Conclusion & Drawbacks

Bitcoin is an extremely interesting culmination of computer science concepts (ordered logs, hashes, conflict resolution). However, it has significant downsides:

  • Extremely Slow: Hardcoded to only process one block every ten minutes.
  • Resource Intensive: Causes computers to use a bunch of CPU cycles and massive electricity to essentially "not accomplish anything" other than guessing random numbers (wasteful).

As a result, many alternative blockchain implementations have sprung up trying to fix these problems and add new features beyond just cryptocurrency.

Exam Prep: Key Takeaways

  • Public/Private Keys: Private key encrypts to prove ownership. Public key is used by others to verify that signature.
  • Proof of Work & The Nonce: The Nonce is a random number brute-forced to make a block's hash start with $n$ zeros. It proves CPU work was done, preventing spam.
  • Difficulty Adjustment: The network automatically adjusts the required zeros ($n$) to ensure a new block is mined roughly every 10 minutes, regardless of network computing power.
  • Consensus (Longest Chain Rule): Network truth is always dictated by the longest chain. Shorter chains are abandoned.
  • Confirmations: Always wait for multiple blocks to be built on top of a transaction to ensure it isn't lost during a temporary blockchain fork.
